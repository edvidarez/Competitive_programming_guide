Tarea 6 — Parrots

Loros mensajeros que llegan en cualquier orden · PDF original (oficial): statement.pdf

El problema

Yanee es una entusiasta de las aves. Desde que leyó sobre IP over Avian Carriers (IPoAC, “IP sobre portadores aviares”), ha dedicado mucho de su tiempo a entrenar una parvada de loros inteligentes para llevar mensajes a grandes distancias.

El sueño de Yanee es usar sus aves para enviar un mensaje \(M\) a una tierra muy muy lejana. Su mensaje \(M\) es una secuencia de \(N\) enteros (no necesariamente distintos), cada uno entre \(0\) y \(255\) inclusive. Yanee tiene \(K\) loros especialmente entrenados. Todos los loros se ven igual; Yanee no los puede distinguir. Cada ave puede recordar un solo entero entre \(0\) y \(R\) inclusive.

Al principio intentó un esquema sencillo: para enviar un mensaje, Yanee dejaba salir a las aves de la jaula una por una. Antes de que cada ave alzara el vuelo, le enseñaba un número de la secuencia del mensaje, en orden. Desafortunadamente, este esquema no funcionó. Al final todas las aves llegaban al destino, pero no necesariamente en el orden en que salieron. Con este esquema Yanee podía recuperar todos los números que envió, pero no era capaz de ponerlos en el orden correcto.

Para hacer realidad su sueño, Yanee necesita un mejor esquema, y para eso necesita tu ayuda. Dado un mensaje \(M\), planea soltar a las aves una por una como antes. Necesita que escribas un programa que realice dos operaciones separadas:

  • Primero, tu programa debe poder leer un mensaje \(M\) y transformarlo en una secuencia de a lo más \(K\) enteros entre \(0\) y \(R\) que ella enseñará a las aves.
  • Segundo, tu programa debe poder leer la lista de enteros entre \(0\) y \(R\) que se reciben cuando las aves llegan a su destino, y transformarla de regreso al mensaje original \(M\).

Puedes suponer que todos los loros siempre llegan al destino, y que cada uno recuerda el número que se le asignó. Yanee te recuerda una vez más que los loros pueden llegar en cualquier orden. Nota que Yanee solo tiene \(K\) loros, así que la secuencia de enteros entre \(0\) y \(R\) que produzcas debe contener a lo más \(K\) enteros.

Tu tarea

Escribe dos procedimientos separados. Uno lo usará el emisor (codificador) y el otro el receptor (decodificador).

Detalles de implementación

Los dos procedimientos que debes escribir son:

Firmas — lo que implementa el concursante (encoder.c[pp] / decoder.c[pp]) 
void encode(int N, int M[]);
  // N    = longitud del mensaje.
  // M    = arreglo de N enteros, el mensaje;  0 <= M[i] <= 255  para 0 <= i < N.
  // Debe codificar M en una secuencia de enteros entre 0 y R que se envía con las
  // aves. Para reportar cada entero a que quieras dar a un ave, llama a send(a).

void decode(int N, int L, int X[]);
  // N    = longitud del mensaje original.
  // L    = longitud del mensaje recibido (cuántas aves se enviaron).
  // X    = arreglo de L enteros recibidos: son EXACTAMENTE los enteros que produjo
  //        encode, pero posiblemente reordenados (permutados) de otra forma.
  // Debe recuperar M. Para reportarlo, llama a output(b) para cada entero b del
  // mensaje decodificado, EN EL ORDEN CORRECTO.
Firmas — las provee el evaluador (encoderlib.h / decoderlib.h) 
void send(int a);     // la llamas desde encode: agrega el entero a al mensaje codificado
void output(int b);   // la llamas desde decode:  emite el entero b del mensaje decodificado

Nota que \(R\) y \(K\) no se dan como parámetros de entrada; consulta las descripciones de las subtareas más abajo. Para resolver correctamente una subtarea dada, tus procedimientos deben satisfacer las siguientes condiciones:

  • Todos los enteros enviados por encode deben estar en el rango especificado en la subtarea (entre \(0\) y \(R\)).
  • El número de veces que encode llama a send no debe exceder el límite \(K\) especificado en la subtarea. \(K\) depende de la longitud del mensaje.
  • decode debe recuperar correctamente el mensaje original \(M\) y llamar a output exactamente \(N\) veces, con \(b\) igual a \(M[0], M[1], \dots, M[N-1]\), respectivamente.

En la última subtarea, tu puntaje varía según la razón entre las longitudes del mensaje codificado y el original.

Importante (independiente del orden). El canal —las aves— destruye el orden: decode recibe los mismos números que mandó encode, pero permutados arbitrariamente (y el evaluador real prueba muchas permutaciones distintas). Lo único en lo que te puedes apoyar es en qué números llegaron y cuántas veces llegó cada uno.

Ejemplo

Considera el caso en que \(N = 3\) y \(M = (10,\ 30,\ 20)\).

El procedimiento encode(N, M), usando algún método extraño, podría codificar el mensaje como la secuencia \((7, 3, 2, 70, 15, 20, 3)\). Para reportar esta secuencia, llamaría a send así:

Llamadas en encode

send(7)
send(3)
send(2)
send(70)
send(15)
send(20)
send(3)

Llamadas en decode

output(10)
output(30)
output(20)

Una vez que todos los loros llegan a su destino, supón que obtenemos la siguiente lista de números transcritos: \((3, 20, 70, 15, 2, 3, 7)\) —los mismos siete números, revueltos—. Entonces decode se llama con \(N = 3\), \(L = 7\) y \(X = (3, 20, 70, 15, 2, 3, 7)\), y debe producir el mensaje original \((10, 30, 20)\) llamando a output como arriba.

La secuencia \((7, 3, 2, 70, 15, 20, 3)\) es solo una ilustración de la API (cómo se llama a send / output); no corresponde a ningún algoritmo correcto en particular. Lo único exigible es que el viaje completo \(M \to\) codificar \(\to\) revolver \(\to\) decodificar reproduzca \(M\).

Resumen

El mensaje es \(N\) bytes \(M[0..N-1]\) con \(M[i] \in [0,255]\). Implementa encode, que llama a send a lo más \(K\) veces con valores en \([0,R]\), y decode, que recibe esos mismos valores permutados arbitrariamente (con posibles repeticiones) y debe reconstruir \(M\) exacto. El canal borra el orden: lo único que sobrevive es el multiconjunto de valores enviados. En la Subtarea 5 el puntaje premia menor razón \(L/N\).

Lo que importa de los límites Como el orden se pierde, codificar “posición + dato” en el valor de cada ave es lo natural en las subtareas con \(R\) o \(K\) holgados. La Subtarea 5 (\(N \le 64\), \(R = 255\), \(K = 15N\), puntaje por razón \(L/N \le 5\)) es la que obliga a exprimir la información: cada ave carga \(\log_2 256 = 8\) bits, pero como el orden no cuenta, la información real está en las multiplicidades, no en una secuencia ordenada.

Pistas

Pista 1 — ¿Qué sobrevive al canal?

El decodificador recibe los mismos números que mandaste, pero en cualquier orden (y el evaluador real prueba muchas permutaciones). ¿Qué información no depende del orden y por tanto siempre puedes usar con seguridad?

Solo el multiconjunto: qué valores llegaron y cuántas veces llegó cada uno. Si denotas \(c_v\) = número de aves que cargan el valor \(v\) (para \(v = 0 \dots 255\)), entonces \((c_0, c_1, \dots, c_{255})\) es lo único robusto. Cualquier esquema que dependa de “el ave \(i\)-ésima trae tal cosa” está condenado: el ave \(i\)-ésima puede llegar en cualquier posición.

¿Y si los valores que mandas son distintos? Entonces el decodificador puede simplemente ordenarlos y recuperar un orden canónico. Esa es la idea base de las primeras subtareas.

Pista 2 — Subtarea 1 (17 pts): \(N=8\), bits, \(R=65535\)

Aquí \(M[i] \in \{0,1\}\): el mensaje son \(8\) bits. ¿Qué pasa si solo te interesan las posiciones donde el bit vale \(1\)? ¿Cómo las mandas para que el desorden no estorbe?

Envía las posiciones de los unos: para cada \(i\) con \(M[i] = 1\), llama send(i). Son a lo más \(N = 8\) valores y todos son distintos, así que el decodificador los ordena (o solo marca un arreglo de bits en \(1\)) y reconstruye \(M\). Longitud \(\le N \le K = 10N\). Pasa la subtarea.

Pista 3 — Subtarea 2 (17 pts): \(N \le 16\), \(R=65535\)

Ahora los datos son bytes completos (\(0 \dots 255\)), no bits; pero \(R = 65535\) te da \(16\) bits por ave. ¿Cómo metes posición y dato en el mismo número para no perder el orden?

Empaca: \(E_i = 256 \cdot i + M[i]\). Los \(8\) bits altos guardan la posición \(i\) y los \(8\) bajos el dato \(M[i]\). Como cada \(E_i\) tiene una posición distinta en los bits altos, los \(E_i\) son distintos; el decodificador ordena \(X\) (quedan por posición creciente) y lee \(M[i] = E_i \bmod 256\). A lo más \(N \le K = 10N\) aves.

Pista 4 — Subtarea 3 (18 pts): \(N \le 16\), \(R=255\)

Se acabó el espacio “extra”: ahora \(R = 255\), solo \(8\) bits por ave, justo lo que mide un dato. No caben posición y dato juntos en un byte… ¿y si bajas la granularidad a bits en vez de bytes?

Ve el mensaje como \(8N \le 128\) bits \(B_0, B_1, \dots\). Como hay a lo más \(128\) posiciones de bit, te bastan \(7\) bits para la posición y \(1\) para el valor del bit: envía \(E = 2 \cdot (\text{posición del bit}) + B\) por cada bit (o solo por los bits en \(1\), con el valor implícito). Cada \(E\) cabe en \([0,255]\), los valores son distintos por posición, el decodificador ordena y reconstruye los bits. Longitud \(\le 8N \le K\).

Pista 5 — Subtarea 4 (29 pts): \(N \le 32\), \(R=255\)

Con \(N \le 32\) hay hasta \(8N = 256\) posiciones de bit: ¡ahora la posición de un bit ya necesita los \(8\) bits completos! No queda lugar para el valor. ¿Te suena al truco de la Subtarea 1?

Igual que con los unos de la Subtarea 1, pero a nivel de bits: envía la posición de cada bit que vale \(1\) (un número en \([0, 8N-1] \subseteq [0,255]\)). El decodificador parte de todo en cero y prende esos bits. Las posiciones son distintas, así que el desorden no importa. Longitud \(\le 8N \le K = 10N\). Pasa la subtarea 4 (y de paso las anteriores).

Hasta aquí ya juntas \(17+17+18+29 = 81\) puntos con ideas “posición + dato”. La Subtarea 5 pide algo cualitativamente mejor.

Pista 6 — Subtarea 5 (hasta 19 pts): la razón \(L/N\) manda

Mandar \(8N\) posiciones de bit da razón \(L/N \le 8\) en el peor caso (puntaje pobre). Para \(P \le 5\) necesitas \(\le 5N\) aves. ¿Cuánta información cabe realmente en un multiconjunto, y cómo la cuentas?

Cuenta los grados de libertad. Un mensaje de \(N\) bytes es un entero \(V \in [0,\ 256^N)\): hay \(256^N\) mensajes posibles. Un multiconjunto de tamaño \(\le S\) sobre \(256\) símbolos equivale a una tupla de cuentas \((c_0, \dots, c_{255}, \text{holgura})\) de \(257\) variables no negativas que suman exactamente \(S\). Por estrellas y barras, hay \[\binom{S+256}{256}\] tales tuplas.

Si eliges el menor \(S\) con \(\binom{S+256}{256} \ge 256^N\), ¿no tendrías una biyección entre mensajes y multiconjuntos que puedes recorrer con ranking / unranking?

Esa es la dirección al óptimo: en lugar de gastar una ave por bit, gastas aves solo para “apuntar” a cuál multiconjunto de entre los \(\binom{S+256}{256}\) representa tu mensaje. Resulta que ese \(S\) crece mucho más lento que \(8N\): el peor caso da razón \(\approx 4.08\), bajo el umbral \(5\). La última pista lo construye.

Pista final — Solución completa (100 puntos)

Idea (editorial, “razón óptima”). El canal entrega un multiconjunto de valores en \([0,255]\). Codifica el mensaje completo, visto como un único entero \(V\), en cuál multiconjunto mandas. Concretamente:

  • Interpreta \(M\) en base \(256\): \(V = M[0]\cdot 256^{N-1} + M[1]\cdot 256^{N-2} + \dots + M[N-1]\), con \(M[0]\) el dígito más significativo. Así \(V \in [0, 256^N)\).
  • Elige \(S = L_{\max}(N) = \) el menor \(S\) tal que \(\binom{S+256}{256} \ge 256^N\). Con eso hay al menos tantos multiconjuntos (de tamaño exactamente \(S\) sobre \(257\) variables, contando una de holgura) como mensajes: una biyección.
  • Define \(f(\text{numvars}, \text{suma}) = \binom{\text{suma}+\text{numvars}-1}{\text{numvars}-1}\) = número de tuplas de numvars variables no negativas que suman suma.

encode — “unrank”: \(V \to\) multiconjunto. Recorre \(i = 0 \dots 255\); tras fijar \(c_0, \dots, c_{i-1}\) quedan after \(= 256 - i\) variables (incluyendo la holgura) y una suma restante \(S_{\text{rest}}\). Para \(c_i = t = 0, 1, 2, \dots\), el número de formas de completar el resto es \(\text{ways} = f(\text{after},\ S_{\text{rest}} - t)\). Mientras \(V \ge \text{ways}\), resta \(V \mathrel{-}= \text{ways}\) e incrementa \(t\); en cuanto \(V < \text{ways}\), fija \(c_i = t\), resta \(S_{\text{rest}} \mathrel{-}= t\) y pasa a \(i+1\). Al final manda cada valor: send(v) repetido \(c_v\) veces. La holgura no se envía, así que la longitud es \(S - \text{holgura} \le S = L_{\max}(N) \le 5N\).

decode — “rank”: multiconjunto \(\to V\). Cuenta multiplicidades \(c_v\) en \(X\) (esto ignora el orden y absorbe duplicados). Recomputa \(S = L_{\max}(N)\) idéntico. Recorre \(i = 0 \dots 255\); para \(t = 0 \dots c_i - 1\) acumula \(V \mathrel{+}= f(256 - i,\ S_{\text{rest}} - t)\), luego \(S_{\text{rest}} \mathrel{-}= c_i\). Es el inverso exacto del unrank. Finalmente expande \(V\) a \(N\) dígitos base \(256\) y llama output en orden \(M[0], \dots, M[N-1]\).

Pseudocódigo 
// Binomios GRANDES C(n,r) precomputados una vez (Pascal, n<=600, r<=257); bignum.
f(numvars, suma):
    si suma < 0: regresar 0
    regresar C(suma + numvars - 1, numvars - 1)

Lmax(N):                              // menor S con C(S+256,256) >= 256^N
    objetivo = 256^N                  // bignum
    S = 0
    mientras C(S+256, 256) < objetivo: S += 1
    regresar S

encode(N, M):
    V = 0                             // bignum
    para i = 0..N-1: V = V*256 + M[i] // M[0] es el dígito más significativo
    S = Lmax(N);  Srest = S
    para i = 0..255:
        after = 256 - i               // variables restantes (incluye holgura)
        t = 0
        repetir:
            ways = f(after, Srest - t)
            si V < ways: romper
            V -= ways;  t += 1
        c[i] = t;  Srest -= t
    para v = 0..255:
        repetir c[v] veces: send(v)   // longitud = S - holgura <= 5N

decode(N, L, X):
    c[0..255] = 0
    para cada x en X: c[x] += 1       // SOLO multiplicidades: inmune al orden
    S = Lmax(N);  Srest = S;  V = 0   // bignum
    para i = 0..255:
        after = 256 - i
        para t = 0..c[i]-1:
            V += f(after, Srest - t)
        Srest -= c[i]
    // V -> N dígitos base 256, de más a menos significativo
    para i = N-1 .. 0: dig[i] = V mod 256; V = V div 256
    para i = 0..N-1: output(dig[i])

Por qué es correcto. El unrank y el rank son inversos exactos sobre el orden lexicográfico de las tuplas \((c_0, \dots, c_{255}, \text{holgura})\) que suman \(S\); como \(\binom{S+256}{256} \ge 256^N\), todo \(V \in [0, 256^N)\) cae en una tupla válida y se recupera sin ambigüedad. decode solo usa multiplicidades, así que es inmune a cualquier permutación y a los duplicados (que son precisamente \(c_v > 1\)). Como solo se envían valores en \([0,255]\), la solución es válida en el rango de todas las subtareas; y como la longitud nunca pasa de \(5N \le K\), satisface también el límite \(K\) de cada una: una sola solución da los \(100\) puntos.

Por qué da puntaje pleno en la Subtarea 5. Se verifica que \(L_{\max}(N)/N \le 5\) para todo \(N = 1 \dots 64\); el peor caso es \(N = 64\) con \(L_{\max}(64) = 261\) y razón \(261/64 \approx 4.078 \le 5\), así que \(P \le 5 \Rightarrow 19/19\).

Trampas que cuestan el problema.
  1. Asumir orden. El evaluador baraja con orden ascendente, descendente y aleatorio. decode debe contar multiplicidades, nunca apoyarse en posiciones.
  2. Los duplicados son el dato. La información se codifica justo en cuántas copias de cada valor mandas; el multiconjunto (no el conjunto) es la clave.
  3. \(S\) debe depender de \(N\). Usa \(S = L_{\max}(N)\). Fijar \(S = L_{\max}(64) = 261\) para todo \(N\) haría que mensajes cortos rebasen \(K = 10N\) (p. ej. \(N = 8\): \(K = 80\) pero longitud hasta \(261\)). Con \(L_{\max}(N)\), la longitud \(\le 5N \le K\) siempre.
  4. El índice crítico es after \(= 256 - i\) (variables después de \(i\), incluyendo la holgura). Usar \(255 - i\) rompe la biyección.
  5. Hace falta bignum. \(256^{64} = 2^{512}\) y \(\binom{517}{256} \approx 1.47\times10^{154}\) no caben en \(64\) bits ni en __int128: implementa enteros grandes (base \(2^{32}\)) y un triángulo de Pascal de binomios grandes.
  6. Convención de dígitos. \(M[0]\) es el dígito más significativo de \(V\); mantén la misma convención al construir \(V\) (encode) y al extraer dígitos (decode) o no invierte.

Complejidad. El triángulo de Pascal de binomios grandes se construye una sola vez (perezosamente): \(O(N_{\max} \cdot R_{\max})\) sumas de bignum con \(N_{\max}=600\), \(R_{\max}=257\) (\(\sim\)155k sumas), \(< 100\) ms. Por encode: construir \(V\) en \(O(N)\), hallar \(L_{\max}(N)\) (\(\le 261\) comparaciones) y el unrank (\(\sum_i (c_i+1) = L + 256 \le 517\) evaluaciones de \(f\), cada una una consulta \(O(1)\) más resta \(O(\text{limbs})\)). Por decode: contar \(O(L)\) y el rank (\(\le 261\) sumas de bignum). Medido: \(0.02\) s para \(50\) casos con \(N\) hasta \(64\); memoria \(\ll 256\) MB.
