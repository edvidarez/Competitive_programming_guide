IOI 2020 · Día 2

Tarea 6 — Stations

PDF original (oficial): statement.pdf

El problema

La Red Troncal de Internet de Singapur (SIB) está formada por \(n\) estaciones, a las que se les asignan índices del \(0\) al \(n-1\). También hay \(n-1\) enlaces bidireccionales, numerados del \(0\) al \(n-2\). Cada enlace conecta dos estaciones distintas. Dos estaciones unidas por un solo enlace se llaman vecinas.

Un camino de la estación \(x\) a la estación \(y\) es una secuencia de estaciones distintas \(a_0, a_1, \ldots, a_p\) tal que \(a_0 = x\), \(a_p = y\), y cada dos estaciones consecutivas del camino son vecinas. Entre cualquier estación \(x\) y cualquier otra estación \(y\) existe exactamente un camino. (En otras palabras: la red es un árbol.)

Cualquier estación \(x\) puede crear un paquete (un dato) y enviarlo a otra estación \(y\), llamada el destino del paquete. Ese paquete debe ruteársele por el único camino de \(x\) a \(y\) de la siguiente forma. Considera una estación \(z\) que en este momento tiene un paquete cuyo destino es \(y\) (con \(z \neq y\)). En esa situación, la estación \(z\):

  1. ejecuta un procedimiento de ruteo que determina cuál vecino de \(z\) está sobre el único camino de \(z\) a \(y\), y
  2. reenvía el paquete a ese vecino.

Sin embargo, las estaciones tienen memoria limitada y no guardan la lista completa de enlaces de la SIB para usarla en el ruteo.

Tu tarea es implementar un esquema de ruteo para la SIB, que consta de dos procedimientos.

  • El primer procedimiento recibe como entrada \(n\), la lista de enlaces de la SIB y un entero \(k \ge n-1\). Asigna a cada estación una etiqueta entera única entre \(0\) y \(k\), inclusive.
  • El segundo procedimiento es el de ruteo, que se despliega a todas las estaciones una vez asignadas las etiquetas. Recibe únicamente lo siguiente:
    • \(s\), la etiqueta de la estación que tiene ahora el paquete,
    • \(t\), la etiqueta de la estación destino (\(t \neq s\)),
    • \(c\), la lista de las etiquetas de todos los vecinos de \(s\).
    Debe devolver la etiqueta del vecino de \(s\) al que se le debe reenviar el paquete.

En una de las subtareas, el puntaje de tu solución depende del valor de la etiqueta máxima asignada a cualquier estación (en general, entre más chica, mejor).

Detalles de implementación

Debes implementar los dos procedimientos siguientes. No hay lectura de stdin ni escritura a stdout: el evaluador llama a tus funciones.

Firma 1 — etiquetado 
int[] label(int n, int k, int[] u, int[] v)
  • \(n\): número de estaciones de la SIB.
  • \(k\): etiqueta máxima que se puede usar.
  • \(u\) y \(v\): arreglos de tamaño \(n-1\) que describen los enlaces. Para cada \(i\) (\(0 \le i \le n-2\)), el enlace \(i\) conecta las estaciones de índices \(u[i]\) y \(v[i]\).
  • Debe devolver un arreglo \(L\) de tamaño \(n\). Para cada \(i\) (\(0 \le i \le n-1\)), \(L[i]\) es la etiqueta asignada a la estación de índice \(i\). Todos los elementos de \(L\) deben ser únicos y estar entre \(0\) y \(k\), inclusive.
Firma 2 — ruteo 
int find_next_station(int s, int t, int[] c)
  • \(s\): etiqueta de la estación que tiene el paquete.
  • \(t\): etiqueta de la estación destino.
  • \(c\): arreglo con las etiquetas de todos los vecinos de \(s\). El arreglo \(c\) viene ordenado en forma ascendente.
  • Debe devolver la etiqueta del vecino de \(s\) al que se le debe reenviar el paquete.

Cada caso de prueba involucra uno o más escenarios independientes (es decir, distintas descripciones de la SIB). Para un caso con \(r\) escenarios, un programa que llama a los procedimientos anteriores se corre exactamente dos veces:

Primera corrida del programa:

  • se llama a label \(r\) veces,
  • el sistema de evaluación guarda las etiquetas devueltas, y
  • find_next_station no se llama.

Segunda corrida del programa:

  • find_next_station puede llamarse varias veces. En cada llamada se elige un escenario arbitrario y se usan como entrada las etiquetas que label produjo en ese escenario.
  • label no se llama.

En particular, cualquier información guardada en variables estáticas o globales durante la primera corrida NO está disponible dentro de find_next_station. Toda la decisión de ruteo debe tomarse solo con \(s\), \(t\) y \(c\).

Ejemplo

Considera la siguiente llamada:

label(5, 10, [0, 1, 1, 2], [1, 2, 3, 4])

Hay \(5\) estaciones y \(4\) enlaces que conectan los pares de índices \((0,1)\), \((1,2)\), \((1,3)\) y \((2,4)\). Cada etiqueta puede ser un entero de \(0\) a \(k = 10\). Para reportar el siguiente etiquetado:

ÍndiceEtiqueta
06
12
29
33
47

el procedimiento label debe devolver \([6, 2, 9, 3, 7]\). En la figura, los números muestran los índices (panel izquierdo) y las etiquetas asignadas (panel derecho).

Supón que las etiquetas se asignaron como arriba y considera la llamada:

find_next_station(9, 6, [2, 7])

Significa que la estación con el paquete tiene etiqueta \(9\) y el destino tiene etiqueta \(6\). Las etiquetas de las estaciones en el camino al destino son \([9, 2, 6]\). Por eso la llamada debe devolver \(2\), que es la etiqueta de la estación a la que se debe reenviar el paquete (la de índice \(1\)).

Considera otra llamada posible:

find_next_station(2, 3, [3, 6, 9])

El procedimiento debe devolver \(3\), ya que la estación destino con etiqueta \(3\) es vecina de la estación con etiqueta \(2\), así que recibe el paquete directamente.

Evaluador de ejemplo

El evaluador de ejemplo lee la entrada en este formato:

  • línea \(1\): \(r\).

Siguen \(r\) bloques, cada uno describiendo un escenario:

  • línea \(1\): \(n\ \ k\).

  • línea \(1+n\): \(q\), el número de llamadas a find_next_station.
  • línea \(2+n+j\) (\(0 \le j \le q-1\)): \(z[j]\ \ y[j]\ \ w[j]\), los índices de las estaciones de la \(j\)-ésima llamada: \(z[j]\) tiene el paquete, \(y[j]\) es el destino y \(w[j]\) es la estación a la que debería reenviarse.

El evaluador imprime, en la línea \(1\), la etiqueta máxima \(m\); y luego, por escenario, el índice de la estación cuya etiqueta devolvió cada llamada a find_next_station.

Resumen

La red es un árbol de \(n\) estaciones. Debes label cada nodo con un entero único en \([0, k]\) y luego, en find_next_station(s, t, c), decidir a cuál vecino reenviar el paquete usando solo \(s\), \(t\) y la lista ordenada de vecinos \(c\) (¡sin estado global entre las dos corridas!). La etiqueta debe codificar, ella sola, suficiente información de la estructura del árbol para reconstruir el siguiente salto. En la subtarea 5 el puntaje crece conforme baja la etiqueta máxima \(m\); el pleno se alcanza con \(m \le 1000\).

Lo que importa de los límites \(n \le 1000\) y \(k \ge n-1\): existe espacio de etiquetas justo del tamaño del árbol, así que apuntar a \(m = n-1 \le 999\) da el pleno (\(m \le 1000\)). find_next_station no puede usar variables globales: la etiqueta es el único canal de información. Las subtareas chicas (línea, árbol binario, estrella, \(n \le 8\)) son escalones que sugieren la idea general: si el subárbol de cada nodo ocupa un intervalo contiguo de etiquetas, basta comparar \(t\) con esos intervalos.

Pistas

Pista 1 — ¿Cómo modelarlo?

¿Ya notaste que con \(n\) nodos, \(n-1\) aristas y camino único entre cualquier par, la red es forzosamente un árbol? En un árbol, reenviar de \(s\) hacia \(t\) equivale a: "¿\(t\) está en el subárbol de alguno de mis hijos, o tengo que subir hacia mi padre?".

La etiqueta es el único dato que sobrevive entre la corrida de label y la de find_next_station. Entonces la pregunta clave es: ¿qué información puede codificar la etiqueta misma para que un nodo, viendo solo su etiqueta \(s\), la del destino \(t\) y las de sus vecinos \(c\), deduzca hacia dónde va \(t\)?

Pista 2 — Empieza por las subtareas chicas

¿Qué te regala cada forma especial del árbol?

  • Subtarea 1 (línea, \(k=1000\)): nadie tiene más de 2 vecinos. Si numeras los nodos \(0,1,2,\ldots\) a lo largo de la línea, ¿no basta con comparar \(t\) contra \(s\) para saber si avanzas al vecino mayor o al menor?
  • Subtarea 2 (árbol binario por índice): el enlace \(i\) une \(i+1\) con \(\lfloor i/2 \rfloor\). ¿Podrías etiquetar con la posición tipo heap y deducir el padre/hijos por aritmética? (Pista mejor: piensa en un recorrido del árbol.)
  • Subtarea 3 (casi-camino con una estrella, \(k=10^6\)): a lo más un nodo de alto grado. ¿Qué pasaría si a cada nodo le das un rango de etiquetas para "su" parte del árbol?
  • Subtarea 4 (\(n \le 8\)): \(k = 10^9\) gigante y árbol minúsculo. Aquí cabe hasta fuerza bruta: puedes darte el lujo de codificar mucha información en etiquetas grandes. Útil para validar tu idea antes de optimizar la constante.

El patrón que conecta todas: ¿y si el conjunto de etiquetas de un subárbol siempre fuera un intervalo contiguo de enteros?

Pista 3 — La observación clave: intervalos de un DFS

¿Recuerdas el recorrido de Euler / los tiempos de entrada de un DFS? Si etiquetas cada nodo con su tiempo de pre-orden (un contador que crece de \(0\) a \(n-1\)), el subárbol de cada nodo ocupa un intervalo contiguo de etiquetas \([\text{entrada}, \text{salida}]\).

Con eso, en un nodo \(s\) bastaría saber el intervalo de cada hijo para mandar \(t\) al hijo "dueño" del sub-intervalo que contiene a \(t\), o al padre si \(t\) cae fuera. Pero ojo: find_next_station solo ve \(s\), \(t\) y \(c\) (las etiquetas de los vecinos), no los intervalos. ¿Cómo recuperar el intervalo del subárbol de \(s\) a partir únicamente de su etiqueta y de las de sus vecinos?

Truco fino: ¿y si alternas el criterio según la paridad de la profundidad del nodo, de modo que en un nivel el nodo sea el mínimo de su subárbol y en el siguiente sea el máximo? Así, comparando \(s\) con su vecino más chico podrías deducir en qué caso estás, y de ahí el intervalo completo.

Pista final — Solución completa (100 puntos)

Idea

Etiquetado por intervalos de DFS con codificación por paridad de profundidad. Enraíza el árbol en el nodo \(0\) y haz un DFS llevando un contador global que avanza de \(0\) a \(n-1\). La clave es la paridad de la profundidad del nodo:

  • En profundidad par, el nodo recibe su etiqueta al entrar (tiempo de pre-orden): toma el valor del contador antes de visitar a sus hijos. Así queda como el mínimo de su subárbol.
  • En profundidad impar, el nodo recibe su etiqueta al salir (tiempo de post-orden): toma el valor del contador después de visitar a todos sus hijos. Así queda como el máximo de su subárbol.

Como el contador es único y crece monótonamente, cada etiqueta es única y cae en \([0, n-1]\). Con \(n \le 1000\) eso da \(m = n-1 \le 999 \le 1000\): escalón pleno de la subtarea 5, y de paso cumple \(k = 1000\) y \(k = 10^6\) de las subtareas 1–4. No se hace ninguna minimización por búsqueda: el bound sale directo de la construcción.

Cómo decide find_next_station(s, t, c)

El arreglo \(c\) viene ordenado. La paridad de \(s\) se deduce comparándola con su vecino más chico \(c[0]\):

  • Caso etiqueta de ENTRADA (\(s\) en nivel par, \(s < c[0]\)): \(s\) es menor que todos sus descendientes. El vecino más grande \(c[\text{último}]\) es el padre, y el subárbol de \(s\) ocupa \([\,s,\ c[\text{último}]\,]\) (su descendiente de etiqueta máxima). Si \(t < s\) o \(t > c[\text{último}]\), \(t\) está fuera del subárbol: reenvía al padre \(c[\text{último}]\). Si \(t\) cae dentro, reenvía al hijo de menor etiqueta que sea \(\ge t\) (un lower_bound sobre \(c\)): ese hijo "posee" el sub-intervalo que contiene a \(t\).
  • Caso etiqueta de SALIDA (\(s\) en nivel impar, \(s > c[0]\)): el vecino más chico \(c[0]\) es el padre y el subárbol de \(s\) termina en \(s\); su intervalo es \([\,c[0]+1,\ s\,]\) hacia abajo. Si \(t > s\) o \(t \le c[0]\), \(t\) está fuera: reenvía al padre \(c[0]\). En otro caso, reenvía al hijo de mayor etiqueta que aún sea \(\le t\) (un upper_bound menos uno).

Por qué es correcto

El DFS asigna valores monótonos, así que el subárbol de cada nodo es un intervalo contiguo de etiquetas. La paridad garantiza que en nivel par el nodo es el mínimo de su subárbol (y su padre es el vecino de etiqueta máxima), y en nivel impar es el máximo (y su padre es el vecino de etiqueta mínima). Comparando \(t\) con el intervalo deducido de \(s\) y \(c\), find_next_station identifica sin ambigüedad si \(t\) está en el subárbol —y en cuál hijo, por la frontera entre sub-intervalos consecutivos de los hijos— o fuera (entonces sube al padre). Esto reproduce, salto a salto, el único camino del árbol. Verificado contra el manager oficial: respuesta correcta en todas las rutas de los \(103\) tests, con etiqueta máxima global \(999\).

Pseudocódigo 
FUNCION label(n, k, u, v):
    construir adyacencia ady[] a partir de las aristas (u[i], v[i])
    contador <- 0                       // un solo contador compartido
    etiquetas <- arreglo de tamano n
    DFS(raiz = 0, padre = -1, prof = 0)
    devolver etiquetas

PROCEDIMIENTO DFS(x, padre, prof):
    SI prof es PAR:                      // tiempo de ENTRADA (pre-orden)
        etiquetas[x] <- contador
        contador <- contador + 1
    PARA cada vecino y de x con y != padre:
        DFS(y, x, prof + 1)
    SI prof es IMPAR:                    // tiempo de SALIDA (post-orden)
        etiquetas[x] <- contador
        contador <- contador + 1

FUNCION find_next_station(s, t, c):      // c viene ORDENADA ascendente
    SI s < c[0]:                         // s tiene etiqueta de ENTRADA (nivel par)
        padre        <- c[ultimo]        // vecino mayor = padre
        max_subarbol <- c[ultimo]        // descendiente de etiqueta maxima
        SI t < s O t > max_subarbol:      // t fuera del subarbol de s
            devolver padre
        // t dentro: ir al hijo de MENOR etiqueta que sea >= t  (lower_bound)
        devolver el primer c[i] con c[i] >= t
    SINO:                                // s tiene etiqueta de SALIDA (nivel impar)
        padre <- c[0]                    // vecino menor = padre
        SI t > s O t <= padre:            // t fuera del subarbol de s
            devolver padre
        // t dentro: ir al hijo de MAYOR etiqueta que sea <= t  (upper_bound - 1)
        devolver el ultimo c[i] con c[i] <= t

NOTA: nada se guarda en variables globales entre label y find_next_station;
toda la decision usa solo s, t y c. La etiqueta maxima es n-1 (<= 1000),
asi que el bound pleno sale directo, sin minimizar por busqueda.

Complejidad: label es \(O(n)\) (un DFS más \(O(n-1)\) para armar la adyacencia); etiquetas en \([0, n-1]\). find_next_station es \(O(\deg(s))\), u \(O(\log \deg(s))\) con búsqueda binaria sobre \(c\) (que ya viene ordenada). Sobre todas las consultas de un caso, \(O(\sum |c|) \le O(100\,000)\). Memoria \(O(n)\). Con \(n \le 1000\) la recursión del DFS no corre riesgo de desbordar la pila.

Truco de C++ (opcional) 
Como c llega ordenado, usa busqueda binaria de la STL:
  - hijo de menor etiqueta >= t   ->  std::lower_bound(c.begin(), c.end(), t)
  - hijo de mayor etiqueta <= t   ->  std::upper_bound(c.begin(), c.end(), t) - 1
Esto baja cada consulta a O(log deg(s)). No es necesario para pasar
(con O(deg) basta), pero ayuda cuando algun nodo tiene grado grande.
