Tarea 6 — Game

PDF original (oficial): statement.pdf

El problema

Juan administra una mina de oro que tiene la forma de una rejilla rectangular de \(R\) filas por \(C\) columnas. Las filas se numeran de \(0\) a \(R-1\) y las columnas de \(0\) a \(C-1\), así que cada celda se identifica por un par \((P, Q)\) con \(0 \le P \le R-1\) y \(0 \le Q \le C-1\).

Cada celda guarda una cantidad de oro, que es un entero no negativo. Al principio todas las celdas valen \(0\) (no se ha medido oro en ninguna). Conforme la mina se explora, ocurren dos tipos de eventos, que pueden intercalarse en cualquier orden:

  • Actualizar (update). Se mide la celda \((P, Q)\) y se determina que contiene exactamente \(K\) unidades de oro. Esto asigna el valor \(K\) a esa celda (sobrescribe lo que tuviera antes; no se suma).
  • Consultar (calculate). Para un subrectángulo dado por sus esquinas \((P, Q)\) (arriba-izquierda) y \((U, V)\) (abajo-derecha), con \(P \le U\) y \(Q \le V\), hay que reportar el máximo común divisor (GCD) de las cantidades de oro de todas las celdas dentro de ese subrectángulo \([P..U] \times [Q..V]\).

Para el GCD se usa la convención estándar con el \(0\) como elemento neutro: \(\gcd(x, 0) = x\) y \(\gcd(0, 0) = 0\). Por lo tanto las celdas que todavía valen \(0\) no afectan el resultado de una consulta, y si todas las celdas del subrectángulo valen \(0\), la respuesta es \(0\).

Detalles de implementación

Esta es una tarea interactiva / reactiva (online): no lees de la entrada estándar ni escribes en la salida estándar. Debes entregar un archivo (game.c, game.cpp o game.pas) que implemente tres funciones. El evaluador (grader) las llama en el orden en que ocurren los eventos y no te deja ver las llamadas futuras: cada calculate debe responderse de inmediato, con la información recibida hasta ese momento.

  • init(R, C) — se llama una sola vez, al inicio, con las dimensiones de la mina. Aquí preparas tus estructuras. Después de esto la rejilla está completamente en \(0\).
  • update(P, Q, K) — asigna \(K\) unidades de oro a la celda \((P, Q)\). Se puede llamar varias veces sobre la misma celda; vale la última asignación.
  • calculate(P, Q, U, V) — regresa el GCD de todas las celdas del subrectángulo \([P..U] \times [Q..V]\).
Firmas a implementar 
// C / C++
void      init(int R, int C);
void      update(int P, int Q, long long K);
long long calculate(int P, int Q, int U, int V);

// Pascal
procedure init(R, C : LongInt);
procedure update(P, Q : LongInt; K : Int64);
function  calculate(P, Q, U, V : LongInt) : Int64;
Evaluador de muestra. El grader incluido lee primero una línea con R C N (dimensiones y número total de eventos). Luego lee \(N\) líneas: una que empieza con 1 P Q K es un update; una que empieza con 2 P Q U V es un calculate. Por cada calculate imprime, en su propia línea, el valor que regresa tu función. Tú no tocas stdin/stdout: de eso se encarga el grader.

Ejemplo

Secuencia de llamadas

init(2, 3)            // R=2, C=3

update(0, 0, 20)
update(0, 2, 15)
update(1, 1, 12)
calculate(0, 0, 0, 2)   ->  5
calculate(0, 0, 1, 1)   ->  4
update(0, 1, 6)
update(1, 1, 14)        // sobrescribe el 12
calculate(0, 0, 0, 2)   ->  1
calculate(0, 0, 1, 1)   ->  2

Salida del grader

5
4
1
2

Tras los tres primeros update la rejilla es \(\begin{bmatrix}20 & 0 & 15\\ 0 & 12 & 0\end{bmatrix}\). La primera consulta cubre toda la fila \(0\): \(\gcd(20, 0, 15) = \gcd(20, 15) = 5\). La segunda cubre \((0,0),(0,1),(1,0),(1,1)\): \(\gcd(20, 0, 0, 12) = \gcd(20, 12) = 4\). Después de poner \(6\) en \((0,1)\) y \(14\) en \((1,1)\), la fila \(0\) da \(\gcd(20, 6, 15) = \gcd(2, 15) = 1\), y el último rectángulo da \(\gcd(20, 6, 0, 14) = \gcd(2, 14) = 2\).

Resumen

Tienes una rejilla \(R\times C\) (hasta \(10^9\times 10^9\)) de enteros, toda en \(0\) al inicio. Debes implementar update(P,Q,K) que asigna el valor \(K\) a la celda \((P,Q)\), y calculate(P,Q,U,V) que regresa el GCD de todas las celdas del subrectángulo \([P..U]\times[Q..V]\). La API es online: las llamadas llegan en orden y respondes cada consulta sin ver el futuro.

Lo que importa de los límites El GCD es asociativo y tiene al \(0\) como neutro (\(\gcd(x,0)=x\)): es una agregación perfecta para un árbol de segmentos. \(R, C \le 1000\) deja meter una matriz; pero \(R=C=10^9\) prohíbe la matriz (\(10^{18}\) celdas) y, al ser online, tampoco puedes comprimir coordenadas. Con \(\le\!22\,000\) updates y \(\le\!250\,000\) consultas, una estructura de costo \(O(\log R\cdot\log C)\) por operación va sobradísima para los 13 s.

Pistas

Pista 1 — ¿Qué agregación necesitas?

¿El GCD se porta como una "suma" que puedas precomputar por bloques? ¿Es asociativo? ¿Qué pasa con las celdas que valen \(0\)?

El GCD cumple \(\gcd(a,\gcd(b,c))=\gcd(\gcd(a,b),c)\) y \(\gcd(x,0)=x\). Eso lo vuelve un monoide con neutro \(0\): exactamente lo que un árbol de segmentos sabe combinar. Una celda no escrita (valor \(0\)) no estorba. La pregunta real no es "¿qué estructura?" sino "¿cómo la hago bidimensional y que quepa cuando \(R,C\) son enormes?".

Pista 2 — Subtarea 1 (10 pts): \(R,C\) chicas

Si \(R\times C\) es pequeño, ¿por qué no guardar la matriz tal cual y, en cada consulta, recorrer el rectángulo celda por celda haciendo GCD?

Una matriz g[R][C]; update es \(O(1)\) y calculate recorre el subrectángulo en \(O((U-P+1)(V-Q+1))\). Con \(R,C\le 100\) y pocas operaciones eso basta para los primeros \(10\) puntos.

Pista 3 — Subtarea 2 (27 pts): \(R\) chica, \(C\) grande

Si solo hay un puñado de filas pero muchísimas columnas, ¿qué estructura 1D pondrías por columna? ¿Cómo combinas las pocas filas?

Para cada una de las \(R\) filas, un árbol de segmentos 1D sobre las columnas que soporte asignación puntual y GCD de un rango de columnas. Una consulta combina (con GCD) los resultados de las \(\le R\) filas involucradas. Como \(R\) es chica, recorrer las filas explícitas no duele. Esto ya te da la mitad de la idea: el eje de columnas se maneja con un segtree; falta hacer también el eje de filas con un segtree en vez de un bucle.

Pista 4 — Subtarea 3 (26 pts): \(R,C \le 1000\)

Ahora \(R\) ya no es chica, pero \(R\times C\) todavía cabe. ¿Cómo conviertes el bucle sobre filas en otra dimensión logarítmica?

Árbol de segmentos 2D denso: un segtree sobre filas, y cada nodo de filas tiene su propio segtree sobre columnas que guarda, para cada rango de columnas, el GCD de las celdas que caen en el rango de filas de ese nodo. update baja por el árbol de filas y actualiza una posición de columna en cada nodo del camino; calculate parte \([P,U]\) en \(O(\log R)\) nodos y consulta \([Q,V]\) en cada uno. Costo \(O(\log R\cdot\log C)\), memoria \(O(R\cdot C)\). Con \(R,C\le 1000\) todavía cabe.

Pista 5 — La observación clave (Subtareas 4–5: \(R=C=10^9\))

¿Por qué no puedes comprimir coordenadas aquí? Y si no compactas, ¿de dónde sale tanta memoria… y cómo la evitas?

Cuando haces update(P,Q,K), ¿cuántas posiciones de columna cambian realmente en cada nodo de filas? ¿Una sola, la columna \(Q\)?

Como la API es online, no conoces de antemano qué celdas se tocarán, así que comprimir coordenadas está prohibido: necesitas un segtree 2D disperso / dinámico que cree nodos bajo demanda. La clave de eficiencia: tras un update solo cambió la columna \(Q\), así que en cada nodo-fila ancestro basta recomputar esa columna: \(g=\gcd(\text{hijoIzq}@Q,\ \text{hijoDer}@Q)\) y asignarla. Fusionar árboles de columnas enteros sería \(O(C)\) y reventaría tiempo y memoria.

Pista final — Solución completa (100 puntos)

La misma estructura resuelve las 5 subtareas; lo único que cambia con la escala es que para \(R=C=10^9\) el árbol debe ser disperso (nodos creados bajo demanda), no una matriz.

Idea

Árbol de segmentos 2D disperso con agregación GCD:

  • Dimensión externa = filas \([0, R-1]\): un segtree disperso. Cada nodo-fila cubre un rango de filas y posee su propio segtree disperso de columnas \([0, C-1]\), que guarda, para cada rango de columnas, el GCD de las celdas que caen en el rango de filas de ese nodo.
  • update(P,Q,K): baja por el árbol de filas hasta la hoja \(P\) (\(O(\log R)\) nodos, creando los dinámicos que falten). En la hoja, asigna \(K\) en su segtree de columnas en la posición \(Q\) (\(O(\log C)\)). Al volver, en cada nodo-fila ancestro recomputa solo la columna \(Q\): \(g=\gcd(\text{hijoIzq.consultaCol}(Q),\ \text{hijoDer.consultaCol}(Q))\), y la asigna en su propio segtree de columnas. Costo \(O(\log R\cdot\log C)\).
  • calculate(P,Q,U,V): descompone \([P,U]\) en \(O(\log R)\) nodos-fila cubrientes; en cada uno consulta su segtree de columnas el rango \([Q,V]\) (\(O(\log C)\)); combina todo con GCD. Costo \(O(\log R\cdot\log C)\).

Por qué es correcto

El GCD es asociativo y conmutativo con neutro \(0\); por eso un nodo inexistente (nulo) devuelve \(0\) sin alterar la respuesta, y eso modela justo las celdas no escritas. Como update es una asignación de una sola celda, después de actualizar la hoja solo cambia el GCD de la columna \(Q\) en cada ancestro; recomputar únicamente esa columna (no fusionar árboles completos) mantiene el invariante "cada nodo guarda el GCD de su submatriz" y baja el costo a \(O(\log R\cdot\log C)\).

Pseudocódigo 
// Pools index-based (indice 0 = NULO). Reserva una sola vez.
//   columnas: cl[], cr[], cval[]                (segtree disperso por columnas)
//   filas:    lcRow[], rcRow[], colRoot[]       (segtree disperso por filas;
//             colRoot[nf] = raiz del segtree de columnas del nodo-fila nf)
// cval[0] = 0  -> un hijo nulo aporta el neutro del GCD.

funcion gcd(a, b):                 // 0 es la identidad
    mientras b != 0: (a, b) = (b, a mod b)
    regresa a

// ----- SEGTREE DE COLUMNAS (1D disperso sobre [0, C-1]) -----
funcion col_assign(nodo, lo, hi, q, val):    // asigna val en la columna q
    si nodo == 0: nodo = nuevo_nodo_col()     // crea bajo demanda
    si lo == hi: cval[nodo] = val; regresa nodo
    mid = (lo + hi) / 2
    si q <= mid: cl[nodo] = col_assign(cl[nodo], lo, mid,   q, val)
    si no:        cr[nodo] = col_assign(cr[nodo], mid+1, hi, q, val)
    cval[nodo] = gcd(cval[cl[nodo]], cval[cr[nodo]])   // hijo nulo -> 0
    regresa nodo

funcion col_query(nodo, lo, hi, a, b):       // GCD de columnas [a, b]
    si nodo == 0 o b < lo o hi < a: regresa 0
    si a <= lo y hi <= b: regresa cval[nodo]
    mid = (lo + hi) / 2
    regresa gcd(col_query(cl[nodo], lo, mid,   a, b),
                col_query(cr[nodo], mid+1, hi, a, b))

// ----- SEGTREE DE FILAS (1D disperso sobre [0, R-1]) -----
funcion row_update(nf, lo, hi, p, q, val):
    si nf == 0: nf = nuevo_nodo_fila()
    si lo == hi:                              // hoja: fila p exacta
        colRoot[nf] = col_assign(colRoot[nf], 0, C-1, q, val)
        regresa nf
    mid = (lo + hi) / 2
    si p <= mid: lcRow[nf] = row_update(lcRow[nf], lo, mid,   p, q, val)
    si no:        rcRow[nf] = row_update(rcRow[nf], mid+1, hi, p, q, val)
    // SOLO cambio la columna q -> recomputa esa columna en este nodo-fila:
    g = gcd(col_query(colRoot[lcRow[nf]], 0, C-1, q, q),
            col_query(colRoot[rcRow[nf]], 0, C-1, q, q))
    colRoot[nf] = col_assign(colRoot[nf], 0, C-1, q, g)
    regresa nf

funcion row_query(nf, lo, hi, p, u, q, v):   // filas [p,u] x columnas [q,v]
    si nf == 0 o u < lo o hi < p: regresa 0
    si p <= lo y hi <= u: regresa col_query(colRoot[nf], 0, C-1, q, v)
    mid = (lo + hi) / 2
    regresa gcd(row_query(lcRow[nf], lo, mid,   p, u, q, v),
                row_query(rcRow[nf], mid+1, hi, p, u, q, v))

// ----- API -----
init(R, C):       guarda R, C; rowRoot = 0; reserva los pools
update(P, Q, K):  rowRoot = row_update(rowRoot, 0, R-1, P, Q, K)
calculate(P,Q,U,V): regresa row_query(rowRoot, 0, R-1, P, U, Q, V)

Complejidad. init \(O(1)\); update y calculate ambas \(O(\log R\cdot\log C)\). Con \(\log R,\log C\approx 30\) y \(\le\!22\,000\) updates \(+\) \(\le\!250\,000\) consultas, son \(\sim\!2.3\times 10^8\) operaciones de GCD en el peor caso: holgadísimo para 13 s (medido \(\le 2.3\) s). Memoria: \(O(U\cdot\log R\cdot\log C)\) nodos.

Cómo subir de subtarea en subtarea

  • S1: matriz directa + recorrido del rectángulo.
  • S2: segtree de columnas por fila, combinando las pocas filas con GCD.
  • S3: segtree 2D denso (filas × columnas), \(O(R\cdot C)\) memoria.
  • S4–S5: el mismo segtree 2D pero disperso/dinámico (nodos bajo demanda) porque \(R=C=10^9\) y, al ser online, no se pueden comprimir coordenadas.

Trampas que cuestan puntos

  • GCD con \(0\) como identidad, no como "min": un nodo nulo aporta \(0\) y eso es correcto (celdas no escritas). Todo-cero \(\Rightarrow 0\).
  • update es asignación, no acumulación (el ejemplo pone \(14\) sobre el \(12\) en \((1,1)\)).
  • En cada ancestro recomputa solo la columna \(Q\); fusionar árboles completos sería \(O(C)\) y daría TLE/MLE.
  • No comprimas coordenadas: es online; el árbol disperso es obligatorio.
  • Tipos: \(K\le 10^{18}\) exige enteros de 64 bits; el GCD sobre 64 bits no se desborda (solo módulos).

Memoria: la parte fina del 100

El cuello de botella real no es el tiempo sino la memoria. Una versión ingenua con nodos de \(16\) bytes (\(2\) índices \(+\) un entero de \(64\) bits) alcanza \(\sim\!845\) nodos por update (\(=\log R\cdot\log C\), el máximo teórico) en el caso adversarial smallGCD: \(\sim\!18.6\) M nodos, \(\approx 287\) MB, que rebasa el presupuesto práctico de heap (\(\sim\!230\) MB tras el grader). El RSS se infla todavía más si dejas que std::vector haga realloc-doubling. Correcciones sin tocar el algoritmo ni su complejidad:

  • Reservar el pool de una sola vez (arreglo de tamaño fijo), para evitar el doblado transitorio del vector.
  • Compresión de cadenas (path-compression): los tramos de un solo hijo que genera el rango \([0,10^9]\) se colapsan, reduciendo el número de nodos drásticamente.
  • BBST / treap por eje: la profundidad pasa a \(\sim\!\log(\#updates)\approx 15\) en lugar de \(\log(10^9)\approx 30\), bajando el conteo de nodos por debajo del límite.
