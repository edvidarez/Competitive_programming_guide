Tarea 5 — Unscrambling a Messy Bug

Traducción aproximada del título: «Descifrando un bug travieso». PDF original (oficial): statement.pdf

El problema

Ilshat es un ingeniero de software que trabaja en estructuras de datos eficientes. Un día inventó una estructura nueva. Esta estructura puede guardar un conjunto de enteros no negativos de \(n\) bits, donde \(n\) es una potencia de dos; es decir, \(n = 2^{b}\) para algún entero no negativo \(b\).

La estructura empieza vacía. Un programa que la use debe seguir estas reglas:

  • El programa puede agregar elementos (enteros de \(n\) bits) a la estructura, uno por uno, llamando a la función add_element(x). Si intenta agregar un elemento que ya está presente, no pasa nada.
  • Después de agregar el último elemento, el programa debe llamar a la función compile_set() exactamente una vez.
  • Por último, el programa puede llamar a check_element(x) para comprobar si el elemento \(x\) está presente en el conjunto. Esta función se puede usar varias veces.

Cuando Ilshat implementó por primera vez esta estructura, cometió un bug en la función compile_set(). El bug reordena los dígitos binarios de cada elemento del conjunto de la misma manera. Ilshat quiere que encuentres el reordenamiento exacto de dígitos que causa el bug.

Formalmente, considera una sucesión \(p = [p_0, \dots, p_{n-1}]\) en la que cada número de \(0\) a \(n-1\) aparece exactamente una vez. A una sucesión así la llamamos permutación. Considera un elemento del conjunto cuyos dígitos en binario son \(a_0, \dots, a_{n-1}\) (con \(a_0\) siendo el bit más significativo). Cuando se llama a compile_set(), este elemento se reemplaza por el elemento \(a_{p_0}, a_{p_1}, \dots, a_{p_{n-1}}\).

La misma permutación \(p\) se usa para reordenar los dígitos de todos los elementos. Cualquier permutación es posible, incluyendo la posibilidad de que \(p_i = i\) para cada \(0 \le i \le n-1\).

Por ejemplo, supongamos que \(n = 4\), \(p = [2, 1, 3, 0]\), y que insertaste en el conjunto los enteros cuyas representaciones binarias son \(0000\), \(1100\) y \(0111\). Llamar a compile_set cambia estos elementos a \(0000\), \(0101\) y \(1110\), respectivamente.

Tu tarea es escribir un programa que encuentre la permutación \(p\) interactuando con la estructura de datos. Debe (en este orden):

  1. elegir un conjunto de enteros de \(n\) bits,
  2. insertar esos enteros en la estructura,
  3. llamar a la función compile_set para disparar el bug,
  4. comprobar la presencia de algunos elementos en el conjunto modificado,
  5. usar esa información para determinar y devolver la permutación \(p\).

Ten en cuenta que tu programa puede llamar a compile_set solo una vez.

Además, hay un límite en cuántas veces tu programa llama a las funciones de la biblioteca. En concreto, puede:

  • llamar a add_element a lo más \(w\) veces (\(w\) es por «writes», escrituras),
  • llamar a check_element a lo más \(r\) veces (\(r\) es por «reads», lecturas).

Detalles de implementación

Debes implementar una función (método):

Firma a implementar 
int[] restore_permutation(int n, int w, int r)
  • \(n\): el número de bits en la representación binaria de cada elemento del conjunto (y también la longitud de \(p\)).
  • \(w\): el máximo número de operaciones add_element que tu programa puede realizar.
  • \(r\): el máximo número de operaciones check_element que tu programa puede realizar.
  • la función debe devolver la permutación restaurada \(p\).

En el lenguaje C el prototipo es un poco distinto: en lugar de devolver el arreglo, lo escribes en uno provisto.

Variante en C 
void restore_permutation(int n, int w, int r, int* result)

\(n\), \(w\) y \(r\) tienen el mismo significado; para cada \(i\), debes guardar el valor \(p_i\) en result[i].

Funciones de la biblioteca

Para interactuar con la estructura, tu programa debe usar estas tres funciones (métodos):

API provista por el evaluador 
void    add_element(string x)
void    compile_set()
boolean check_element(string x)
  • add_element(x): agrega al conjunto el elemento descrito por \(x\), una cadena de caracteres '0' y '1' que da la representación binaria del entero a agregar. La longitud de \(x\) debe ser \(n\).
  • compile_set(): debe llamarse exactamente una vez. Tu programa no puede llamar a add_element() después de esta llamada, ni a check_element() antes de ella.
  • check_element(x): comprueba si el elemento \(x\) está en el conjunto modificado. \(x\) es una cadena de '0' y '1' de longitud \(n\). Devuelve true si \(x\) está en el conjunto modificado, y false en caso contrario.

Si tu programa viola cualquiera de estas restricciones, el resultado de calificación será «Wrong Answer».

Para todas las cadenas, el primer carácter da el bit más significativo del entero correspondiente. El evaluador fija la permutación \(p\) antes de llamar a restore_permutation.

Resumen

Un evaluador fija una permutación oculta \(p\). Tú implementas restore_permutation(n, w, r). Primero eliges y agregas cadenas de \(n\) bits con add_element (a lo más \(w\)). Luego llamas compile_set() una vez: a partir de ahí, cada elemento guardado \(a_0\dots a_{n-1}\) queda transformado en \(a_{p_0}\dots a_{p_{n-1}}\) (la posición de salida \(i\) recibe el bit que estaba en la posición de entrada \(p_i\)). Por último puedes preguntar con check_element (a lo más \(r\)) si una cadena está en el conjunto modificado. Con esas respuestas debes reconstruir \(p\).

Lo que importa de los límites El costo no es el tiempo, sino \(w\) y \(r\). La subtarea 5 (\(n = 128\), \(w = r = 896\)) exige un esquema \(O(n\log n)\): \(\tfrac{n}{2}\log_2 n = 448\) escrituras y \(n\log_2 n = 896\) lecturas. Como \(\log_2 128 = 7\), eso da exactamente \(448\) y \(896\). La subtarea 1 (\(p\) difiere de la identidad en \(\le 2\) índices) invita a fuerza bruta; la 4 (\(w = r = 1792\)) permite el doble de holgura que la 5.

Pistas

Pista 1 — La restricción que lo cambia todo

¿Notaste que compile_set() se llama exactamente una vez, y que todas las escrituras van antes y todas las lecturas van después?

Eso parte el problema en dos fases globales que no se pueden mezclar: una fase de escritura en la que diseñas el conjunto sin conocer todavía ninguna respuesta, y una fase de lectura en la que ya puedes ser adaptativo (usar lo que respondió check_element para decidir la siguiente pregunta). ¿Qué información puedes «codificar» en las cadenas que agregas, de modo que después una simple pregunta de pertenencia te revele algo sobre \(p\)?

¿Y qué hace exactamente el bug? Si la entrada tiene un '1' en la posición \(j\), ¿en qué posición de salida aparece ese '1' después de compilar?

Conviene fijar la convención: la salida \(i\) toma el bit de la entrada \(p_i\), o sea \(t_i = a_{p_i}\). Trabaja directamente con índices de cadena \(0\dots n-1\), sin invertir nada.

Pista 2 — Subtarea 1 (n=8, p casi la identidad): fuerza bruta

Si te prometen que \(p\) difiere de la identidad en a lo más \(2\) posiciones, ¿no basta con probar todas las posibles «transposiciones»?

Con \(n = 8\) hay solo \(\binom{8}{2} = 28\) pares de posiciones. Puedes agregar pocas cadenas «testigo» (por ejemplo, las que tienen un solo bit encendido) y, después de compilar, hacer preguntas que distingan «¿el bit \(i\) se quedó en su lugar o se intercambió con el \(j\)?». Con \(w = r = 256\) te sobra muchísimo presupuesto. Es la forma de asegurar los primeros \(20\) puntos sin la idea general.

¿Te das cuenta de que esta subtarea solo te pide una detección local de cambios, no reconstruir toda una permutación arbitraria?

Pista 3 — Subtareas 2 y 3 (n=32): el compromiso entre w y r

Para \(p\) arbitraria, ¿cómo localizarías el origen de cada bit de salida?

Una idea directa: en la escritura agregas \(n\) «prefijos de unos» (la cadena con los primeros \(k\) bits en '1' y el resto en '0', para cada \(k\)). Eso son \(O(n)\) escrituras. En la lectura, para cada posición de salida vas preguntando hasta ubicar de qué bit de entrada proviene; eso cuesta del orden de \(n^2/2\) lecturas.

Mira los presupuestos: la subtarea 2 da \(w = 320\) (apretado en escrituras) pero \(r = 1024\) (holgado en lecturas) — favorece pocas escrituras y muchas lecturas. La subtarea 3 invierte la balanza: \(w = 1024\), \(r = 320\) — necesitas recortar lecturas.

¿No te sugiere ese intercambio que la solución ideal debería gastar pocas y deterministas lecturas, sin depender del azar? ¿Qué estructura recursiva gastaría solo \(O(n\log n)\) en cada fase?

Pista 4 — La pregunta de pertenencia que decide «mitad izquierda o derecha»

Piensa en dividir y vencer sobre las posiciones. Toma un subconjunto \(B\) (ordenado) de índices de entrada y su mitad izquierda \(A\). En la escritura, por cada índice \(i \in A\) agregas una cadena que tiene: '1' en \(i\), '0' en los demás índices de \(B\), y '1' en todo lo que está fuera de \(B\) (un «contexto fijo» que hace únicos a los elementos entre subproblemas distintos).

Después de compilar, ¿qué pinta tiene ese elemento? Sus unos caen en \(\{i\}\) más el complemento de \(B\), pero movidos por \(p\): la posición de salida \(k\) vale \(1\) sii \(p_k \in \{i\}\cup (\text{complemento de }B)\). Como \(p\) es biyección, hay una sola posición \(k^\ast\) con \(p_{k^\ast} = i\).

En la lectura, para una posición de salida \(k\) construyes una consulta con '1' en todas las posiciones de salida que no están en tu subproblema (el mismo «contexto fijo» visto del lado de salida), '0' en las demás, y un '1' extra en \(k\). Esa consulta coincide con algún elemento compilado sii \(p_k\) cayó en la mitad izquierda \(A\).

¿Ves entonces lo que significa la respuesta? check_element verdadero \(\Rightarrow\) «\(p_k\) está en \(A\)»; falso \(\Rightarrow\) «\(p_k\) está en la mitad derecha». ¿Y si recurres en cada mitad hasta llegar a conjuntos de tamaño \(1\)?

Pista final — Solución completa (100 puntos)

Idea. Un único algoritmo de divide y vencerás \(O(n\log n)\) resuelve las cinco subtareas. Como compile_set se llama una sola vez, separamos todas las escrituras (no adaptativas, planificadas solo con índices de entrada que tú controlas), luego compile_set, y después todas las lecturas (adaptativas, que descienden por el árbol de recursión).

Fase de escritura. En un nodo que cubre el subconjunto ordenado \(B\) de índices de entrada (\(|B| = m \ge 2\)), sea \(A\) la primera mitad de \(B\). Construyes una «base» con '1' en todos los índices fuera de \(B\) y '0' dentro de \(B\). Por cada \(i \in A\) agregas la base con el bit \(i\) encendido (exactamente un bit izquierdo de \(B\) en \(1\); los demás bits de \(B\) en \(0\); el contexto externo en \(1\)). Luego recurres en \(A\) y en la segunda mitad \(C\).

Fase de lectura (adaptativa). Cada nodo maneja un par \((P, B)\): \(P\) son posiciones de salida y \(B\) los bits de entrada correspondientes, con la invariante \(\{p_k : k \in P\} = B\) y \(|P| = |B|\). Para cada \(k \in P\) haces una consulta con '1' en todas las posiciones de salida fuera de \(P\), '0' en las demás de \(P\), y un '1' extra en \(k\). Si check_element es verdadero, \(p_k\) está en la mitad izquierda \(A\) (grupo izquierdo); si es falso, está en la mitad derecha \(C\). Como \(p\) es biyección, el grupo izquierdo tiene exactamente \(|A|\) elementos. Recurres en \((\text{grupo izquierdo}, A)\) y \((\text{grupo derecho}, C)\).

Caso base. Cuando \(|P| = 1\), el único índice de entrada \(B_0\) es el origen de la única posición de salida \(P_0\), así que \(p_{P_0} = B_0\).

Por qué es correcto. La transformación del evaluador es \(t_i = a_{p_i}\). El elemento escrito para el bit \(i \in A\) compila a unos en (complemento de \(P\)) más la única posición \(k^\ast\) con \(p_{k^\ast} = i\); la consulta de lectura para \(k\) pone unos en (complemento de \(P\)) más la posición \(k\). Ambos patrones coinciden sii \(k = k^\ast\), es decir \(p_k = i\) para algún \(i \in A\) — equivalente a «\(p_k\) está en la mitad izquierda». La correspondencia «complemento de \(B\) en entrada» \(=\) «complemento de \(P\) en salida» se sigue de la invariante. En el caso base la biyección obliga a \(p_{P_0} = B_0\). Por inducción sobre el tamaño del nodo, cada \(p_k\) se fija al valor correcto. Como el enunciado garantiza que la permutación consistente es única, la salida iguala exactamente la \(p\) oculta.

Detalle fino. Es esencial que la escritura y la lectura partan \(B\) de forma idéntica (misma lista ordenada, misma mitad izquierda) para que el caso base sea coherente.

Pseudocódigo 
// Convención del evaluador: salida[i] = entrada[p[i]].
// Trabajamos con índices de cadena. perm[k] guardará p[k]. N = n global.

funcion writePhase(B):            // B: lista ORDENADA de índices de entrada
    m = |B|
    si m == 1: regresar
    half = m / 2
    base = cadena de longitud N
    para cada posicion j en 0..N-1:
        base[j] = '1' si j NO esta en B, si no '0'
    para t en 0..half-1:
        e = base
        e[B[t]] = '1'             // un bit de la mitad izquierda encendido
        add_element(e)
    A = B[0 .. half-1];  C = B[half .. m-1]
    writePhase(A);  writePhase(C)

funcion readPhase(P, B):          // P: posiciones de salida; B: bits de entrada; |P| = |B|
    m = |P|
    si m == 1: perm[P[0]] = B[0]; regresar
    half = m / 2
    base = cadena de longitud N
    para cada posicion j en 0..N-1:
        base[j] = '1' si j NO esta en P, si no '0'
    grupoIzq = []; grupoDer = []
    para cada k en P:
        q = base; q[k] = '1'
        si check_element(q): grupoIzq.agregar(k)   // p[k] en mitad izquierda
        si no:               grupoDer.agregar(k)   // p[k] en mitad derecha
    A = B[0 .. half-1];  C = B[half .. m-1]
    readPhase(grupoIzq, A)
    readPhase(grupoDer, C)

funcion restore_permutation(n, w, r):
    N = n;  perm = arreglo de tamaño n
    todos = [0, 1, ..., n-1]
    writePhase(todos)
    compile_set()
    readPhase(todos, todos)
    regresa perm

Complejidad. Operaciones de biblioteca: \(\tfrac{n}{2}\log_2 n\) escrituras y \(n\log_2 n\) lecturas. Trabajo de cómputo \(O(n^2 \log n)\) por construir cadenas de longitud \(n\); memoria \(O(n)\) más pila \(O(\log n)\). Para \(n = 128\): \(448\) escrituras y \(896\) lecturas — caben holgadas en la subtarea 4 (\(1792\) cada una) y justo en la 5 (\(896\) cada una). Por eso una sola implementación se lleva los \(100\) puntos.

Escalera de subtareas. S1 (\(p\) casi la identidad): este mismo D&V gasta solo \(12\) escrituras y \(24\) lecturas. S2 (\(w = 320\), \(r = 1024\)): el método directo de \(O(n)\) escrituras y \(\sim n^2/2 = 496\) lecturas también pasa; el D&V usa \(80\) y \(160\). S3 (\(w = 1024\), \(r = 320\)): el directo debe recortar lecturas (barajar el orden da \(\sim n^2/4 = 248\) en promedio); el D&V ya usa \(160\) deterministas. S4 y S5 (\(n = 128\)) exigen \(O(n\log n)\): el D&V produce \(448\) y \(896\) exactos.

Gotchas. (1) compile_set va exactamente una vez: todas las add_element antes, todas las check_element después. (2) x[0] es el bit más significativo y el bug hace \(t_i = a_{p_i}\); se trabaja con índices de cadena directamente, sin invertir. (3) En la subtarea 5 las lecturas son \(896 = r\) exactamente; el evaluador falla si se baja de \(0\), pero \(896\) llamadas dejan \(r\) en \(0\) (válido). (4) Verificar por igualdad con la \(p\) de entrada es legítimo porque la permutación consistente es única.
