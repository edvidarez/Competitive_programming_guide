Tarea 8 — Save It

Guárdalo (codifica distancias a los hubs) · PDF original (oficial): statement.pdf

El problema

La empresa de mensajería Xedef Courier Company ofrece entrega aérea de paquetes entre varias ciudades. Algunas de esas ciudades son hubs de Xedef, donde hay instalaciones especiales de procesamiento. Cada avión de Xedef hace viajes de ida y vuelta entre un par de ciudades, llevando paquetes en cualquiera de los dos sentidos según se necesite.

Para enviarse de una ciudad a otra, un paquete debe transportarse mediante una secuencia de saltos (hops), donde cada salto lleva el paquete entre un par de ciudades servidas por uno de los aviones. Además, la secuencia debe incluir al menos uno de los hubs de Xedef.

Para facilitar el ruteo, Xedef quiere codificar, en la etiqueta de envío de cada paquete, la longitud de la secuencia más corta de saltos desde cada ciudad hasta cada hub. (La longitud de la secuencia más corta que va de un hub a sí mismo es cero.) Como es de esperarse, hace falta una representación compacta de esta información.

Debes implementar dos procedimientos, encode(N,H,P,A,B) y decode(N,H). \(N\) es el número de ciudades y \(H\) el número de hubs. Supón que las ciudades están numeradas de \(0\) a \(N-1\), y que los hubs son las ciudades con números entre \(0\) y \(H-1\). Supón además que \(N \le 1000\) y \(H \le 36\). \(P\) es el número de pares de ciudades conectados por avión. Todos los pares (no ordenados) de ciudades conectados son distintos. \(A\) y \(B\) son arreglos de tamaño \(P\), tales que el primer par de ciudades conectado es \((A[0],B[0])\), el segundo es \((A[1],B[1])\), y así sucesivamente.

Detalles de implementación

Esta es una tarea de comunicación / codificación: se ejecutan dos corridas independientes. En la primera corre encode, que ve el grafo y emite un flujo de bits; en la segunda corre decode, que no ve el grafo y solo recibe ese flujo de bits. La calificación mide cuántos bits emitió encode.

Firmas a implementar por el concursante 
void encode(int N, int H, int P, int *A, int *B);   // N ciudades, H hubs, P aristas (A[i],B[i])
void decode(int N, int H);
Funciones provistas por el evaluador (grader) 
void encode_bit(int b);   // b en {0,1}; encode emite bits llamando a esto
int  decode_bit(void);    // devuelve el i-esimo bit emitido por encode_bit
void hops(int h, int c, int d);   // decode reporta: del hub h a la ciudad c hay d saltos

Qué debe hacer encode: calcular una secuencia de bits a partir de la cual decode pueda determinar el número de saltos desde cada ciudad hasta cada hub. encode transmite la secuencia al servidor de calificación mediante una sucesión de llamadas a encode_bit(b), con \(b\) igual a \(0\) o \(1\).

Qué debe hacer decode: recibe la secuencia de bits del servidor llamando a decode_bit. La \(i\)-ésima llamada a decode_bit devuelve el valor \(b\) de la \(i\)-ésima llamada a encode_bit(b). Importante: debes garantizar que el número de veces que decode llama a decode_bit nunca exceda el número de veces que encode previamente llamó a encode_bit.

Después de decodificar los números de saltos, decode debe llamar a hops(h,c,d) para todo hub \(h\) y toda ciudad \(c\) (incluyendo todo hub, es decir, también para \(c=h\)), entregando el número mínimo \(d\) de saltos necesarios para enviar un paquete entre \(h\) y \(c\). Es decir, debe haber exactamente \(N\cdot H\) llamadas a hops(h,c,d). El orden no importa. Se garantiza que siempre es posible enviar un paquete entre cada hub y cada ciudad.

Nota: encode y decode deben comunicarse únicamente a través de la interfaz especificada. Están prohibidas las variables compartidas, el acceso a archivos y el acceso a red. En C o C++ puedes declarar las variables persistentes como static para que encode o decode conserven información sin compartirla. En Pascal, puedes declararlas en la parte implementation de tus archivos.

Resumen

Grafo no ponderado de \(N\le 1000\) ciudades; los hubs son los nodos \(0\dots H-1\) con \(H\le 36\). encode ve el grafo y escribe un flujo de bits; decode no ve el grafo y, leyendo solo ese flujo, debe reportar con hops(h,c,d) la distancia (en saltos) de cada hub \(h\) a cada ciudad \(c\). Hay que lograrlo con muy pocos bits: \(\le 70\,000\) para los 100 puntos.

Lo que importa de los límites El grafo es no ponderado ⇒ "distancia" es BFS. La tabla de respuestas tiene \(N\cdot H\) números \(\le N-1\); guardarla cruda son \(\sim 360\,000\) bits (subtarea 2). Bajar de ahí a \(70\,000\) exige no mandar números absolutos, sino diferencias, y empaquetarlas bien. Cada bloque de 25 puntos esconde exactamente una de esas ideas.

Pistas

Pista 1 — ¿Cómo modelar la "distancia"?

El enunciado dice "secuencia más corta de saltos". En un grafo donde todas las aristas cuestan lo mismo, ¿qué algoritmo da esa distancia? ¿Y cuántas distancias distintas tienes que entregar en total?

Como ningún avión "pesa" más que otro, la distancia de un hub a una ciudad es simplemente el número de aristas del camino más corto: un BFS desde el hub. Tienes \(H\) hubs, así que toda la información es una tabla de \(H\times N\) distancias (cada una entre \(0\) y \(N-1\)). El reto no es calcularla (eso es fácil para encode), sino transmitirla con poquísimos bits a un decode que no tiene el grafo.

Pista 2 — Subtarea 1 (≤ 16 000 000 bits): lo más burdo que sí cabe

¿Cuál es la cosa más tonta que decode podría reconstruir si le mandaras el grafo entero? ¿Cuántos bits cuesta describir el grafo?

Con \(16\) millones de bits te alcanza para mandar el grafo crudo: la matriz de adyacencia son \(N^2 \le 10^6\) bits (un bit por par de ciudades). decode lee la matriz, reconstruye las listas de adyacencia y corre \(H\) BFS por su cuenta para sacar todas las distancias. Costoso en bits, pero correcto: 25 puntos.

Pista 3 — Subtarea 2 (≤ 360 000 bits): manda la respuesta, no el grafo

Si encode ya conoce la tabla de distancias, ¿para qué mandar el grafo? ¿Cuántos bits necesita cada número de la tabla?

Cada distancia está entre \(0\) y \(N-1 \le 999\), así que cabe en \(\lceil\log_2 N\rceil = 10\) bits. Manda la tabla directa: \(H\cdot N\cdot 10 = 36\cdot 1000\cdot 10 = 360\,000\) bits. decode solo lee y reporta con hops; ni siquiera necesita el grafo. Justo en el límite: 25 puntos más.

A partir de aquí toca comprimir: \(10\) bits por número es demasiado. ¿Y si los números vecinos casi no cambian?

Pista 4 — Subtarea 3 (≤ 80 000 bits): la observación ternaria

Fija un hub \(s\) y dos ciudades \(u,v\) unidas por una arista. ¿Cuánto pueden diferir \(\operatorname{dist}_s(u)\) y \(\operatorname{dist}_s(v)\)? Por la desigualdad del triángulo en un grafo no ponderado…

A lo largo de cualquier arista \((u,v)\), para cualquier fuente \(s\) se cumple \(|\operatorname{dist}_s(u)-\operatorname{dist}_s(v)|\le 1\). Entonces, si fijas un árbol generador del grafo y recorres sus aristas \((\text{padre}(v), v)\), la diferencia \(\operatorname{dist}_s(v)-\operatorname{dist}_s(\text{padre}(v))\) siempre vale \(-1\), \(0\) o \(+1\): un dígito ternario (un "trit"), que codificamos como \(0,1,2\).

Plan: encode hace BFS desde el nodo 0, fija ese árbol generador y transmite el árbol (el padre de cada \(v=1\dots N-1\), \(10\) bits c/u, porque decode no tiene el grafo). Con el árbol, las distancias del hub 0 salen gratis: son las profundidades. Para cada otro hub \(h=1\dots H-1\) manda un trit por nodo no raíz (uno por arista del árbol, \(v=1\dots N-1\)). A \(2\) bits por trit: \((N-1)\cdot 10 + (H-1)(N-1)\cdot 2 = 9990 + 69930 = 79920 \le 80\,000\). 25 puntos más, pero raspando el límite.

Pista 5 — Subtarea 4 (≤ 70 000 bits): empaqueta 3 trits en 5 bits

Gastar \(2\) bits por trit desperdicia: \(2\) bits guardan \(4\) valores y un trit solo tiene \(3\). ¿Cuántos trits caben en \(5\) bits si los agrupas?

Tres trits tienen \(3^3 = 27\) combinaciones, y \(5\) bits dan \(2^5 = 32 \ge 27\). Así que junta los trits de \(3\) en \(3\) y manda \(5\) bits por grupo: pasas de \(2\) a \(5/3 \approx 1.667\) bits por trit. Esa es exactamente la diferencia entre la subtarea 3 y la 4.

Cuidado con un detalle: ¿el relleno (padding) del último grupo va por cada hub o una sola vez al final? Si rellenas por hub, los límites de bloque se desalinean.

Concatena todos los trits (de todos los hubs) en un único flujo, empaca ese flujo completo de 3 en 3 (rellenando solo al final), y del lado de decode rebana en bloques de \(N-1\) trits por hub. La solución completa está abajo.

Pista final — Solución completa (100 puntos)

Una sola solución cumple el límite más estricto (subtarea 4) y, por tanto, las cuatro: \(68\,265\) bits en el peor caso \((N=1000,\ H=36)\), por debajo de \(70\,000\).

Idea

  • El grafo es no ponderado ⇒ distancias por BFS, y a lo largo de cualquier arista las distancias a una misma fuente difieren en \(\{-1,0,+1\}\).
  • Usa un árbol generador (BFS desde el nodo 0). decode no tiene el grafo, así que hay que transmitir el árbol (el padre de cada \(v\), \(W=\lceil\log_2 N\rceil = 10\) bits c/u). A cambio, las distancias del hub 0 salen gratis como profundidades del árbol.
  • Para cada hub \(h\ge 1\), manda un trit por nodo no raíz (\(v=1\dots N-1\), uno por arista del árbol): \(t_v=\operatorname{dist}_h(v)-\operatorname{dist}_h(\text{padre}(v))+1\). Empaca los trits de 3 en 3 en 5 bits.
  • No mandes ninguna distancia base por hub: ancla con el hecho conocido \(\operatorname{dist}_h(h)=0\) en el nodo hub.
Pseudocódigo — encode(N, H, P, A, B) 
construir listas de adyacencia desde (A, B)
BFS(0) -> par[v] (padre en el arbol), prof[v] (= dist_0(v))   // arbol generador, raiz 0
W = ceil(log2(N))                                             // 10 bits si N = 1000

// 1) transmitir el arbol  (las distancias del hub 0 = prof[], gratis)
para v = 1 .. N-1:
    emitir par[v] en W bits      // W llamadas a encode_bit

// 2) reunir un trit por nodo no raiz (v=1..N-1), hub por hub
trits = []                       // orden: hub 1, hub 2, ...; dentro, v = 1 .. N-1
para h = 1 .. H-1:
    BFS(h) -> dist_h[]
    para v = 1 .. N-1:
        trits.append( dist_h[v] - dist_h[par[v]] + 1 )   // en {0,1,2}

// 3) empaquetar TODO el flujo de 3 en 3 (rellenar con 0 solo al final)
para cada grupo (t0, t1, t2) de trits:
    val = 9*t0 + 3*t1 + t2       // 0 .. 26
    emitir val en 5 bits
Pseudocódigo — decode(N, H) 
W = ceil(log2(N))
// 1) leer el arbol (mismo ancho de bits)
para v = 1 .. N-1: par[v] = leer W bits        // par[0] = ninguno
construir hijos[] y un orden BFS desde 0 (padre antes que hijo)
prof[0] = 0;  para v en orden BFS (v != 0): prof[v] = prof[par[v]] + 1

// 2) hub 0: distancias = profundidades
para c = 0 .. N-1: hops(0, c, prof[c])

// 3) leer los grupos de 5 bits y reconstruir TODOS los trits
numTrits = (H-1) * (N-1)
para cada grupo: val = leer 5 bits; (t0,t1,t2) = (val/9, (val/3)%3, val%3)
trits[] = primeros numTrits trits reconstruidos

// 4) por cada hub h >= 1, propagar en el arbol y anclar con dist_h(h)=0
para h = 1 .. H-1:
    bloque = trits[(h-1)*(N-1) .. h*(N-1)-1]    // longitud N-1; bloque[v-1] = trit del nodo v
    rel[0] = 0
    para v en orden BFS (v != 0):
        rel[v] = rel[par[v]] + (bloque[v-1] - 1)  // rel[v] = dist_h(v) - dist_h(0)
    off = -rel[h]                                 // porque dist_h(h) = 0
    para c = 0 .. N-1: hops(h, c, rel[c] + off)   // dist_h(c) = rel[c] - rel[h]

Por qué es correcto

Para un hub fijo \(h\), define \(\operatorname{rel}(v)=\operatorname{dist}_h(v)-\operatorname{dist}_h(0)\). Sobre cada arista del árbol, \(\operatorname{dist}_h(v)-\operatorname{dist}_h(\text{padre}(v)) = t_v-1 \in\{-1,0,+1\}\), así que propagando desde la raíz obtenemos exactamente \(\operatorname{rel}(v)\) para todo nodo. Como \(\operatorname{dist}_h(h)=0\), el desfase real es \(\operatorname{off}=-\operatorname{rel}(h)=\operatorname{dist}_h(0)\), y entonces \(\operatorname{dist}_h(c)=\operatorname{rel}(c)-\operatorname{rel}(h)\) es la distancia verdadera. No hace falta mandar ninguna distancia base.

Trampas a evitar:
  • El hub 0 no lleva trits: sus distancias son las profundidades del árbol. Si le mandas trits, gastas \(\sim 999\) de más y te pasas de \(70\,000\).
  • El relleno del empaquetado va una sola vez al final del flujo global; si rellenas por hub, los bloques de \(N-1\) se desalinean al decodificar.
  • El índice del trit es por nodo \(v\) (encode los escribe en orden \(v=1\dots N-1\)), pero la propagación debe ir en orden BFS del árbol (padre antes que hijo), porque \(\text{padre}(v)\) puede tener índice mayor que \(v\).
  • decode_bit no debe leerse más veces de las que encode emitió: lee exactamente el mismo número y orden de bits. Usa un ancho fijo \(W=10\) consistente en ambos lados.

Conteo de bits (peor caso)

\(N=1000,\ H=36\): árbol \((N-1)\cdot 10 = 9990\) bits; trits \((H-1)(N-1)=34\,965\), empacados \(\lceil 34965/3\rceil\cdot 5 = 11\,655\cdot 5 = 58\,275\) bits. Total \(= 9990 + 58\,275 = 68\,265 < 70\,000\). ✓

Complejidad: encode hace \(H\) BFS, \(O\big(H\,(N+P)\big)\) tiempo; decode es \(O(N\cdot H)\). Memoria \(O(N+P)\). Corre en milisegundos aun con \(P\) cercano a su máximo.
