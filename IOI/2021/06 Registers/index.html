Tarea 6 — Bit Shift Registers

PDF original (oficial): statement.pdf

El problema

Christopher, el ingeniero, trabaja en un nuevo tipo de procesador. El procesador tiene acceso a \(m\) celdas de memoria de \(b\) bits cada una (con \(m = 100\) y \(b = 2000\)), llamadas registros y numeradas de \(0\) a \(m-1\). Los denotamos \(r[0], r[1], \dots, r[m-1]\). Cada registro es un arreglo de \(b\) bits, numerados del \(0\) (el bit de la derecha) al \(b-1\) (el de la izquierda). El bit \(j\) del registro \(i\) se denota \(r[i][j]\).

Para cualquier secuencia de bits \(d_0, d_1, \dots, d_{l-1}\), su valor entero es \(2^0 d_0 + 2^1 d_1 + \dots + 2^{l-1} d_{l-1}\). El valor entero almacenado en el registro \(i\) es el valor entero de la secuencia de sus bits, es decir \(2^0\, r[i][0] + 2^1\, r[i][1] + \dots + 2^{b-1}\, r[i][b-1]\).

El procesador tiene 9 tipos de instrucciones para modificar los bits de los registros. Cada instrucción opera sobre uno o más registros y guarda el resultado en uno de ellos. Usamos \(x := y\) para indicar que el valor de \(x\) se cambia hasta volverse igual a \(y\). Las operaciones son:

  • move(t, y): copia el arreglo de bits del registro \(y\) al registro \(t\). Para cada \(j\): \(r[t][j] := r[y][j]\).
  • store(t, v): pone el registro \(t\) igual a \(v\), donde \(v\) es un arreglo de \(b\) bits. Para cada \(j\): \(r[t][j] := v[j]\).
  • and(t, x, y): AND bit a bit de \(x\) y \(y\); el resultado va en \(t\). \(r[t][j] := 1\) si ambos \(r[x][j]\) y \(r[y][j]\) valen \(1\); si no, \(0\).
  • or(t, x, y): OR bit a bit. \(r[t][j] := 1\) si al menos uno de \(r[x][j]\), \(r[y][j]\) vale \(1\); si no, \(0\).
  • xor(t, x, y): XOR bit a bit. \(r[t][j] := 1\) si exactamente uno de \(r[x][j]\), \(r[y][j]\) vale \(1\); si no, \(0\).
  • not(t, x): NOT bit a bit del registro \(x\). \(r[t][j] := 1 - r[x][j]\).
  • left(t, x, p): corre todos los bits de \(x\) a la izquierda \(p\) posiciones. El resultado \(v\) cumple \(v[j] = r[x][j-p]\) si \(j \ge p\), y \(v[j] = 0\) en otro caso; luego \(r[t][j] := v[j]\).
  • right(t, x, p): corre todos los bits de \(x\) a la derecha \(p\) posiciones. \(v[j] = r[x][j+p]\) si \(j \le b-1-p\), y \(v[j] = 0\) en otro caso; luego \(r[t][j] := v[j]\).
  • add(t, x, y): suma los valores enteros de \(x\) y \(y\) y guarda el resultado en \(t\). La suma se hace módulo \(2^b\): si \(X + Y \lt 2^b\) entonces \(T = X + Y\); de lo contrario \(T = X + Y - 2^b\) (la acarreo más alta se descarta).

La entrada y la salida del programa

Christopher quiere que resuelvas dos tipos de tareas. El tipo lo indica un entero \(s\). En ambos casos debes producir un programa, es decir, una secuencia de las instrucciones de arriba.

La entrada del programa son \(n\) enteros \(a[0], a[1], \dots, a[n-1]\), cada uno de \(k\) bits (\(a[i] \lt 2^k\)). Antes de ejecutar el programa, todos los números de entrada están guardados en secuencia dentro del registro 0: para cada \(i\), el valor entero de los \(k\) bits \(r[0][i\cdot k], r[0][i\cdot k+1], \dots, r[0][(i+1)\cdot k-1]\) es igual a \(a[i]\). Se cumple \(n\cdot k \le b\). El resto de los bits del registro 0 (índices de \(n\cdot k\) a \(b-1\)) y todos los bits de los demás registros empiezan en \(0\).

La salida se calcula con el valor final del registro 0: es la secuencia de \(n\) enteros \(c[0], c[1], \dots, c[n-1]\), donde \(c[i]\) es el valor entero de los bits \(i\cdot k\) a \((i+1)\cdot k - 1\) del registro 0. Los demás bits del registro 0 (índices desde \(n\cdot k\)) y todos los demás registros pueden quedar con valores arbitrarios.

  • Tarea \(s = 0\) (mínimo): \(c[0]\) debe ser el mínimo de \(a[0], \dots, a[n-1]\). Los valores \(c[1], \dots, c[n-1]\) pueden ser cualquier cosa.
  • Tarea \(s = 1\) (ordenar): para cada \(i\), \(c[i]\) debe ser el \((1+i)\)-ésimo entero más pequeño de la entrada (es decir, \(c[0]\) es el más pequeño, \(c[1]\) el siguiente, etc.: la entrada ordenada de forma no decreciente).

Debes entregar programas con a lo más \(q\) instrucciones cada uno.

Detalles de implementación

Debes implementar el siguiente procedimiento, que se llama exactamente una vez y debe construir la secuencia de instrucciones que resuelve la tarea. No lees de la entrada estándar ni escribes a la salida estándar: solo construyes el programa.

Firma a implementar 
void construct_instructions(int s, int n, int k, int q)
  • s: tipo de tarea (\(0\) = mínimo, \(1\) = ordenar).
  • n: cantidad de enteros de la entrada.
  • k: número de bits de cada entero.
  • q: número máximo de instrucciones permitidas.

Para construir la secuencia, llamas a uno o más de estos procedimientos (cada llamada agrega la instrucción correspondiente al final del programa):

API del evaluador (cada llamada agrega una instrucción) 
void append_move(int t, int y)
void append_store(int t, bool[] v)
void append_and(int t, int x, int y)
void append_or(int t, int x, int y)
void append_xor(int t, int x, int y)
void append_not(int t, int x)
void append_left(int t, int x, int p)
void append_right(int t, int x, int p)
void append_add(int t, int x, int y)
  • En toda instrucción, \(t, x, y\) deben estar entre \(0\) y \(m-1\) (no tienen que ser distintos entre sí).
  • En append_left / append_right, \(p\) debe estar entre \(0\) y \(b\).
  • En append_store, el arreglo \(v\) debe tener longitud exactamente \(b\).
  • Existe también append_print(int t) para depurar: imprime los primeros \(n\cdot k\) bits del registro \(t\), se ignora al calificar y no cuenta como instrucción.

Posibles mensajes de error al calificar: índice de registro inválido; longitud de \(v\) distinta de \(b\); valor de desplazamiento \(p\) fuera de \([0, b]\); o «demasiadas instrucciones» si intentas agregar más de \(q\).

Ejemplos

Ejemplo 1 (\(s = 0\), \(n = 2\), \(k = 1\), \(q = 1000\))

Hay dos enteros \(a[0]\) y \(a[1]\) de \(1\) bit. Antes de ejecutar, \(r[0][0] = a[0]\) y \(r[0][1] = a[1]\). Necesitamos \(c[0] = r[0][0] = \min(a[0], a[1])\). Hay solo 4 entradas posibles, y en las 4 se cumple \(\min(a[0], a[1]) = a[0] \;\text{AND}\; a[1]\). Una solución:

Llamadas

append_move(1, 0)
append_right(1, 1, 1)
append_and(0, 0, 1)

Qué hace

r[1] := copia de r[0]
r[1] := r[1] >> 1   (r[1][0] = a[1])
r[0] := r[0] AND r[1]
       (r[0][0] = a[0] AND a[1] = min)

Ejemplo 2 (\(s = 1\), \(n = 2\), \(k = 1\), \(q = 1000\))

Mismas 4 entradas posibles. Aquí \(\min = a[0]\;\text{AND}\;a[1]\) y \(\max = a[0]\;\text{OR}\;a[1]\). Queremos \(c[0] = \min\) y \(c[1] = \max\). Una solución:

append_move(1, 0)
append_right(1, 1, 1)
append_and(2, 0, 1)
append_or(3, 0, 1)
append_left(3, 3, 1)

r[1][0] = a[1]

Resumen

Tienes \(n\) enteros de \(k\) bits empacados en el registro 0 y un set de 9 instrucciones que operan sobre registros de \(b = 2000\) bits. Debes construir una secuencia de a lo más \(q\) instrucciones que deje en el registro 0 el mínimo (\(s = 0\)) o los \(n\) números ordenados (\(s = 1\)). No buscas un algoritmo «en tiempo de ejecución»: el reto es diseñar el programa de bits y que quepa en \(q\).

Lo que importa de los límites Cada instrucción actúa sobre los \(2000\) bits a la vez: es SIMD bit a bit, así que puedes comparar/seleccionar todos los slots en paralelo. Lo que se cuenta es el número de instrucciones, no el tiempo. La subtarea 4 (\(q = 150\), \(s = 0\)) exige un mínimo en \(O(\log n)\) instrucciones; la subtarea 6 (\(q = 4000\), \(s = 1\)) deja holgura para un ordenamiento tipo \(O(n)\) pasadas.

Pistas

Pista 1 — ¿Cómo modelarlo?

¿Ya notaste que una sola instrucción (AND, OR, ADD, shift…) toca los \(2000\) bits de un registro simultáneamente? ¿Qué pasa si colocas un número en cada «slot» de \(k\) bits y aplicas la misma operación a todos?

Piensa en el registro como \(n\) celdas (slots) de \(k\) bits, una por número. Como las operaciones son por bit, puedes hacer la misma comparación o selección sobre todos los slots al mismo tiempo. Un right por \(d\cdot k\) bits alinea cada número con otro a \(d\) posiciones de distancia, y ahí puedes compararlos en paralelo. Esa es toda la maquinaria: no hay bucles «de verdad», solo instrucciones que afectan a todos los slots.

Pista 2 — Subtareas 1 y 2 (\(s=0\), \(n=2\), \(k \le 2\))

Para \(k = 1\) el ejemplo te regala \(\min(a,b) = a\;\text{AND}\;b\). Pero con \(k = 2\) eso ya no funciona, ¿cómo comparas dos números de varios bits? Y con \(q = 20\) (subtarea 2), ¿te alcanza un comparador general?

El truco AND solo sirve para \(1\) bit. Para \(k\) general necesitas un comparador que decida, en paralelo, cuál de los dos números es menor, y luego un multiplexor (AND/OR con una máscara) que elija el menor. Con \(n = 2\) basta una sola comparación: el comparador general entra en \(16\) instrucciones, cómodo dentro de \(q = 20\). Lo bueno es que ese mismo comparador escala a cualquier \(n\) y \(k\).

Pista 3 — La observación clave: comparar con una resta

¿Cómo sabes si \(a \le b\) usando solo sumas binarias? ¿Qué te dice el bit de acarreo que sale de la posición \(k\) al calcular \(a + \lnot b\)?

En complemento a dos, \(\lnot b = -b - 1\), así que dentro de un bloque de \(k\) bits \(a + \lnot b = a + (2^k - 1 - b) = 2^k + (a - b - 1)\). Entonces:

  • Si \(a \gt b\): \(a - b - 1 \ge 0\), hay acarreo hacia el bit \(k\) → ese bit vale \(1\).
  • Si \(a \le b\): \(a - b - 1 \lt 0\), no hay acarreo → el bit \(k\) vale \(0\).

Aíslas ese bit de acarreo (con una máscara que tiene un \(1\) en la posición baja de cada bloque) y lo expandes a todo el bloque sumándole una máscara de unos: el bloque queda todo-1 si \(a \le b\) y todo-0 si \(a \gt b\). Con esa máscara y su complemento, un par AND/OR selecciona el mínimo y el máximo de cada par de slots, todo en paralelo.

Pista 4 — Subtarea 4 (\(s=0\), \(q=150\)): el torneo

Comparar el slot 0 contra cada uno de los otros \(n-1\) costaría \(O(n)\) comparaciones, que no caben en \(q = 150\). ¿Y si reduces los candidatos a la mitad en cada ronda, como en un torneo?

Haz un torneo de \(\lceil \log_2 n \rceil\) rondas. En la ronda \(r\) desplazas el registro a la derecha un shift que empieza en \(k\) y se duplica cada ronda; así cada candidato superviviente se compara, en paralelo, contra el que está a medio grupo de distancia, y el comparador deja el mínimo de cada par. Tras \(\lceil \log_2 n \rceil\) rondas el mínimo global queda en el slot 0. Con \(\approx 12\) instrucciones por ronda y \(n \le 100\) (\(7\) rondas) son \(88\) instrucciones: cabe holgado en \(150\). Ojo: hay que rellenar los slots vacíos (índices \(\ge n\cdot k\)) con unos para que esos ceros «fantasma» no ganen el mínimo.

Pista final — Solución completa (100 puntos)

Tipo: construcción de un programa de instrucciones de registro que explota el SIMD bit a bit. Tres registros auxiliares fijos se preparan una sola vez con append_store:

  • reg3: un \(1\) en cada posición \(i\) con \(i \bmod 2k = 0\) → marca el bit de signo/acarreo de cada bloque.
  • reg4: máscara de slots (\(1\) donde \(\lfloor i/k\rfloor\) es par); sirve para alinear los operandos y para expandir el bit de acarreo a todo el bloque.
  • reg5: \(1\) en el rango \([\,n\cdot k,\ b\,)\) (relleno alto). Con un OR se rellenan los slots vacíos con unos para que su valor \(0\) nunca gane el mínimo; al final del ordenamiento se quita con NOT + AND.

Comparador (corazón de todo). Con reg1 = «el otro» operando ya alineado y reg0 = operando base, se calcula \(a + \lnot b\). Como \(\lnot b = 2^k - 1 - b\) por bloque, \(a + \lnot b = 2^k + (a - b - 1)\); el acarreo hacia el bit \(k\) vale \(1\) si \(a \gt b\) y \(0\) si \(a \le b\). Se lleva ese bit a la posición baja con un shift de \(k\), se aísla con reg3 y se expande sumándole reg4, dejando una máscara por slot: todo-1 cuando \(a \le b\), todo-0 cuando \(a \gt b\). La máscara y su complemento forman el multiplexor min/max.

\(s = 0\) (mínimo): torneo de \(\lceil \log_2 n\rceil\) rondas (shift empieza en \(k\) y se duplica). Cada ronda deja el mínimo de cada par; al final el mínimo global queda en el slot 0.

\(s = 1\) (ordenar): red de ordenamiento tipo burbuja con \(n\) pasadas. En cada pasada se compara cada slot con su vecino (alternando shift derecha o izquierda de \(k\) bits según la paridad de la pasada), se arman dos «carriles» (el del mínimo y el del máximo, cada uno enmascarado a sus slots), se reinsertan desplazando de vuelta \(k\) bits y se fusionan con OR. \(n\) pasadas de comparar-e-intercambiar entre vecinos equivalen a un bubblesort, que ordena cualquier permutación. Al final se limpia el relleno alto.

Correctitud. El comparador es exacto bit a bit (la derivación del acarreo de arriba). El relleno con unos en \([\,n\cdot k, b\,)\) evita que las celdas vacías contaminen el mínimo. \(s = 0\): cada ronda del torneo reduce a la mitad los posibles ganadores y los acerca al slot 0, así que tras \(\lceil \log_2 n\rceil\) rondas el mínimo global está en el slot 0. \(s = 1\): \(n\) pasadas de bubblesort dejan todo ordenado. El juez oficial es un special judge que simula el programa sobre baterías dirigidas de arrays (aleatorios, ordenados, inversos, todos iguales, \(\max\) y \(\max-1\), \(0\) y MAX, casi ordenados, etc.) y verifica el mínimo / el orden y que el número de instrucciones sea a lo más \(q\).

Pseudocódigo 
construir_instrucciones(s, n, k, q):
  # --- preparacion (comun a s=0 y s=1) ---
  store reg3 <- 1 en cada posicion i con (i mod 2k) == 0   # bit de signo por bloque
  store reg4 <- 1 en posiciones i con floor(i/k) par       # mascara de slots / expansion
  store reg5 <- 1 en el rango [n*k, b)                      # relleno alto
  reg0 <- reg0 OR reg5                                      # rellena el padding con 1s

  # subrutina conceptual: comparador en paralelo (deja mascara en reg2)
  comparar():            # reg1 = "otro" operando alineado ; reg0 = operando base
     reg8 <- NOT reg1
     reg8 <- reg8 AND reg4 ;  reg0 <- reg0 AND reg4
     reg2 <- reg0 + reg8       # = a + (~b) = a - b - 1
     reg2 <- reg2 >> k         # lleva el acarreo a la posicion baja del bloque
     reg2 <- reg2 AND reg3     # aisla el bit de signo (1 sii a > b ; 0 sii a <= b)
     reg2 <- reg2 + reg4       # expande: bloque todo-1 si a <= b , todo-0 si a > b

  si s == 0:                     # MINIMO (torneo)
     rondas <- techo(log2 n) ;  salto <- k
     repetir rondas veces:
        reg1 <- reg0 >> salto ;  salto <- salto * 2
        comparar()
        reg6 <- reg2 AND reg0              # toma el base donde a <= b
        reg7 <- NOT reg2 ;  reg7 <- reg1 AND reg7
        reg0 <- reg6 OR reg7               # min de cada par
     # el minimo global queda en el slot 0
  si no:                         # s == 1 , ORDENAR (red burbuja)
     para iter en 0 .. n-1:
        si iter par:  reg1 <- reg0 >> k    si no:  reg1 <- reg0 << k
        comparar()                          # mascara en reg2 ; reg7 = NOT reg2
        carril_min <- ((reg2 AND reg0) OR (reg1 AND reg7)) AND reg4
        carril_max <- ((reg2 AND reg1) OR (reg7 AND reg0)) AND reg4
        si iter par:  carril_max <- carril_max << k   si no:  carril_min <- carril_min >> k
        reg0 <- carril_max OR carril_min     # vecinos ordenados, reinsertados
     reg5 <- NOT reg5 ;  reg0 <- reg0 AND reg5   # quita el relleno alto

Complejidad (número de instrucciones, que es lo que puntúa):

  • \(s = 0\): \(4 + 12\lceil \log_2 n\rceil\). Para \(n = 100\): \(4 + 12\cdot 7 = 88 \le 150\). Independiente de \(k\). (\(n = 2\): \(16 \le 20\).)
  • \(s = 1\): \(4 + 19n + 2\). Para \(n = 100\): \(1906 \le 4000\). (\(n = 10\): \(196\).)
  • Usa los registros \(0\) a \(10\) de los \(100\) disponibles. No hay recursión ni búsqueda: el número de instrucciones está acotado a priori.

Puntaje pleno \(100/100\): todas las subtareas caben dentro de su \(q\). Para \(s = 1\) una red de Batcher daría \(O(n \log^2 n)\) instrucciones si \(q\) fuera más apretado, pero aquí no hace falta: la burbuja en \(\approx 19n\) entra con holgura.
