IOI 2017 · Día 1

Tarea 2 — Wiring (Cableado)

PDF original (oficial): statement.pdf

El problema

Maryam es ingeniera eléctrica. Está diseñando el cableado de una torre de comunicaciones. Sobre la torre hay varios puntos de conexión, colocados a alturas distintas. Un cable puede usarse para unir dos puntos de conexión cualesquiera, y cada punto de conexión puede tener conectados tantos cables como se quiera. Hay dos tipos de puntos de conexión: rojos y azules.

Para efectos de este problema, la torre se ve como una recta y los puntos de conexión como puntos rojos y azules ubicados en coordenadas enteras no negativas sobre esa recta. La longitud de un cable es la distancia entre los dos puntos de conexión que une.

Tu objetivo es ayudar a Maryam a encontrar un esquema de cableado tal que:

  1. Cada punto de conexión tenga al menos un cable hacia un punto de conexión de color distinto.
  2. La longitud total de los cables sea mínima.

Detalles de implementación

Debes implementar el siguiente procedimiento (es una tarea funcional: no lees de la entrada estándar ni escribes en la salida; el evaluador llama a tu función una sola vez con todos los datos y usa el valor que devuelves):

Firma a implementar 
int64 min_total_length(int[] r, int[] b)
  • r: arreglo de longitud \(n\) con las posiciones de los puntos de conexión rojos, en orden creciente.
  • b: arreglo de longitud \(m\) con las posiciones de los puntos de conexión azules, en orden creciente.
  • El procedimiento debe devolver la longitud total mínima de los cables, entre todos los esquemas de cableado válidos.
  • Nota que el tipo de retorno del procedimiento es int64 (entero de 64 bits).

Ejemplo

Llamada

min_total_length(
  [1, 2, 3, 7],
  [0, 4, 5, 9, 10]
)

Valor de retorno

10

Hay \(4\) puntos rojos, en las posiciones \(1,\, 2,\, 3\) y \(7\); y \(5\) puntos azules, en las posiciones \(0,\, 4,\, 5,\, 9\) y \(10\). Una solución óptima (la de la figura) usa estos cables:

  • azul \(0\) — rojo \(1\)\(\;\Rightarrow\;\) longitud \(1\)
  • rojo \(2\) — azul \(4\)\(\;\Rightarrow\;\) longitud \(2\)
  • rojo \(3\) — azul \(5\)\(\;\Rightarrow\;\) longitud \(2\)
  • rojo \(7\) — azul \(9\)\(\;\Rightarrow\;\) longitud \(2\)
  • rojo \(7\) — azul \(10\)\(\;\Rightarrow\;\) longitud \(3\)

La longitud total es \(1 + 2 + 2 + 2 + 3 = 10\), que es óptima, así que el procedimiento debe devolver \(10\). Nota que al punto en la posición \(7\) llegan dos cables.

Evaluador de ejemplo

El evaluador de ejemplo lee la entrada en el siguiente formato:

  • línea \(1\): \(\;n\;\; m\)
  • línea \(2\): \(\;r[0]\;\; r[1]\;\;\ldots\;\; r[n-1]\)
  • línea \(3\): \(\;b[0]\;\; b[1]\;\;\ldots\;\; b[m-1]\)

Imprime una sola línea con el valor que devuelve min_total_length.

Etiquetas Funcional Programación dinámica (DP) Greedy Two pointers

Resumen

Te dan \(n\) puntos rojos y \(m\) azules sobre una recta, en posiciones enteras distintas y ordenadas. Debes tender cables (cada uno une dos puntos) de modo que cada punto tenga al menos un cable hacia un punto de color distinto, minimizando la suma de longitudes. Implementa min_total_length(r, b) y devuelve esa longitud mínima como int64.

Lo que importa de los límites \(n,m \le 100\,000\) (hasta \(2\cdot10^5\) puntos) pide algo cercano a lineal, \(O((n+m)\log(n+m))\) o \(O(n+m)\). La subtarea 1 con \(n,m \le 200\) admite una DP \(O(nm)\) de fuerza bruta. Las posiciones llegan a \(10^9\) y las sumas a \(\sim 2\cdot10^{14}\): todo en \(\texttt{int64}\).

Pistas

Pista 1 — ¿Cómo se ven los datos juntos?

Te dan rojos y azules por separado, pero el problema vive sobre una sola recta. ¿Y si fusionas ambos arreglos en una única secuencia ordenada por posición, recordando el color de cada punto?

Al hacerlo verás que la recta se parte en bloques maximales de un mismo color que se alternan: un grupo de rojos, luego un grupo de azules, luego rojos otra vez, etc. ¿Notas que cada punto necesita un cable hacia el otro color, y que el otro color más cercano está siempre en un bloque vecino? Esa fusión (un merge de dos listas ordenadas, igual que en mergesort) es la estructura sobre la que conviene razonar.

Pista 2 — Empieza por las subtareas chicas

Con \(n,m \le 200\) (subtarea 1), ¿puedes permitirte una DP que recorra todos los pares rojo-azul?

Conviene emparejar de forma monótona: como ambos arreglos están ordenados, define \(dp[i][j]\) = costo mínimo para dejar válidos los primeros \(i\) rojos y los primeros \(j\) azules. La transición conecta el siguiente rojo, el siguiente azul, o ambos:

\[ dp[i][j] = \min\big(dp[i-1][j],\; dp[i][j-1],\; dp[i-1][j-1]\big) + \lvert r_i - b_j\rvert \]

con \(dp[0][0]=0\) y el resto \(+\infty\); la respuesta es \(dp[n][m]\). Es \(O(nm)\): suficiente para 7 puntos.

Ahora la subtarea 2 (todos los rojos antes que todos los azules): ¿cuántos cables cruzan la frontera entre el último rojo y el primer azul, y cómo emparejas el resto? Si \(n \le m\), cada rojo se ata hacia los azules y sobran azules que cuelgan del rojo más cercano. ¿Puedes escribir una fórmula cerrada para ese caso? (Pista: arrastra todos los rojos hacia la derecha y todos los azules hacia la izquierda, y paga \(\max(n,m)\) veces el hueco de la frontera.)

Pista 3 — La observación clave: cables solo entre bloques vecinos

Toma cualquier solución óptima. ¿Qué pasa si un cable salta un bloque entero de color opuesto? ¿Podrías reemplazarlo por un cable hacia el bloque adyacente (más cercano) de ese color sin aumentar la longitud, y sin dejar a nadie sin cobertura?

Sí: entre los dos extremos de ese cable hay puntos del color opuesto más cercanos; redirigir el cable a uno de ellos solo acorta y los grados de los demás puntos no bajan. Conclusión: existe un óptimo donde cada cable une dos bloques consecutivos. Entonces cada punto de un bloque elige cruzar hacia su bloque izquierdo o hacia su bloque derecho.

Y dentro de un bloque, ¿la decisión es caprichosa o monótona? Como el vecino izquierdo está a la izquierda y el derecho a la derecha, conviene que un prefijo del bloque (los de más a la izquierda) miren a la izquierda y el sufijo (los de más a la derecha) miren a la derecha. Así solo eliges un punto de corte por bloque. (Esto es justo lo que aprovechan las subtareas 3 —bloques de a lo más \(6\)— y 4 —partir cada bloque por la mitad—.)

Pista 4 — El costo de una frontera

Fíjate en una sola frontera, entre el bloque izquierdo \(L\) y el derecho \(R\) (colores opuestos). Si el sufijo de \(L\) que cruza tiene \(a\) puntos y el prefijo de \(R\) que cruza tiene \(b\) puntos, ¿cómo los emparejas para pagar lo mínimo?

El caso fundamental es \(a=b\): emparejas en escalera los \(a\) puntos del sufijo de \(L\) con los \(a\) primeros de \(R\), y el costo resulta exactamente

\[ \big(\text{suma de las } a \text{ posiciones de } R\big) \;-\; \big(\text{suma de las } a \text{ posiciones de } L\big), \]

porque cada par "derecha menos izquierda" es positivo y al sumar se telescopa a "suma de un lado menos suma del otro". Cuando \(a \ne b\), el lado que sobra se ata al punto pivote más cercano a la frontera. ¿Ves que con sumas de prefijos puedes evaluar cualquier frontera en \(O(1)\)? Eso es justo lo que hará la DP lineal de la última pista.

Pista final — Solución completa (100 puntos)

Idea (DP lineal \(O(n+m)\)).

  1. Fusión por posición. Mezcla rojos y azules en un único arreglo \(p[\,]\) ordenado por posición, guardando el color en col (\(0\)=rojo, \(1\)=azul). Quedan bloques maximales de un mismo color \(B_1, B_2, \ldots, B_k\) que alternan de color en cada frontera. Sea \(N=n+m\).
  2. Estructura del óptimo. Por la Pista 3, existe un óptimo en el que todo cable une bloques consecutivos, y cada bloque tiene un único punto de corte: un prefijo mira a la izquierda y el sufijo a la derecha.
  3. Definición de la DP (de derecha a izquierda). Sea \(dp[i]\) = costo mínimo para dejar válidos todos los puntos \(p[i..N-1]\), suponiendo que el bloque que empieza en \(i\) cruza hacia la derecha (hacia el siguiente bloque). La respuesta es \(dp[0]\): el primer bloque, al ser el de más a la izquierda, está obligado a cruzar a la derecha. Sea nxt[i] el índice donde empieza el siguiente bloque de color distinto (o \(N\) si de \(i\) al final solo hay un color, caso imposible \(\Rightarrow dp[i]=+\infty\)).
  4. Dos transiciones (siendo cross(i) el costo de cruzar el bloque que empieza en \(i\) completo contra un prefijo del siguiente, calculado con sumas de sufijos como en la Pista 4):
    • Extender: el punto \(p[i]\) cruza la frontera "igual" que su vecino \(p[i+1]\): \(\;dp[i] = dp[i+1] + (p[\,\texttt{nxt}[i]\,] - p[i])\).
    • Cruce completo de bloque: conecta el bloque actual \([i,\ \texttt{nxt}[i]-1]\) (de \(\textit{sz}=\texttt{nxt}[i]-i\) puntos) con los primeros \(\textit{sz}\) puntos del bloque siguiente; su costo \(\big(\text{suma}(R)-\text{suma}(L)\big)\) se suma a \(f[\,\cdot\,]\), que resuelve óptimamente el resto del bloque siguiente.
  5. El auxiliar \(f[\,]\). Solo cuando \(i\) es el último punto de su bloque (es decir, \(\texttt{nxt}[i]=i+1\)) se rellena \(f[\,]\) recorriendo el bloque siguiente: \(\;f[t] = \min\big(dp[t],\; (p[t]-p[i]) + f[t+1]\big)\). Cada entrada de \(f[\,]\) se calcula una sola vez, así que la actualización es \(O(1)\) amortizado.

Por qué es correcta.

  • (a) Reducción a bloques vecinos. Un cable entre bloques no adyacentes cruza al menos un bloque entero de color opuesto; redirigirlo al bloque adyacente más cercano de ese color no aumenta la longitud y solo sube los grados. Luego hay un óptimo que usa exclusivamente fronteras entre bloques consecutivos.
  • (b) Corte monótono. Como el bloque izquierdo queda a la izquierda y el derecho a la derecha, la elección óptima de cada punto es monótona: un prefijo va a la izquierda y un sufijo a la derecha, así que basta un punto de corte por bloque.
  • (c) Costo de frontera. Con \(a=b\) el emparejamiento en escalera da \(\text{suma}(R)-\text{suma}(L)\); el sobrante de tamaños distintos lo resuelve \(f[\,]\). La DP enumera exactamente las dos decisiones (extender el cruce o cerrar el bloque actual contra un prefijo del siguiente) recorriendo todas las particiones, y por inducción de derecha a izquierda \(dp[0]\) es el óptimo global. El caso imposible (un solo color de \(i\) al final) queda en \(+\infty\), pero nunca afecta a \(dp[0]\) porque hay al menos un rojo y un azul.
Pseudocódigo — DP O(n+m) 
funcion min_total_length(r[0..n-1], b[0..m-1]) -> int64:
    # 1) fusionar por posicion (merge de dos listas ordenadas)
    p, col = [], []                 # col: 0 = rojo, 1 = azul
    i, j = 0, 0
    mientras i < n o j < m:
        si j >= m o (i < n y r[i] < b[j]):
            p.push(r[i]); col.push(0); i += 1
        si no:
            p.push(b[j]); col.push(1); j += 1
    N = len(p)

    # 2) sumas de sufijos y estructuras
    suf[N] = 0
    para t = N-1 ... 0:  suf[t] = suf[t+1] + p[t]      # suf[t] = p[t]+...+p[N-1]
    rangeSum(lo, hi) = suf[lo] - suf[hi+1]             # = p[lo]+...+p[hi]

    dp[*] = INF;  f[N] = 0
    dp[N-1] = INF;  nxt[N-1] = N

    # 3) DP de derecha a izquierda
    para idx = N-2 ... 0:
        nxt[idx] = (col[idx] == col[idx+1]) ? nxt[idx+1] : idx+1   # inicio del bloque sig.

        si nxt[idx] == idx+1:                          # idx cierra su bloque -> hay frontera
            para t = nxt[idx+1]-1 ... idx+1:           # recorrer el bloque siguiente
                f[t] = min(dp[t], (p[t] - p[idx]) + f[t+1])

        si nxt[idx] == N:  continuar                   # de idx al final un solo color -> imposible

        # opcion A: extender el cruce como el vecino p[idx+1]
        dp[idx] = dp[idx+1] + (p[nxt[idx]] - p[idx])

        # opcion B: cruzar el bloque actual completo (sz puntos) vs prefijo del siguiente
        sz = nxt[idx] - idx
        si nxt[idx] + sz - 1 < nxt[nxt[idx]]:          # el bloque siguiente tiene >= sz puntos
            cross = rangeSum(nxt[idx], nxt[idx]+sz-1) - rangeSum(idx, nxt[idx]-1)
            dp[idx] = min(dp[idx], cross + f[nxt[idx]+sz])

    devolver dp[0]
Pseudocódigo — fuerza bruta (subtarea 1, O(n·m)) 
funcion brute(r[1..n], b[1..m]) -> int64:
    dp[0][0] = 0;  el resto = INF
    para i = 0 ... n:
        para j = 0 ... m:
            si i > 0 y j > 0:
                dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + |r[i] - b[j]|
    devolver dp[n][m]

Complejidad. La fusión es \(O(n+m)\) (merge de dos arreglos ya ordenados). En la DP, cada punto se procesa una vez y cada entrada de \(f[\,]\) se rellena exactamente una vez, así que el costo amortizado por punto es \(O(1)\): total \(O(n+m)\) en tiempo y \(O(n+m)\) en memoria. Con \(n+m = 2\cdot10^5\) el peor caso corre en \(\approx 0.02\) s, muy por debajo del límite de \(1.0\) s.
