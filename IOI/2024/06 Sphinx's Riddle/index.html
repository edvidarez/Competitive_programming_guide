Tarea 6 — Sphinx's Riddle

PDF original (oficial): statement.pdf

El acertijo de la Esfinge

La Gran Esfinge tiene un acertijo para ti. Te entregan un grafo con \(N\) vértices, numerados de \(0\) a \(N-1\). El grafo tiene \(M\) aristas, numeradas de \(0\) a \(M-1\). Cada arista conecta un par de vértices distintos y es bidireccional: para cada \(j\) (con \(0 \le j < M\)), la arista \(j\) une los vértices \(X[j]\) y \(Y[j]\). No hay dos aristas que conecten el mismo par de vértices. Dos vértices son adyacentes si los une una arista.

Una sucesión de vértices \(v_0, v_1, \dots, v_k\) (con \(k \ge 0\)) es un camino si cada par de vértices consecutivos \(v_l\) y \(v_{l+1}\) es adyacente. Decimos que ese camino conecta a \(v_0\) con \(v_k\). En el grafo que te dan, cada par de vértices está conectado por algún camino (es decir, el grafo es conexo).

Hay \(N+1\) colores, numerados de \(0\) a \(N\). El color \(N\) es especial y se llama color de la Esfinge. Cada vértice tiene asignado un color: el vértice \(i\) (con \(0 \le i < N\)) tiene color \(C[i]\). Varios vértices pueden compartir color, y puede haber colores que no use ningún vértice. Ningún vértice tiene el color de la Esfinge en el coloreo real, esto es, \(0 \le C[i] < N\).

Un camino \(v_0, v_1, \dots, v_k\) es monocromático si todos sus vértices tienen el mismo color, es decir, \(C[v_l] = C[v_{l+1}]\) para cada \(l\) con \(0 \le l < k\). Además, decimos que los vértices \(p\) y \(q\) están en la misma componente monocromática si y solo si los conecta un camino monocromático.

Conoces los vértices y las aristas, pero no sabes el color de cada vértice. Quieres averiguar los colores realizando experimentos de recoloreo.

En un experimento de recoloreo puedes recolorear cuantos vértices quieras. Para realizarlo, primero eliges un arreglo \(E\) de tamaño \(N\), donde para cada \(i\) (con \(0 \le i < N\)), \(E[i]\) está entre \(-1\) y \(N\) inclusive. Entonces, el color de cada vértice \(i\) pasa a ser \(S[i]\), definido así:

  • \(C[i]\), es decir, el color original de \(i\), si \(E[i] = -1\); o
  • \(E[i]\), en cualquier otro caso.

Nota que esto significa que sí puedes usar el color de la Esfinge (\(N\)) en tu recoloreo.

Finalmente, la Gran Esfinge anuncia el número de componentes monocromáticas del grafo, después de fijar el color de cada vértice \(i\) en \(S[i]\). El nuevo coloreo se aplica solo durante ese experimento: los colores de todos los vértices vuelven a los originales al terminar.

Tu tarea es identificar los colores de los vértices realizando a lo sumo 2 750 experimentos de recoloreo. También puedes recibir puntaje parcial si determinas correctamente, para cada par de vértices adyacentes, si tienen o no el mismo color.

Detalles de implementación

Esta es una tarea de comunicación (interactiva): no lees de la entrada estándar ni escribes en la salida estándar. Debes implementar el siguiente procedimiento, que el evaluador llama exactamente una vez por cada caso de prueba.

Firma a implementar 
std::vector<int> find_colours(int N, std::vector<int> X, std::vector<int> Y);
  • \(N\): número de vértices del grafo.
  • \(X\), \(Y\): arreglos de longitud \(M\) que describen las aristas.
  • Debe devolver un arreglo \(G\) de longitud \(N\) con los colores que crees que tienen los vértices.

Dentro de find_colours puedes llamar al siguiente procedimiento para realizar experimentos de recoloreo:

Procedimiento del evaluador 
int perform_experiment(std::vector<int> E);
  • \(E\): arreglo de longitud \(N\) que indica cómo recolorear los vértices (cada \(E[i]\) entre \(-1\) y \(N\)).
  • Devuelve el número de componentes monocromáticas después de recolorear según \(E\).
  • Puede llamarse a lo sumo \(2\,750\) veces.

El evaluador no es adaptativo: los colores de los vértices quedan fijos antes de la llamada a find_colours.

En cada subtarea obtienes el puntaje completo si en todos sus casos el arreglo \(G\) que devuelve find_colours es exactamente igual a \(C\) (es decir, \(G[i] = C[i]\) para todo \(i\)). En caso contrario, obtienes el 50 % del puntaje de la subtarea si en todos sus casos se cumple:

  • \(0 \le G[i] < N\) para todo \(i\); y
  • para cada arista \(j\): \(G[X[j]] = G[Y[j]]\) si y solo si \(C[X[j]] = C[Y[j]]\) (es decir, aciertas, por cada arista, si sus extremos son del mismo color o no).

Ejemplo

Considera la siguiente llamada:

find_colours(4, [0, 1, 0, 0], [1, 2, 2, 3])

Supón que los colores (ocultos) de los vértices son \(C = [2, 0, 0, 0]\). El escenario se muestra en la figura: el número grande dentro de cada círculo es el índice del vértice y la etiqueta blanca pequeña es su color.

El procedimiento puede llamar:

perform_experiment([-1, -1, -1, -1])

Aquí no se recolorea nada (todos conservan su color original). Los vértices \(1\) y \(2\) tienen color \(0\) y el camino \(1, 2\) es monocromático, así que están en la misma componente. Los vértices \(1\) y \(3\), aunque ambos tienen color \(0\), están en componentes distintas porque no hay un camino monocromático que los una. En total hay \(3\) componentes monocromáticas: \(\{0\}\), \(\{1, 2\}\) y \(\{3\}\). Esta llamada devuelve \(3\).

Luego puede llamar:

perform_experiment([0, -1, -1, -1])

Esta llamada devuelve \(1\), pues todos los vértices quedan en la misma componente monocromática. Podemos deducir que los vértices \(1\), \(2\) y \(3\) tienen color \(0\).

Después puede llamar:

perform_experiment([-1, -1, -1, 2])

Esta llamada devuelve \(2\), con componentes \(\{0, 3\}\) y \(\{1, 2\}\). Podemos deducir que el vértice \(0\) tiene color \(2\).

El procedimiento find_colours devuelve entonces el arreglo \([2, 0, 0, 0]\). Como \(C = [2, 0, 0, 0]\), se otorga el puntaje completo. (También se daría el \(50\%\) por valores como \([1, 2, 2, 2]\) o \([1, 2, 2, 3]\), que respetan la relación mismo-color/distinto-color por arista.)

Evaluador de ejemplo

El evaluador de ejemplo lee la entrada en este formato:

N M
C[0] C[1] ... C[N-1]
X[0] Y[0]
X[1] Y[1]
...
X[M-1] Y[M-1]

Y produce la salida:

Donde \(L\) es la longitud del arreglo \(G\) que devolvió find_colours y \(Q\) es la cantidad de llamadas a perform_experiment.

Resumen

Te dan un grafo conexo de \(N\) vértices con colores ocultos. Implementas find_colours y puedes llamar perform_experiment(E) hasta \(2\,750\) veces: \(E\) recolorea cada vértice (\(E[i]\) = color, o \(-1\) = deja su color real, o \(N\) = color de la Esfinge) y te responde cuántas componentes monocromáticas hay con ese coloreo. Devuelve el color de cada vértice (100 %) o, al menos, acierta por cada arista si sus extremos son del mismo color (50 %).

Lo que importa de los límites \(N \le 250\) y un tope duro de \(2\,750\) experimentos: la solución de pleno usa \(\approx 3N + N\log_2 N \approx 2742\) llamadas para \(N=250\), apenas debajo del tope. La métrica que se optimiza es el número de experimentos, no el tiempo.
El 50 % parcial Acertar, por arista, mismo-color/distinto-color equivale a identificar las componentes monocromáticas. Eso ya vale la mitad de cada subtarea.

Pistas

Pista 1 — ¿Qué te dice cada experimento, y para qué sirve el color de la Esfinge?

¿Ya pensaste que lo único que recibes de vuelta es un número (la cantidad de componentes monocromáticas)? ¿Cómo conviertes eso en información concreta sobre los colores?

Un experimento te deja pintar cualquier subconjunto de vértices como quieras y te devuelve un solo entero. La clave es diseñar el coloreo para que ese entero responda una pregunta de sí/no.

¿Y si usas el color de la Esfinge (\(N\))? Como ningún vértice lo tiene de verdad y nunca lo usarás dos veces a la vez salvo a propósito, pintar un vértice de color \(N\) lo vuelve un color único que no se funde con nada. ¿No es eso un \"aislante\" perfecto?

Pista 2 — El 50 % parcial y la Subtarea 1 (\(N = 2\))

¿Notaste que el crédito parcial solo te pide acertar, por arista, si los extremos comparten color? ¿Eso a qué objeto del grafo equivale?

Equivale a conocer las componentes monocromáticas: dos vértices adyacentes son \"mismo color\" exactamente cuando están en la misma componente. Si descubres las componentes, ya tienes el 50 %.

Para \(N = 2\) (Subtarea 1): con un solo experimento sin recolorear, ¿qué significa que devuelva \(1\) frente a \(2\)? Si devuelve \(1\), ambos vértices comparten color; si \(2\), son distintos. ¿Cómo hallarías además el color exacto probando colores \(0, 1, \dots\) en un vértice?

Pista 3 — Subtarea 2 (\(N \le 50\)): construir las componentes vértice por vértice

Con \(N \le 50\) tienes muchísimos experimentos de sobra (\(2\,750\) para \(50\) vértices). ¿Y si construyes las componentes inductivamente, agregando un vértice a la vez?

Procesa los vértices \(1, 2, \dots, N-1\) uno por uno. Al agregar el vértice nuevo \(u\), quieres saber a cuáles de las componentes ya descubiertas se pega (por aristas monocromáticas).

¿Cómo aíslas el ruido? Pinta de color Esfinge (\(N\)) todo lo que todavía no procesas, deja a \(u\) con su color real y deja a cada componente ya hallada con su color real. ¿No te dice el cambio en el conteo de componentes a cuántas se unió \(u\)?

Pista 4 — Fase 1: localizar la componente exacta con búsqueda binaria (ya da el 50 %)

Ya sabes cuántas componentes toca \(u\). ¿Pero cuáles son? ¿Probar una por una no gasta demasiados experimentos?

En vez de probar componente por componente, usa búsqueda binaria. Ordena las componentes ya descubiertas y trabaja sobre un rango \([lo, hi)\): pinta de color Esfinge (\(N\)) todas las componentes que están fuera del rango para que no estorben. Según si \(u\) se fusiona con alguien dentro de la mitad probada, te quedas con esa mitad. Así localizas cada componente con \(\log_2 N\) experimentos.

Repitiendo mientras queden fusiones pendientes, ¿no terminas con un \"bosque generador monocromático\" cuyas componentes son exactamente las componentes monocromáticas reales? Con eso ya tienes el \(50\%\) de todas las subtareas.

Pista 5 — Subtareas 3 (camino) y 4 (completo): ¿qué cambia con la estructura?

En un camino (Subtarea 3) cada componente monocromática es un tramo contiguo de vértices, y al colapsar quedan componentes vecinas alternadas. ¿No simplifica eso el conteo y la búsqueda?

En un grafo completo (Subtarea 4) cualquier par de vértices es adyacente, así que dos vértices del mismo color están siempre en la misma componente: cada componente es justo el conjunto de vértices de un color. ¿Cómo aprovecha eso el conteo?

Estas dos subtareas son casos particulares del método general: hallar componentes y luego sus colores. La idea importante para el caso general aparece al colapsar cada componente en un super-vértice.

Pista 6 — Fase 2: descubrir los colores reales bipartiendo el grafo colapsado

Ya tienes las componentes (50 %). Para el 100 % necesitas el color real de cada una. ¿Cómo lo averiguas sin gastar de más?

Colapsa cada componente en un super-vértice y arma un árbol generador del grafo colapsado. Bipártelo por paridad de profundidad: el conjunto \(A\) (capas pares) y el \(B\) (capas impares). Las aristas del árbol siempre cruzan de \(A\) a \(B\), así que cada uno es independiente.

Para un color candidato \(f\): pinta TODO de \(f\) dejando solo el conjunto que pruebas con su color real. Una componente cuyo color real sea \(f\) se funde con el \"mar\" de color \(f\) y baja el conteo de componentes. ¿No te revela eso cuántas componentes del conjunto tienen color \(f\)? Con búsqueda binaria localizas cuáles, y lo haces para \(A\) y para \(B\).

Pista final — Solución completa (100 puntos)

La solución tiene dos fases. La Fase 1 descubre las componentes monocromáticas (eso solo ya vale el 50 %). La Fase 2 descubre el color real de cada componente para llegar al 100 %.

Fase 1 — Hallar las componentes monocromáticas

Se construye, de forma inductiva, un bosque generador monocromático. Procesas los vértices \(1, 2, \dots, N-1\). Para el vértice nuevo \(u\) usas un coloreo \"aislante\": todo lo aún no procesado se pinta con el color de la Esfinge \(N\) (queda inerte, cada vértice es su propia componente), \(u\) mantiene su color real (\(E[u] = -1\)) y las componentes ya descubiertas mantienen su color real. Como conoces el grafo, calculas localmente cuántas componentes esperarías con ese coloreo; la diferencia con lo que devuelve el experimento te dice a cuántas componentes existentes se une \(u\).

Para saber cuáles, haces búsqueda binaria sobre las componentes ya descubiertas, acotando el rango: las componentes fuera del rango de búsqueda se pintan con \(N\) para que no estorben. Así localizas exactamente cada componente con la que \(u\) se fusiona. Repites mientras queden fusiones pendientes y luego fusionas todo junto con \(u\). Al terminar, las componentes del bosque coinciden con las componentes monocromáticas reales.

Fase 2 — Descubrir los colores reales \(0, \dots, N-1\)

Colapsa cada componente en un super-vértice. Sobre el grafo colapsado arma un árbol generador y bipártelo por paridad de profundidad en dos conjuntos independientes \(A\) (capas pares) y \(B\) (capas impares); toda arista del árbol cruza de \(A\) a \(B\). Para cada color candidato \(f = 0, 1, \dots, N-1\): pinta TODO con \(f\) dejando el conjunto que pruebas con su color real. Una componente cuyo color real sea \(f\) se funde con el \"mar\" de color \(f\) y baja el conteo, revelando cuántas hay; con búsqueda binaria (pintando el prefijo/sufijo fuera del rango con \(N\) o con \(f\)) localizas cuáles componentes tienen color \(f\). Lo haces para \(A\) y para \(B\), y así determinas el color de toda componente.

Caso especial: si todo el grafo resulta una sola componente, basta probar colores \(f = 0, 1, \dots, N-1\) (dejando un vértice con su color real) hasta que perform_experiment devuelva \(1\); ese \(f\) es el color de todos.

¿Por qué es correcta?

En la Fase 1, aislar con el color \(N\) todo lo irrelevante hace que el cambio en el conteo identifique unívocamente a qué componente(s) ya descubierta(s) se pega el vértice nuevo, y la búsqueda binaria (pintando fuera-del-rango con \(N\)) localiza la componente exacta. Al terminar, las componentes coinciden con las monocromáticas reales, lo que garantiza la relación mismo-color/distinto-color por arista (el 50 %). En la Fase 2, al pintar un conjunto independiente del grafo colapsado con un color de prueba \(f\) y dejar el resto real, cada componente del otro lado cuyo color sea \(f\) se funde con el mar \(f\) y reduce el conteo de forma detectable; haciéndolo para \(A\) y \(B\) se determina el color de todas. Como el evaluador solo da pleno si \(G = C\) exactamente, la verificación contra el juez es la prueba definitiva.

Pseudocódigo 
FIND_COLOURS(N, X, Y):
    construir lista de adyacencia del grafo
    comps <- HALLAR_COMPONENTES(N, ady)          # Fase 1
    si |comps| == 1:                             # todo es una sola componente
        para f de 0 a N-1:
            E <- [f, f, ..., f];  E[0] <- -1      # deja el vertice 0 con su color real
            si perform_experiment(E) == 1:
                devolver [f, f, ..., f]
    asignar cid[v] = id de componente de cada vertice v
    construir grafo colapsado (una arista entre componentes vecinas distintas)
    bipartir el grafo colapsado por BFS:  A = capas pares,  B = capas impares
    FC <- color por componente, todo en -1
    RESOLVER(A, FC);  RESOLVER(B, FC)            # Fase 2
    devolver  F[v] = FC[cid[v]]  para todo v

HALLAR_COMPONENTES(N, ady):                      # bosque generador monocromatico
    comps <- [ {0} ]
    para u de 1 a N-1:
        # aislar: vertices aun no procesados -> color N (Esfinge);
        #         u -> su color real;  cada componente ya hallada -> su color real
        esperado <- componentes_esperadas(ese coloreo)        # calculado en local
        toca     <- esperado - perform_experiment(ese coloreo)  # cuantas comps toca u
        si toca == 0:
            comps <- comps + {u};  continuar
        mientras toca > 0:
            # busqueda binaria sobre el rango [lo, hi) de componentes;
            # las de FUERA del rango se pintan con N para no estorbar
            localizar por biseccion la componente que se funde con u
            marcarla para fusionar;  toca <- toca - 1
        fusionar todas las componentes halladas junto con u
    devolver comps

RESOLVER(S, FC):           # S = una clase (A o B) del grafo colapsado bipartido
    para f de 0 a N-1:
        repetir:
            # pintar TODO con f, dejando el rango [lo, hi) de S con su color real
            baja <- componentes_esperadas - perform_experiment(ese coloreo)
            si baja == 0:  salir         # ya no quedan componentes de color f en S
            por busqueda binaria sobre [lo, hi) localizar una componente de color f
            FC[esa componente] <- f
            reiniciar el rango;  si solo quedaba 1, salir

Complejidad (número de experimentos, que es la métrica que importa):

  • Fase 1: \(1\) experimento por vértice nuevo más \(\log_2 N\) por cada arista del bosque generador (a lo sumo \(N-1\) aristas), es decir \(N + (N-1)\log_2 N\).
  • Fase 2: \(2\) experimentos por color (uno para \(A\) y otro para \(B\)) más \(\log_2 N\) por cada color realmente hallado.
  • Total \(\le 3N + N\log_2 N\), que para \(N = 250\) es \(\approx 750 + 1992 = 2742\), justo debajo del tope de \(2\,750\). En los datos oficiales el peor caso real gastó \(2741\) experimentos.

Cómputo y memoria: cada experimento se simula/cuenta con un BFS en \(O(N+M)\); memoria \(O(N+M)\), más una matriz \(N \times N\) (\(O(N^2)\)) para bipartir el grafo colapsado. Todo holgado para \(N \le 250\).

Nota de implementación (C++) 
El tope de 2 750 es muy ajustado: el peor caso real (2741) deja apenas
9 de margen. NO conviene agregar experimentos \"extra\" por componente
(p. ej. una busqueda lineal en vez de binaria, o una verificacion de mas):
una sola query adicional por componente rompe los casos grandes de
\"camino\" y \"completo\". El conteo de componentes (en el solver y en el
manager) es un BFS iterativo, sin recursion profunda.

No es un truco de lenguaje: es una advertencia de presupuesto. El algoritmo del editorial ya entra; no intentes minimizar la constante con búsquedas exponenciales de asignaciones.
