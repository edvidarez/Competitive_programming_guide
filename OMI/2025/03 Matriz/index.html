Problema 03 — Matriz

Enunciado oficial: statement.pdf · Sitio oficial: olimpiadadeinformatica.org.mx

El problema

Sebas y Saúl son criaturas bidimensionales simples que estaban pensando en el siguiente problema:

Sebas le quiere transmitir un mensaje a Saúl, que es un arreglo \(a\) de tamaño \(n\) (con \(1 \le n \le 100\) y \(1 \le a_i \le 10^4\)), a través de una matriz binaria (cuyos valores son \(0\) o \(1\)) de \(100 \times 100\). Ambos se pusieron de acuerdo para resolver el problema, tal que sin importar qué matriz mande Sebas, Saúl pueda reconstruir correctamente el arreglo original de Sebas sin alterar su orden.

Implementaron una solución en Karel, intentando minimizar la cantidad de posiciones de la matriz que manda Sebas que eran iguales a \(1\). Tu reto es resolver este problema de una forma más eficiente.

Detalles de implementación

Deberás implementar dos funciones en el mismo archivo.

La función sebas() deberá implementar tu estrategia para Sebas.

vector<vector<int>> sebas(int n, vector<int> a);

Esta función recibe dos parámetros:

  • Un entero \(n\), el tamaño del arreglo (\(1 \le n \le 100\)).
  • El arreglo a codificar, un vector de enteros \(a\) (con \(1 \le a_i \le 10^4\)).

Y regresa un vector de vectores de enteros, la matriz binaria que manda Sebas. En particular, tu código debe regresar un vector de \(100\) vectores de tamaño \(100\), cuyos elementos sean todos \(1\) o \(0\). Si esto no se cumple, recibirás un veredicto de respuesta incorrecta.

La función saul() deberá implementar tu estrategia para Saúl.

vector<int> saul(vector<vector<int>> matriz);

Esta función recibe un parámetro:

  • Un vector de vectores de enteros, matriz (se garantiza que tiene dimensiones \(100 \times 100\) y todos los elementos son \(1\) o \(0\)).

Y regresa un vector de enteros, que debe corresponder al vector que codificó Sebas.

El evaluador correrá tu programa dos veces por caso. En la primera, llamará la función sebas() múltiples veces, una para cada subcaso de prueba (recuerda reinicializar las variables globales). En la segunda, llamará la función saul() múltiples veces, una por cada subcaso de prueba (recuerda reinicializar las variables globales).

Entrada

No hay lectura de stdin: el evaluador invoca tus funciones. La función sebas() recibe el entero \(n\) y el arreglo \(a\) a codificar. La función saul() recibe la matriz \(100 \times 100\) que produjo sebas().

Salida

No se escribe a stdout: tus funciones regresan el resultado. sebas() regresa la matriz binaria \(100 \times 100\); saul() regresa el arreglo reconstruido, que debe ser exactamente el arreglo que recibió Sebas (mismo contenido y mismo orden).

Consideraciones

  • Tu programa será ejecutado por el evaluador dos veces por caso, por lo que todas las variables globales que guardes en la primera ejecución se perderán en la segunda.
  • La función sebas() debe responder un vector de vectores de enteros, de tamaño \(100 \times 100\). Cada valor debe ser \(0\) o \(1\). De lo contrario recibirás un veredicto de respuesta incorrecta.
  • La función saul() debe responder con un vector de enteros que produce la matriz al llamarse sebas() con el vector como argumento. Cada valor \(i\) del vector debe cumplir \(1 \le i \le 10^4\). De lo contrario recibirás un veredicto de respuesta incorrecta.
  • Para todos los casos, se cumple que \(1 \le n \le 100\).

Subtareas

El puntaje depende de qué tan pocas posiciones iguales a \(1\) use tu matriz. Para cada subtarea, sea \(X\) la máxima cantidad de posiciones de la matriz iguales a \(1\) entre todos los subcasos, y sea \(N\) el tamaño del arreglo (\(N=n\)). Todas las subtareas, a excepción de la primera, otorgan puntos parciales según el rango en que caiga \(X\).

Resumen

Hay que diseñar dos funciones acopladas: sebas(n, a) codifica un arreglo de \(n \le 100\) enteros (cada uno en \([1, 10^4]\)) dentro de una matriz binaria \(100 \times 100\), y saul(M) reconstruye el arreglo exacto a partir de la matriz. El objetivo es minimizar la cantidad de unos (el máximo \(X\) entre todos los subcasos). La meta de \(100\) puntos es lograr \(X \le 2N\) para cualquier arreglo.

Lo que importa de los límites La matriz tiene \(100 \times 100 = 10^4\) celdas; cada número cabe en \([1,10^4]\). Como \(10^4 = 100^2\), un número "vive" naturalmente en un par de coordenadas. La frontera difícil es \(X \le 2N\) (subtareas 4 y 5): hay que codificar con un promedio de 2 unos por número. Saúl no conoce \(n\): la matriz también debe indicar dónde terminan los datos.

Pistas

Pista 1 — ¿Cómo modelarlo?

¿Ya pensaste que esto es un problema de codificación: Sebas y Saúl acuerdan de antemano un "diccionario" y la matriz es el mensaje? ¿Cuánta información hay realmente que transmitir, y cuántos unos puedes pagar por ella?

No importa cómo se vea la matriz: lo único que cuenta es que la regla de Sebas (arreglo \(\to\) matriz) sea invertible por Saúl. "Minimizar unos" sugiere medir el costo en unos, no en celdas: te sobran celdas (\(10^4\)), te faltan unos baratos.

Pista 2 — Empieza por lo fácil (subtarea 1)

La subtarea 1 te regala \(a_i \le 100\) y sin límite de unos. ¿No basta entonces con la codificación más tonta posible?

Por ejemplo: usa la fila \(i\) para el número \(a_i\) y pon un \(1\) en la columna \(a_i\) (o escríbelo en binario). Saúl lee fila por fila. Como sobran celdas y no se penalizan los unos, esto se lleva los \(5\) puntos sin esfuerzo. La idea de una fila por número será la base de todo.

Pista 3 — Pocos unos con combinaciones (subtareas 2 y 3)

Si en una fila de \(100\) celdas eliges colocar exactamente \(k\) unos, ¿cuántos valores distintos puedes representar? ¿Y cuál es el menor \(k\) que alcanza para todos los valores hasta \(10^4\)?

Con \(k\) unos en \(100\) celdas representas \(\binom{100}{k}\) valores distintos. Como \(\binom{100}{3} = 161700 \ge 10^4\), bastan 3 unos por número: numera todas las combinaciones de \(3\) columnas y usa la combinación cuyo índice es \(a_i\). Eso da \(X \le 3N\) (full en subtareas 2 y 3). ¿Y cómo sabe Saúl dónde parar? Como \(a_i \ge 1\), ningún número real produce una fila vacía: la primera fila completamente en ceros marca el fin.

Pista 4 — ¿Por qué 2 unos no caben en una fila?

\(\binom{100}{2} = 4950 < 10^4\). Con \(2\) unos en una sola fila no alcanza. ¿Entonces tendrás que darle a cada número más espacio que una fila... o compartir celdas entre varios números?

La clave es dejar de pensar "una fila por número". Si juntas varios números y los codificas en conjunto sobre varias filas, el número de combinaciones crece muchísimo más rápido que la información que necesitas, y el costo promedio por número puede bajar a \(2\) unos.

Pista 5 — La observación clave: bloques de 4 (subtareas 4 y 5)

Junta los números de 4 en 4. Como cada \(a_i \le 10^4 < 10001\), cuatro números forman un solo entero en base \(B = 10001\): \(\text{num} = a_0 + a_1 B + a_2 B^2 + a_3 B^3 < B^4 \approx 10^{16}\). ¿Cuántos unos necesitas para indexar hasta \(10^{16}\) cosas si los repartes en \(4\) filas (\(400\) celdas)?

El número de formas de poner \(8\) unos en \(400\) celdas es \(\binom{400}{8} \approx 1.5 \times 10^{16} > B^4\). Es decir, 8 unos en 4 filas alcanzan para codificar cualquier bloque de \(4\) números: ¡eso es exactamente \(2\) unos por número! El puente entre "un entero" y "qué \(8\) celdas prender" es el sistema numérico combinatorio (combinadic), que da una biyección entre los enteros \(0 \ldots \binom{400}{8}-1\) y los subconjuntos de \(8\) posiciones.

Pista 6 — Que el peor caso sea exactamente 2N

Si \(n\) no es múltiplo de \(4\) rellenas con ceros, pero entonces el último bloque tiene menos información. ¿Puedes gastar menos unos en bloques chicos para que el total nunca pase de \(2N\)? Y de nuevo: ¿cómo sabe Saúl dónde terminan los datos?

Usa el menor \(k\) (par) tal que \(\text{num} < \binom{400}{k}\). Un bloque con \(r\) números reales cumple \(\text{num} < B^{\,r} \le \binom{400}{2r}\), así que gasta a lo más \(2r\) unos. Sumando: bloques llenos \(8\) unos, el último \(2r\), total \(\le 2N\). Para el fin de datos: como \(a_i \ge 1\), un bloque real nunca queda en todo ceros; el primer bloque de \(4\) filas vacías indica que ya no hay más números. Nota que esto funciona sin suponer que el arreglo esté ordenado, así que resuelve las subtareas 4 y 5 a la vez.

Pista final — Solución completa (100 puntos)

Idea. Trabaja en bloques de \(4\) números. Cada bloque se empaqueta en un entero en base \(B = 10001\); ese entero se traduce a un conjunto de a lo más \(8\) posiciones encendidas dentro de \(4\) filas de la matriz, usando el sistema numérico combinatorio. Saúl invierte el proceso: cuenta los unos de cada \(4\) filas, recupera el entero y de ahí los \(4\) números.

Por qué es correcto. El sistema combinatorio es una biyección entre \(\{0,1,\dots,\binom{m}{k}-1\}\) y los subconjuntos de tamaño \(k\) de \(\{0,\dots,m-1\}\): todo entero \(r\) se escribe de forma única como \(r = \binom{c_k}{k} + \binom{c_{k-1}}{k-1} + \dots + \binom{c_1}{1}\) con \(c_k > c_{k-1} > \dots \ge 0\). Como \(B^4 - 1 < \binom{400}{8}\), todo bloque cabe; y como \(a_i \ge 1\), un bloque con datos nunca produce \(4\) filas vacías, así que Saúl detecta el final sin conocer \(n\). El orden se preserva porque los bloques y los dígitos base \(B\) se procesan en orden.

Por qué \(X \le 2N\). Un bloque con \(r\) números reales (\(r \in \{1,2,3,4\}\)) tiene \(\text{num} < B^{\,r}\), y se verifica que \(B^{\,r} \le \binom{400}{2r}\); por eso elegimos \(k \le 2r\) unos. Bloques completos cuestan \(8 = 2\cdot 4\) unos y el último bloque (relleno) cuesta \(2r\), de modo que el total es \(\le 2N\) para cualquier \(n\).

Pseudocódigo 
const B = 10001            // base; a_i en [1, 10^4] < B

binom(n, k):               // coeficiente binomial exacto (n < 400, k <= 8)
    si n < k: regresa 0
    res = 1
    para i = 0 .. k-1:
        res = res * (n - i) / (i + 1)
    regresa res

elegir_k(num):             // menor k par en {2,4,6,8} con num < C(400,k)
    para k en (2, 4, 6, 8):
        si num < binom(400, k): regresa k

// rango  ->  k posiciones distintas en 0..399  (sistema combinatorio)
codificar(num, k):
    pos = lista vacia
    para i = k bajando hasta 1:
        c = i - 1
        mientras binom(c+1, i) <= num:  c = c + 1
        agrega c a pos
        num = num - binom(c, i)
    regresa pos            // estrictamente decreciente

// k posiciones  ->  rango
decodificar(pos):
    ordena pos de mayor a menor
    k = tamano(pos);  num = 0
    para idx = 0 .. k-1:
        num = num + binom(pos[idx], k - idx)
    regresa num

funcion sebas(n, a):
    M = matriz 100x100 llena de ceros
    mientras tamano(a) no sea multiplo de 4:  agrega 0 a a   // relleno
    para cada bloque b = 0 .. tamano(a)/4 - 1:
        num = a[4b] + a[4b+1]*B + a[4b+2]*B^2 + a[4b+3]*B^3
        si num == 0:  continua                 // no pasa con numeros reales
        k = elegir_k(num)
        para cada x en codificar(num, k):
            fila = 4b + (x / 100);  col = x mod 100
            M[fila][col] = 1
    regresa M

funcion saul(M):
    res = lista vacia
    para b = 0 .. 24:                           // 25 bloques = 100 filas
        pos = lista vacia
        para r = 0 .. 3:  para col = 0 .. 99:
            si M[4b + r][col] == 1:  agrega (r*100 + col) a pos
        si pos esta vacia:  termina             // fin de los datos
        num = decodificar(pos)
        para j = 0 .. 3:
            d = num mod B;  num = num / B
            si d != 0:  agrega d a res           // ignora ceros de relleno
    regresa res

Complejidad: \(O(N)\) bloques; cada bloque hace \(O(k \cdot 400)\) trabajo para codificar/decodificar con \(k \le 8\). En total \(O(N)\) con constante chica, y memoria \(O(100 \times 100)\).
