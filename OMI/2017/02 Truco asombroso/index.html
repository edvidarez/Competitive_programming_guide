Problema 02 — Truco asombroso

Enunciado oficial: statement.md · Practica en línea: omegaUp

Historia

Ethan el matemágico ha empezado a armar su nuevo truco de magia y matemáticas. Este nuevo truco hará que Ethan se convierta en el más famoso de entre todos los matemágicos del mundo. Tú, como mejor amigo de Ethan, has decidido ayudarle para que su acto no tenga fallas y logre su fama.

El truco de Ethan consiste en agarrar un número y modificarlo varias veces con alguno de los siguientes actos:

  • Sumarle \(1\) al número que tiene.
  • Restarle \(1\) al número que tiene.
  • Dividir entre \(2\), siempre y cuando el número que tiene sea par.

El truco termina cuando el número que tiene sea \(1\).

El problema

Tu tarea es ayudar a Ethan a ejecutar su truco en el mínimo número de actos para que el truco sea asombroso.

Para hacer el truco más asombroso, el número inicial está expresado en notación binaria.

Entrada

En la primera línea, el entero \(N\) que representa la cantidad de bits (cifras) del número binario inicial.

En la segunda línea, una cadena de longitud \(N\) que contiene \(0\)s y \(1\)s (sin separación). Esta cadena representa el número inicial. El primer carácter es el bit más significativo.

Salida

Un entero que indique la mínima cantidad de actos que tiene que hacer Ethan el matemágico para que el número original asombrosamente se convierta en \(1\).

Límites

Límite de tiempo1 s (no especificado en el enunciado; valor estándar de omegaUp)
Límite de memoria256 MB (no especificado en el enunciado; valor estándar de omegaUp)

Ejemplo

Ejemplo 1 — el número inicial (en notación decimal) es \(3\), que en binario es 11 (\(N = 2\)).

Entrada

2
11

Salida

2

\(3 \to 2 \to 1\). La primera operación es restarle \(1\), con lo que queda como \(2\). La segunda operación es dividirlo entre \(2\), con lo que se obtiene un \(1\) y el acto ha terminado después de \(2\) pasos.

Ejemplo 2 — el número inicial (en notación decimal) es \(10\), que en binario es 1010 (\(N = 4\)).

Entrada

4
1010

Salida

4

\(10 \to 5 \to 4 \to 2 \to 1\). Se divide entre \(2\) para obtener \(5\), se le resta \(1\) para obtener \(4\), se divide entre \(2\) para obtener \(2\), se divide entre \(2\) para obtener \(1\). El acto ha terminado después de \(4\) pasos.

Etiquetas Greedy Representación binaria Programación dinámica (DP) Matemáticas

Resumen

Te dan un número en binario (hasta \(10^6\) bits). Quieres llevarlo a \(1\) con el mínimo de operaciones, donde cada operación es \(+1\), \(-1\) o \(\div 2\) (esta última sólo si el número es par). Devuelve ese mínimo.

Lo que importa de los límites Con \(N \le 10^6\) bits el número es gigantesco: no cabe en long long ni conviene aritmética de multiprecisión. Hay que trabajar directo sobre la cadena de bits en \(O(N)\). La subtarea con \(N \le 20\) (el número cabe en \(64\) bits) admite fuerza bruta y vale \(27\%\).

Pistas

Pista 1 — ¿Qué le hace cada operación a los bits?

¿Ya pensaste qué significa cada acto sobre la representación binaria del número?

Dividir entre \(2\) (cuando es par, o sea, cuando el bit menos significativo es \(0\)) es simplemente quitar el último bit. Sumar o restar \(1\) cambia la paridad (el último bit) y puede generar acarreos. Para llegar a \(1\) hay que ir "comiendo" todos los bits hasta que sólo quede el \(1\) más alto. ¿Cómo te conviene tratar cada bit empezando por el menos significativo?

Pista 2 — Empieza por lo chico (subtarea de 27%)

¿Qué te regala la restricción \(N \le 20\)?

Con \(N \le 20\) el número cabe en un entero de \(64\) bits (a lo más \(2^{20}-1 \approx 10^6\)). ¿Una BFS desde el valor inicial, con vecinos \(v+1\), \(v-1\) y \(v/2\) (si \(v\) es par), te encuentra la distancia mínima a \(1\)? ¿Por qué BFS y no una recursión ingenua? (Cuidado: \(+1\) puede subir el valor, así que una recursión sin control puede ciclar; BFS por capas da el mínimo número de pasos.) Eso ya te asegura el \(27\%\).

Pista 3 — ¿Cuándo conviene dividir?

Si el número es par, ¿hay alguna razón para hacerle \(+1\) o \(-1\) en lugar de dividir?

No: dividir un número par nunca empeora. Cualquier camino que primero sume o reste a un número par puede reescribirse para dividir antes y salir igual o mejor. Así que las decisiones interesantes están sólo en los números impares: ahí, antes de poder dividir, debes gastar exactamente una operación para volverlo par. La pregunta es: ¿\(+1\) o \(-1\)?

Pista 4 — La decisión en los impares: mira los dos últimos bits

Para un número impar, ¿qué te dicen sus dos bits menos significativos sobre si conviene sumar o restar?

Si termina en …01 (es decir, \(N \equiv 1 \pmod 4\)), restar \(1\) borra ese \(1\) aislado y deja un \(0\) que se divide de inmediato. Si termina en …11 (\(N \equiv 3 \pmod 4\)), sumar \(1\) provoca un acarreo que convierte toda la racha de unos en un único \(1\) más arriba; pagas \(1\) operación ahora pero te ahorras restar \(1\) varias veces. ¿Notas que sumar conviene justo cuando hay una racha de \(\ge 2\) unos al final?

Hay una excepción: cuando lo que queda es exactamente \(3\) (11) conviene restar (\(3 \to 2 \to 1\), \(2\) pasos), no sumar (\(3 \to 4 \to 2 \to 1\), \(3\) pasos).

Pista final — Solución completa (100 puntos)

Procesa el número bit por bit, del menos significativo (LSB) al más significativo (MSB), llevando una cuenta de operaciones:

  • Si el bit actual es \(0\) (número par): divide (cuenta \(1\)) y avanza al siguiente bit.
  • Si el bit actual es \(1\) (número impar) y el siguiente bit es \(0\) (patrón …01): resta \(1\) (apaga ese bit, cuenta \(1\)); en el siguiente paso ese \(0\) se dividirá.
  • Si el bit actual es \(1\) y el siguiente también es \(1\) (racha de unos, …11): suma \(1\) (cuenta \(1\)): propaga el acarreo apagando toda la racha de unos y prendiendo el primer \(0\) por encima de ella. Eso puede crear un bit nuevo arriba del todo.
  • Caso especial: si lo que queda es exactamente \(3\) (los bits más altos son 11), suma \(2\) a la cuenta y termina (\(3 \to 2 \to 1\)).
  • Cuando llegas al bit \(1\) más alto sin nada arriba, el número ya es \(1\): termina.

¿Por qué es correcto? Cada división elimina el último bit; para llegar a \(1\) hay que eliminar todos los bits salvo el más alto. En un impar siempre se paga \(1\) operación para hacerlo par. Restar cuesta lo mismo que sumar para un \(1\) aislado, pero cuando hay una racha de \(k\) unos al final, sumar \(1\) la colapsa en un solo bit con una sola operación (en vez de pelear bit por bit), y ese único bit luego se baja con divisiones baratas. El único caso donde la racha "no paga" es \(3\), por eso se trata aparte. Este greedy coincide con la decisión \(\bmod\ 4\) de la Pista 4.

Pseudocódigo 
leer N, cadena s        // s[0] = bit más significativo
b <- invertir(s)        // b[0] = bit menos significativo
top <- indice del bit 1 mas alto
ops <- 0
i <- 0                  // posicion de las "unidades" del numero que queda

mientras i < top:
    si b[i] = 0:                       // par -> dividir entre 2
        ops <- ops + 1
        i <- i + 1
    si no:                             // impar
        si i+1 = top y b[i+1] = 1:     // lo que queda es 3 ("11")
            ops <- ops + 2             // 3 -> 2 -> 1
            terminar
        si b[i+1] = 1:                 // racha de unos (...11) -> sumar 1
            ops <- ops + 1
            j <- i
            mientras b[j] = 1: b[j] <- 0; j <- j + 1
            b[j] <- 1
            si j > top: top <- j        // crecio un bit arriba
        si no:                         // unidad aislada (...01) -> restar 1
            ops <- ops + 1
            b[i] <- 0

imprimir ops

Complejidad: \(O(N)\) en tiempo y \(O(N)\) en memoria. El acarreo total recorrido a lo largo de todo el algoritmo está acotado por \(O(N)\) (cada bit se enciende y se apaga un número constante de veces). Holgado para \(N \le 10^6\).

Alternativa equivalente: una DP de izquierda a derecha \(\textit{dp}[i][c]\) = costo mínimo de procesar el prefijo de \(i\) bits dejando un acarreo \(c \in \{0,1\}\); cada bit aporta \(1\) operación si queda par o \(2\) si queda impar (división más ajuste). También es \(O(N)\).
