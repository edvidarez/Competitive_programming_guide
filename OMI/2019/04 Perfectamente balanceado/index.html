Problema 04 — Perfectamente balanceado

Enunciado oficial: statement.md · Practica en línea: omegaUp

El problema

Tras pelear con los Avengers para conseguir las gemas del infinito, Thanos está agotado. Sin embargo, tiene una última tarea pendiente. Thanos tiene un arreglo de \(N\) enteros que debe dejar perfectamente balanceado. Un arreglo de números está perfectamente balanceado si la diferencia entre un elemento y el siguiente siempre es la misma para todos los elementos del arreglo.

Es decir: la diferencia entre cualquier elemento y el siguiente siempre debe ser constante.

Para balancear el arreglo, Thanos sólo puede sumar o restar 1 a cada elemento. Cada vez que Thanos sume o reste un 1 a algún elemento se considera que hizo una operación. Sólo se puede hacer una operación por elemento.

Thanos ofreció salvarte cuando elimine a la mitad de la población a cambio de tu ayuda. Imprime el menor número de operaciones que se necesitan para que el arreglo quede perfectamente balanceado, o -1 si no es posible balancearlo.

Entrada

En la primera línea, el entero \(N\). En la segunda línea, \(N\) enteros separados por un espacio que representan los números del arreglo.

Salida

Un entero que indica el menor número de operaciones necesarias para que el arreglo quede perfectamente balanceado, o -1 si no es posible balancearlo.

Límites

Límite de tiempo1 s (estándar de omegaUp; el enunciado no lo especifica)
Límite de memoria256 MB (estándar de omegaUp; el enunciado no lo especifica)
Puntaje total100 puntos
Tamaño del arreglo\(1 \le N \le 100000\)
Valor de cada elemento\(1 \le n_i \le 10^9\)

Subtareas

Ejemplos

Entrada

Salida

Resumen

Te dan un arreglo de \(N\) enteros. A cada elemento puedes sumarle 1, restarle 1 o dejarlo igual (a lo más una operación por elemento). Quieres el mínimo número de operaciones para que el arreglo sea una progresión aritmética (diferencia constante entre elementos consecutivos), o \(-1\) si es imposible.

Lo que importa de los límites \(N\) hasta \(10^5\) pide una solución lineal o casi lineal: nada de probar todas las diferencias posibles. Como cada elemento sólo cambia en \(\{-1,0,+1\}\), el espacio de búsqueda real es minúsculo. Ojo con el desbordamiento: un valor objetivo \(a_0 + i\cdot d\) puede llegar a \(\sim 10^{14}\) (con \(i\) hasta \(10^5\) y \(d\) hasta \(\sim 10^9\)), así que usa enteros de 64 bits.

Pistas

Pista 1 — ¿Cómo modelarlo?

¿Ya pensaste qué significa exactamente "perfectamente balanceado"? ¿No es justo una progresión aritmética: el elemento final en la posición \(i\) (contando desde 0) vale \(a_0 + i\cdot d\) para alguna diferencia común \(d\)?

Y la restricción más importante: cada elemento sólo puede moverse a lo más 1 (sumar 1, restar 1 o quedarse). ¿Cuánta libertad real tienes entonces sobre el valor final de cada posición? ¿No son apenas tres valores posibles para cada una?

Pista 2 — Empieza por lo chico (subtarea por subtarea)

¿Qué te regala cada subtarea?

  • Subtarea 1 (sólo se permite sumar para balancear): aquí cada elemento sólo puede subir o quedarse igual. ¿Eso no reduce tus opciones por posición a sólo dos?
  • Subtarea 2 (el primer elemento no se toca): si \(a_0\) queda fijo, ¿qué te falta por decidir para conocer TODA la progresión?

Pregunta clave que une todo: una progresión aritmética queda totalmente determinada por sus dos primeros valores. ¿Cuántos valores finales posibles tiene el primer elemento? ¿Y el segundo? ¿Cuántas combinaciones hay en total?

Pista 3 — La observación clave

Como el valor final del primer elemento tiene 3 posibilidades y el del segundo otras 3, ¿no son apenas \(3\times 3 = 9\) combinaciones para fijar \(a_0\) y \(d = a_1 - a_0\)?

Una vez fijados esos dos, el valor objetivo de TODOS los demás está forzado: la posición \(i\) debe terminar en \(a_0 + i\cdot d\). Para cada elemento restante, ¿qué condición debe cumplir su valor original respecto a ese objetivo para que sea alcanzable con a lo más una operación? Y si lo es, ¿cuánto cuesta exactamente?

Pista final — Solución completa (100 puntos)

La idea central es que cada elemento sólo puede moverse a lo más 1, así que su valor final está a distancia \(\le 1\) de su valor original: tiene apenas 3 valores posibles. Una progresión aritmética queda determinada por sus dos primeros términos, y a partir de ahí todo lo demás está forzado.

Caso base \(N \le 2\): cualquier arreglo de 1 o 2 elementos ya es una progresión aritmética (con 0 o 1 diferencias, siempre constante), así que la respuesta es \(0\) sin tocar nada.

Caso general \(N \ge 3\): prueba las \(9\) combinaciones de \((\delta_0, \delta_1) \in \{-1,0,1\}^2\) para los dos primeros elementos. Cada combinación fija \(a_0 = \text{arr}[0]+\delta_0\), \(a_1 = \text{arr}[1]+\delta_1\) y la diferencia común \(d = a_1 - a_0\); con eso queda determinada toda la progresión. Recorre el resto del arreglo verificando que cada elemento esté a distancia \(\le 1\) de su objetivo \(a_0 + i\cdot d\) (si no, esa combinación es imposible) y acumulando el costo, que es la suma de esas distancias más \(|\delta_0| + |\delta_1|\). La respuesta es el mínimo costo entre las combinaciones válidas, o \(-1\) si ninguna lo fue.

¿Por qué es correcto? El valor final de cada elemento sólo puede ser su original \(\pm 1\) o igual. Para el primero y el segundo eso da \(3\times 3\) opciones; cada par de valores finales fija por completo la progresión, de modo que para cada posición posterior su objetivo está determinado y el costo es exactamente \(|\text{objetivo}-\text{original}|\) (que sólo es factible si vale 0 o 1). No queda ninguna decisión libre adicional, así que recorrer las 9 combinaciones cubre todas las progresiones alcanzables. Cómo se sube de subtarea en subtarea: la subtarea 1 se resuelve restringiendo los deltas a \(\{0,+1\}\); la subtarea 2 se resuelve fijando \(\delta_0=0\); y la subtarea 3 es justamente el barrido completo de las 9 combinaciones.

Pseudocódigo 
funcion Resolver(N, arr):
    si N <= 2:
        regresa 0
    mejor = INFINITO
    para d0 en {-1, 0, 1}:
        para d1 en {-1, 0, 1}:
            a0 = arr[0] + d0
            a1 = arr[1] + d1
            dif = a1 - a0           # diferencia comun de la progresion
            costo = |d0| + |d1|
            valida = verdadero
            para i de 2 hasta N-1:
                objetivo = a0 + i * dif
                e = |arr[i] - objetivo|
                si e > 1:           # un solo +-1 no alcanza
                    valida = falso
                    rompe
                costo = costo + e
            si valida:
                mejor = min(mejor, costo)
    si mejor == INFINITO:
        regresa -1
    regresa mejor

Complejidad: \(O(9\cdot N) = O(N)\) en tiempo y \(O(N)\) en memoria (sólo para guardar el arreglo; el trabajo extra es \(O(1)\)).

Truco de C++ (obligatorio aquí) 
usa long long para 'objetivo' y 'dif'

El objetivo \(a_0 + i\cdot d\) puede alcanzar \(\sim 10^{14}\) (\(i\) hasta \(10^5\), \(d\) hasta \(\sim 10^9\)); con int de 32 bits se desborda y la comparación \(e \le 1\) puede dar resultados falsos por el "wrap-around". Nunca pongas la solución completa en código.
