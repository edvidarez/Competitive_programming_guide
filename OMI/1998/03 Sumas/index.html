3ª OMI · 1998 · Día 2

Problema 03 — Sumas

Enunciado oficial: statement.pdf · Fuente: olimpiadadeinformatica.org.mx

Descripción del problema

Si tomamos un entero positivo \(N\) lo podemos escribir de \(M\) formas distintas como suma de \(K\) enteros positivos elevados al cuadrado. Dos formas se consideran iguales si sólo difieren en el orden de los sumandos.

Nota: el \(0\) no es un entero positivo.

Ejemplo

El número \(98\) se puede escribir de \(6\) formas distintas como suma de \(5\) números elevados al cuadrado:

  • \(98 = 1^2 + 1^2 + 4^2 + 4^2 + 8^2\)
  • \(98 = 1^2 + 2^2 + 2^2 + 5^2 + 8^2\)
  • \(98 = 1^2 + 3^2 + 4^2 + 6^2 + 6^2\)
  • \(98 = 1^2 + 4^2 + 4^2 + 4^2 + 7^2\)
  • \(98 = 2^2 + 2^2 + 4^2 + 5^2 + 7^2\)
  • \(98 = 4^2 + 4^2 + 4^2 + 5^2 + 5^2\)

La forma \(1^2 + 2^2 + 2^2 + 5^2 + 8^2\) y la forma \(2^2 + 8^2 + 1^2 + 2^2 + 5^2\) son la misma porque sólo difieren en el orden de los sumandos, por lo cual, en el ejemplo, se contó sólo una de ellas.

Problema

Escribe un programa que, dados \(N\) y \(K\), encuentre el número \(M\) de formas en que se puede escribir \(N\) como la suma de \(K\) números elevados al cuadrado.

Entrada

En el primer renglón del archivo INPUT.TXT los enteros \(N\) y \(K\), donde \(1 \le N \le 10\,000\) y \(1 \le K \le 6\).

Salida

En el primer renglón del archivo OUTPUT.TXT el entero \(M\).

Límites

Límite de tiempoNo especificado en el enunciado
Límite de memoriaNo especificado en el enunciado
Puntaje total100 puntos
Cantidad a representar\(1 \le N \le 10\,000\)
Número de sumandos\(1 \le K \le 6\)

Subtareas

Sin subtareas: el puntaje es por casos de prueba (todo o nada).

Ejemplo

INPUT.TXT

98 5

OUTPUT.TXT

6

Con \(N = 98\) y \(K = 5\) existen \(6\) formas distintas (sin importar el orden) de escribir \(98\) como suma de \(5\) cuadrados de enteros positivos.

Grado de dificultad indicado en el enunciado oficial — Implementación: SE · Algoritmo: 2.

Etiquetas Programación dinámica (DP) Combinatoria Conteo Knapsack

Resumen

Dados \(N\) (\(\le 10\,000\)) y \(K\) (\(\le 6\)), cuenta de cuántas formas distintas se puede escribir \(N\) como suma de exactamente \(K\) cuadrados de enteros positivos. El orden de los sumandos no importa: \(1^2 + 2^2\) y \(2^2 + 1^2\) cuentan como una sola forma.

Lo que importa de los límites Cada sumando es \(b^2\) con \(b\) entero positivo, así que las bases posibles son \(b = 1, 2, \dots, \lfloor\sqrt{N}\rfloor \le 100\). Con \(K \le 6\) y \(N \le 10\,000\), una DP de tamaño \((K{+}1)\times(N{+}1)\) es minúscula. La parte fina es contar multiconjuntos (sin orden), no tuplas ordenadas.

Pistas

Pista 1 — ¿Cómo modelarlo?

¿Ya notaste que cada sumando es el cuadrado de un entero positivo \(b \ge 1\)? ¿Entre qué valores puede estar \(b\) si su cuadrado no debe pasarse de \(N\)?

Piensa en las bases \(1, 2, 3, \dots\) hasta \(\lfloor\sqrt{N}\rfloor\). Cuando \(N \le 10\,000\), esa lista llega a lo más hasta \(100\) (porque \(100^2 = 10\,000\)). Cada base aporta una "pieza" de valor \(b^2\), y puedes usar la misma base varias veces. Se parece mucho al problema de "¿de cuántas formas puedo juntar una cantidad con estas monedas?".

Pista 2 — Empieza por lo chico (fuerza bruta)

Con \(K \le 6\), ¿podrías recorrer todas las combinaciones de bases \(b_1 \le b_2 \le \dots \le b_K\) y revisar cuáles suman \(N\)? ¿Por qué pedir que vayan en orden no decreciente ya evita contar dos veces lo mismo?

Un backtracking que elige bases nunca menores a la anterior resuelve los casos chicos y, sobre todo, te enseña la idea central: si fuerzas el orden no decreciente de las bases, cada multiconjunto aparece una sola vez. Eso es justo lo que la versión eficiente debe respetar.

Pista 3 — El orden no importa (observación clave)

¿Cómo evitas que \(1^2 + 2^2\) y \(2^2 + 1^2\) cuenten como dos? ¿Y si procesas las bases una por una, igual que en el conteo de cambio con monedas que cuenta combinaciones (no permutaciones)?

El truco clásico: fija un orden global de las bases (1, luego 2, luego 3, …) e incorpóralas en ese orden. Así, cualquier forma se cuenta una única vez, en su versión ordenada de menor a mayor base.

Pista 4 — DP en dos dimensiones

¿Y si guardas \(\textit{dp}[j][s]\) = número de formas de escribir la suma \(s\) usando exactamente \(j\) cuadrados? Al meter una base nueva \(b\) que puedes repetir, ¿en qué orden conviene recorrer \(j\) y \(s\) para (a) permitir repetir esa base y (b) no contar permutaciones?

Pista dentro de la pista: recorrer \(s\) de menor a mayor (creciente) es lo que habilita usar la misma base muchas veces; procesar las bases por fuera, una a una, es lo que impide permutar.

Pista final — Solución completa (100 puntos)

Define \(\textit{dp}[j][s]\) = número de multiconjuntos de exactamente \(j\) cuadrados de enteros positivos que suman \(s\). El caso base es \(\textit{dp}[0][0] = 1\) (una sola forma de no usar nada y sumar cero).

Recorre las bases \(b = 1, 2, 3, \dots\) mientras \(b^2 \le N\). Cada base es una "moneda" de valor \(b^2\) que puedes usar tantas veces como quieras. Para incorporarla, actualiza \(\textit{dp}[j][s] \mathrel{+}= \textit{dp}[j-1][s - b^2]\), recorriendo \(j\) de \(1\) a \(K\) (creciente) y \(s\) de \(b^2\) a \(N\) (creciente).

¿Por qué es correcta?

  • No permuta: como las bases se procesan una por una en orden creciente, cada multiconjunto se cuenta exactamente una vez, en su forma ordenada de menor a mayor base.
  • Permite repetir una base: al recorrer \(s\) en sentido creciente dentro de la misma base, \(\textit{dp}[j-1][s-b^2]\) ya pudo haber usado \(b\) antes, así que se admiten varias copias de la misma base.
  • Cuenta exactamente \(K\): la dimensión \(j\) obliga a usar precisamente \(K\) sumandos.

La respuesta es \(\textit{dp}[K][N]\).

Pseudocódigo 
funcion ContarSumas(N, K):
    # dp[j][s] = # de multiconjuntos de j cuadrados positivos que suman s
    dp ← matriz (K+1) x (N+1) inicializada en 0
    dp[0][0] ← 1

    b ← 1
    mientras b*b ≤ N:
        cuad ← b*b
        para j desde 1 hasta K:            # j creciente
            para s desde cuad hasta N:     # s creciente -> permite repetir base
                dp[j][s] ← dp[j][s] + dp[j-1][s - cuad]
        b ← b + 1

    regresa dp[K][N]

Complejidad: tiempo \(O(\lfloor\sqrt{N}\rfloor \cdot K \cdot N)\approx 100\cdot 6\cdot 10\,000 = 6\times 10^{6}\) operaciones; memoria \(O(K\cdot N)\). El resultado \(M\) es pequeño (a lo más \(\approx 50\,005\) en este rango), así que cabe holgadamente en un entero de 32 bits; usar 64 bits no estorba.

Detalle de formato (estilo 1998): la entrada se lee del archivo INPUT.TXT y la respuesta se escribe en OUTPUT.TXT.
