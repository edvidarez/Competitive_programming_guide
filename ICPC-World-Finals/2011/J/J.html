ICPC World Finals · 2011

Problema J — Pyramids

PDF original del concurso: ../problems.pdf

El problema

No es muy difícil construir una pirámide si tienes muchos cubos idénticos. Sobre una base plana colocas, digamos, un cuadrado de \(10 \times 10\) cubos. Centrado encima de ese cuadrado colocas un cuadrado de \(9 \times 9\) cubos. Si sigues así terminas con un solo cubo en la cima. La altura de esta pirámide es igual a la longitud de su base, que en este caso es \(10\). A esto le llamamos pirámide alta (la marcamos con H).

Si crees que una pirámide alta es demasiado empinada, puedes hacer lo siguiente. Sobre la base de \(10 \times 10\) colocas un cuadrado de \(8 \times 8\), luego uno de \(6 \times 6\), y así sucesivamente, terminando con un cuadrado de \(2 \times 2\) en la cima (si empiezas con una base de longitud impar, terminas con un solo cubo arriba, claro). La altura de esta pirámide es aproximadamente la mitad de la longitud de su base. A esto le llamamos pirámide baja (la marcamos con L).

Hace mucho tiempo (bastante, de hecho) había un faraón que heredó de su padre un gran número de cubos de piedra. Le ordenó a su arquitecto usar todos los cubos para construir una pirámide, sin dejar ni uno solo sin usar. El arquitecto le explicó amablemente que no cualquier cantidad de cubos puede formar una pirámide. Con \(10\) cubos puedes construir una pirámide baja de base \(3\). Con \(5\) cubos puedes construir una pirámide alta de base \(2\). Pero con exactamente \(7\) cubos no se puede construir ninguna pirámide.

El faraón no se divirtió, pero tras pensarlo dio con nuevas restricciones:

  1. Deben usarse todos los cubos.
  2. Puedes construir más de una pirámide, pero debes construir la menor cantidad posible de pirámides.
  3. Todas las pirámides deben ser diferentes.
  4. Cada pirámide debe tener una altura de al menos 2.
  5. Cumpliendo lo anterior, la pirámide más grande debe ser lo más grande posible (es decir, la que contiene más cubos).
  6. Cumpliendo lo anterior, la siguiente más grande debe ser lo más grande posible.
  7. Y así sucesivamente.

Dibujando figuras en la arena, al arquitecto le tomó bastante tiempo dar con la mejor solución. Escribe un programa que determine cómo cumplir las restricciones del faraón, dado el número de cubos.

Entrada

La entrada consiste en varios casos de prueba, cada uno en una sola línea. Un caso de prueba es un entero \(c\), con \(1 \le c \le 10^6\), que indica el número de cubos disponibles. El último caso de prueba va seguido de una línea con un único cero (que no se procesa).

Salida

Para cada caso de prueba, muestra el número de caso seguido de las pirámides a construir. Las pirámides se listan con la más grande primero. Cada pirámide se especifica por la longitud de su base, seguida de una L para las pirámides bajas o una H para las altas. Si dos pirámides diferentes tienen el mismo número de cubos, lista primero la pirámide alta (H). Imprime impossible si no es posible cumplir las restricciones del faraón. Sigue el formato de la salida de ejemplo.

Ejemplo

Entrada

29
28
0

Salida

Case 1: 3H 3L 2H
Case 2: impossible

Caso 1: \(29 = 14 + 10 + 5\), es decir una alta de base \(3\) (\(H(3)=14\)), una baja de base \(3\) (\(L(3)=10\)) y una alta de base \(2\) (\(H(2)=5\)). No se puede con \(1\) ni con \(2\) pirámides distintas, así que el mínimo son \(3\); y entre todas las formas de usar \(3\), la más grande posible tiene \(14\) cubos. Caso 2: ninguna combinación de pirámides distintas (con altura \(\ge 2\)) suma exactamente \(28\), por eso es impossible.

Etiquetas Matemáticas Programación dinámica (DP) Greedy Backtracking/IDDFS

Resumen

Te dan \(c \le 10^6\). Las pirámides posibles tienen como tamaños los valores \(H(n)=\frac{n(n+1)(2n+1)}{6}\) (alta, \(n\ge 2\)) y \(L(n)=\frac{n(n+1)(n+2)}{6}\) (baja, \(n\ge 3\)). Hay que escribir \(c\) como suma de la menor cantidad posible de pirámides distintas y, entre esas, elegir la que maximiza lexicográficamente la secuencia de tamaños ordenada de mayor a menor. Si no se puede, imprime impossible.

Lo que importa de los límites Solo existen \(\approx 320\) tamaños de pirámide \(\le 10^6\) (los altos hasta base \(\approx 144\), los bajos hasta base \(\approx 181\)). Con tan pocos valores, el problema es un subset-sum chiquito; el mínimo número de pirámides nunca pasa de \(6\).

Pistas

Pista 1 — ¿Cuánto pesa cada pirámide?

¿Ya sacaste una fórmula cerrada para el número de cubos de cada tipo de pirámide? ¿Cuántos tamaños distintos pueden caber en \(10^6\)?

Una alta de base \(n\) apila \(n^2+(n-1)^2+\dots+1^2=\frac{n(n+1)(2n+1)}{6}\) cubos (su altura es \(n\), así que la condición «altura \(\ge 2\)» pide \(n\ge 2\)). Una baja de base \(n\) apila los cuadrados \(n,n-2,n-4,\dots\); sorprendentemente, tanto para \(n\) par como impar el total se simplifica a \(L(n)=\frac{n(n+1)(n+2)}{6}\) (su altura es \(\lceil n/2\rceil\), de modo que «altura \(\ge 2\)» pide \(n\ge 3\)). Genera todos los valores \(\le 10^6\): son solo unos \(320\). El problema ya no habla de geometría, sino de elegir un subconjunto de esos números que sume \(c\).

Pista 2 — La observación clave: «diferentes» y «pocas»

¿Notaste que dentro del rango ninguna pirámide alta vale exactamente lo mismo que una baja? ¿Y que jamás necesitarás muchas pirámides?

Como ningún valor \(H(n)\) coincide con un \(L(m)\) para \(\le 10^6\), cada tamaño identifica una única pirámide; por eso «todas diferentes» equivale a «valores distintos» y cada valor se usa a lo más una vez. La regla de desempate (listar la alta primero cuando empatan) nunca se dispara, pero conviene saber por qué está. Además, con valores tan dispersos, el mínimo número de pirámides para cualquier \(c\) alcanzable es muy chico (a lo sumo \(6\)), lo que abre la puerta a búsqueda con poda.

Pista 3 — Mínimo número de pirámides

Si cada valor se usa cero o una vez, ¿qué clásico te calcula el mínimo número de sumandos para llegar a \(c\)?

Es una mochila 0/1 de cardinalidad mínima: \(\textit{dp}[s]=\) mínimo número de pirámides distintas que suman \(s\), recorriendo cada valor una vez y la suma de mayor a menor. \(\textit{dp}[c]=\infty\) significa impossible (por ejemplo \(c=7,28\)). Esto se precomputa una sola vez para todo \(s\le 10^6\).

Pista 4 — Maximizar lexicográficamente

Fijado el mínimo \(k=\textit{dp}[c]\), ¿cómo eliges la combinación «más grande primero, luego la siguiente más grande…»?

Greedy de mayor a menor: recorre los tamaños de grande a chico e intenta tomar el más grande \(v\le\) (lo que falta) tal que el resto \((c-v)\) todavía se pueda completar con exactamente \(k-1\) pirámides distintas más pequeñas. La primera que pasa esa prueba es, por construcción, la cima más grande posible; repites con \(k-1\). Como \(k\le 6\), la verificación de factibilidad es un subset-sum de profundidad pequeñísima con poda.

Pista final — Solución completa (upsolving)

Idea. Reduce la geometría a una lista de tamaños:

  • Alta base \(n\ge 2\): \(H(n)=\frac{n(n+1)(2n+1)}{6}\).
  • Baja base \(n\ge 3\): \(L(n)=\frac{n(n+1)(n+2)}{6}\).

Genera todos los \((\text{valor},\text{base},\text{tipo})\) con valor \(\le 10^6\) (\(\approx 320\) ítems, todos con valor distinto). El problema es: escribir \(c\) como suma de un subconjunto de esos valores, con cardinalidad mínima, y entre los óptimos el que da la secuencia descendente de tamaños lexicográficamente mayor.

Por qué es correcto. Como no hay valores repetidos, «pirámides diferentes» = «valores distintos», así que es una mochila 0/1 pura. El \(\textit{dp}[\,]\) de cardinalidad mínima da el mínimo exacto (restricción 2). Para las restricciones 5–7, fijar la cima en el mayor valor que aún permita completar el resto con el conteo correcto es un argumento de intercambio estándar: cualquier solución óptima con una cima menor podría mejorarse, así que la elección greedy de mayor a menor produce justo el óptimo lexicográfico. La poda por cotas (suma de los \(t\) valores más grandes / más chicos disponibles) hace que el backtracking de profundidad \(\le 6\) sea instantáneo.

Pseudocódigo 
// ---- Precómputo (una sola vez) ----
ITEMS = []                       // (valor, base, tipo)
para n = 2,3,...:  v = n(n+1)(2n+1)/6;  si v > 1e6 romper;  ITEMS += (v,n,'H')
para n = 3,4,...:  v = n(n+1)(n+2)/6;   si v > 1e6 romper;  ITEMS += (v,n,'L')
ordenar ITEMS por valor DESCENDENTE   // (empate: 'H' antes que 'L'; no ocurre aquí)
VAL = lista de los valores en ese orden

dp[0..1e6] = INF;  dp[0] = 0
para cada valor v en VAL:                 // mochila 0/1, cada valor a lo más una vez
    para s = 1e6 hasta v:
        dp[s] = min(dp[s], dp[s-v] + 1)

// factible(i, t, r): ¿se puede formar r con EXACTAMENTE t ítems de VAL[i..]?
factible(i, t, r):
    si t == 0: regresar (r == 0)
    si i >= |VAL| o (|VAL|-i) < t: regresar falso
    // poda con cotas (t mayores / t menores disponibles desde i)
    si suma_t_mayores(i,t) < r o suma_t_menores(i,t) > r: regresar falso
    v = VAL[i]
    si v <= r y factible(i+1, t-1, r-v): regresar verdadero   // tomar
    regresar factible(i+1, t, r)                                // saltar

// ---- Consulta ----
resolver(c):
    si dp[c] == INF: imprimir "impossible"; regresar
    k = dp[c];  res = [];  i = 0;  faltan = k;  r = c
    mientras faltan > 0:
        // toma el MAYOR valor que permita completar el resto con faltan-1 ítems menores
        mientras VAL[i] > r o NO factible(i+1, faltan-1, r - VAL[i]):  i = i+1
        res += ITEMS[i];  r -= VAL[i];  faltan -= 1;  i = i+1
    imprimir res como  base+tipo  (ej. "3H 3L 2H")

Complejidad. Precómputo de la mochila: \(O(P\cdot C)\) con \(P\approx 320\) ítems y \(C=10^6\) (\(\sim 3\times10^8\) operaciones, una sola vez). Cada consulta reconstruye en \(O(P\cdot k)\) con \(k\le 6\), prácticamente instantáneo. Memoria \(O(C)\) para el arreglo \(\textit{dp}\).
