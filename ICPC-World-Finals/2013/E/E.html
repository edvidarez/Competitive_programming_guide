ICPC World Finals · 2013

Problema E — Harvard

PDF original del concurso: ../problems.pdf

El problema

El término "arquitectura Harvard" describe una computadora que tiene memorias físicamente separadas para las instrucciones y para los datos. El nombre viene de la computadora Harvard Mark I, entregada por IBM en 1944, que usaba cinta de papel para las instrucciones y relevadores para los datos. Algunos microcontroladores modernos usan esta arquitectura (aunque, claro, ya sin cinta de papel ni relevadores).

La memoria de datos está organizada en bancos, y todos los bancos guardan la misma cantidad de datos. Cada instrucción que referencia datos lleva:

  • un desplazamiento (offset) en bytes \(f\) dentro de un banco, y
  • un bit \(a\) que sirve para elegir el banco a referenciar.

Las reglas para elegir el banco son:

  • Si \(a = 0\), se referencia siempre el banco 0.
  • Si \(a = 1\), el banco usado es el que indique el valor del registro selector de banco (BSR, por bank select register).

Suponemos que cada instrucción tarda el mismo tiempo en ejecutarse, y que existe una instrucción capaz de fijar el valor del BSR.

Ejemplo del enunciado. Supón que hay 4 bancos de 8 bytes cada uno. Para acceder a la posición 5 hay dos opciones: usar una sola instrucción con \(a = 0\) y \(f = 5\); o bien fijar el BSR a 0 con una instrucción y luego usar una instrucción con \(a = 1\) y \(f = 5\). La primera opción es más rápida porque no necesita fijar el BSR.

Ahora, con la misma memoria, supón que la posición a acceder es la 20. Aquí solo funciona un camino: ejecutar una instrucción que fije el BSR a 2 (a menos que el BSR ya valga 2) y después usar una instrucción con \(a = 1\) y \(f = 4\).

Programas. Un programa es una secuencia de operaciones. Cada operación es una de dos:

  • una referencia a variable, escrita como Vi, donde \(i\) es un entero positivo; o
  • una repetición, escrita como Rn <programa> E, donde \(n\) es un entero positivo y <programa> es un programa arbitrario. Esta operación equivale a \(n\) ocurrencias sucesivas de <programa>.

Tu tarea. Determinar el tiempo mínimo de ejecución de un programa. Es decir: dados el número y el tamaño de los bancos de memoria y un programa a ejecutar, encuentra el número mínimo de instrucciones (las que referencian una posición de memoria y, posiblemente, las que fijan el BSR) que deben ejecutarse para correr el programa. Para lograrlo debes elegir una asignación de variables a bancos de memoria que produzca el menor tiempo de ejecución, y reportar ese tiempo: o sea, la cantidad de referencias a memoria más los ajustes del registro BSR. El valor inicial del BSR es indefinido, y solo cambia cuando una instrucción lo fija explícitamente.

Entrada

La entrada consiste en un solo caso de prueba, formado por dos líneas.

  • La primera línea trae dos enteros \(b\) y \(s\), donde \(b\) es el número de bancos de memoria y \(s\) es el número de variables que se pueden guardar en cada banco.
  • La segunda línea trae un programa no vacío con a lo más \(1\,000\) elementos separados por espacios (cada Rn, Vi y E cuenta como un elemento).

Puedes asumir lo siguiente:

  • En una repetición Rn, el número de repeticiones cumple \(1 \le n \le 10^6\).
  • En una operación de ciclo Rn <programa> E, el cuerpo del ciclo <programa> no está vacío.
  • En una referencia a variable Vi, el índice cumple \(1 \le i \le \min(b\cdot s,\ 13)\).
  • El número total de referencias a variables realizadas durante una ejecución del programa es a lo más \(10^{12}\).

Salida

Muestra el número mínimo de instrucciones que deben ejecutarse para completar el programa.

Ejemplos

Entrada 1

Entrada 2

Entrada 3

Entrada 4

V1 R2 V2 V4 R2 V1 E V3 E

Por qué dan esos resultados. En el caso 1, \(b=1\) y \(s=2\): solo existe el banco 0, así que las 5 referencias se hacen con \(a=0\) sin tocar nunca el BSR \(\Rightarrow 5\). En el caso 2, \(b=2\) y \(s=1\): una variable cae en el banco 0 (gratis) y la otra en el banco 1; al banco 1 solo se entra una vez \(\Rightarrow 5+1=6\). En el caso 3, todo cabe en el banco 0, así que las \(10\times 3 = 30\) referencias no fijan el BSR \(\Rightarrow 30\). En el caso 4, \(b=4\) y \(s=1\): conviene poner V1 en el banco 0; la secuencia ejecutada es V1 V2 V4 V1 V1 V3 V2 V4 V1 V1 V3 (11 referencias) y, quitando las V1 (banco 0), quedan V2 V4 V3 V2 V4 V3, que obligan a fijar el BSR 6 veces \(\Rightarrow 11+6=17\).

Resumen

Te dan \(b\) bancos de memoria de capacidad \(s\) cada uno y un programa (con ciclos Rn ... E) que referencia hasta 13 variables distintas. Debes asignar cada variable a un banco (a lo más \(s\) variables por banco) y, simulando la ejecución, minimizar el total de instrucciones: las referencias a memoria (cantidad fija) más las veces que hay que fijar el BSR.

Lo que importa de los límites A lo más 13 variables distintas \(\Rightarrow\) caben las asignaciones por particiones de un conjunto pequeño. El programa expandido tiene hasta \(10^{12}\) referencias \(\Rightarrow\) hay que trabajar sobre el árbol comprimido (\(\le 1000\) elementos) con aritmética de 64 bits, nunca expandiendo los ciclos. El límite de 10 s sugiere una búsqueda exponencial sobre asignaciones.

Pistas

Pista 1 — ¿Qué estás contando de verdad?

El número de referencias a memoria es fijo (lo dicta el programa). ¿Entonces qué es lo único que cambia según cómo acomodes las variables?

Cada referencia cuesta una instrucción siempre. Lo único variable es cuántas veces tienes que fijar el BSR. Minimiza eso y suma el total fijo de referencias al final.

Pista 2 — ¿Cuándo cuesta fijar el BSR?

El banco 0 se accede con \(a=0\) sin tocar el BSR. ¿Qué le pasa al valor del BSR cuando accedes una variable del banco 0? ¿Y cuándo necesitas fijar el BSR para un banco \(k\neq 0\)?

Accesos al banco 0 no cambian el BSR (son "transparentes"). Para una variable del banco \(k\neq 0\) necesitas una instrucción de fijado solo si el BSR no vale ya \(k\). Si tachas todos los accesos al banco 0, te queda una secuencia de bancos no nulos: el costo en BSR es exactamente el número de bloques maximales de bancos iguales consecutivos (cada bloque arranca con un fijado; el primero cuenta porque el BSR empieza indefinido).

Pista 3 — ¿Importa la etiqueta exacta del banco?

Dos variables en el mismo banco no nulo "se fusionan" (accederlas seguidas no cuesta fijado). ¿De verdad importa cuál banco no nulo, o solo importa qué variables comparten banco?

Los bancos no nulos son intercambiables: solo importa la partición de las variables. La decisión real es: qué subconjunto va al banco 0 (\(\le s\) variables, transparentes) y cómo agrupas el resto en \(\le b-1\) grupos de tamaño \(\le s\). El banco 0 es especial porque sus accesos desaparecen de la secuencia, ¡y eso puede fusionar bloques a su alrededor!

Pista 4 — ¿Cómo cuentas bloques sin expandir \(10^{12}\) referencias?

No puedes desenrollar los ciclos. ¿Qué resumen mínimo de un fragmento de programa te basta para combinar fragmentos por concatenación y por repetición?

Para una asignación fija, resume cada fragmento con tres datos sobre su secuencia de accesos no nulos: el primer banco, el último banco y el número de bloques. Al concatenar \(A\) con \(B\) restas 1 si último(A)=primero(B) (se fusionan). Al repetir \(P\) \(n\) veces, los \(n-1\) empalmes internos se fusionan si primero(P)=último(P).

Pista final — Solución completa (upsolving)

Modelo. El costo total \(=\) (referencias ejecutadas, fijo) \(+\) (número de instrucciones que fijan el BSR). Llama \(T\) al total de referencias. Asigna cada variable a un banco; el banco 0 se accede con \(a=0\) y no toca el BSR. Borra mentalmente de la traza de ejecución todos los accesos al banco 0: queda una secuencia de bancos no nulos \(c_1,c_2,\dots,c_m\). Como el BSR empieza indefinido y solo cambia al entrar a un banco no nulo, el número de fijados del BSR es el número de bloques maximales de iguales en esa secuencia, es decir \(\#\{i : c_i \neq c_{i-1}\} + 1\) (con \(c_0\) inexistente). La respuesta es \(T + \min_{\text{asignación}} (\text{bloques})\).

Por qué es correcto. Una variable del banco 0 siempre cuesta una sola instrucción y nunca altera el BSR, así que es transparente: puede unir dos accesos del mismo banco no nulo que la rodean. Para los bancos \(k\neq 0\), entrar a \(k\) cuando el BSR ya vale \(k\) cuesta solo la referencia; si no, cuesta un fijado extra. Eso es justo "un fijado por cada bloque nuevo". Como los bancos no nulos son intercambiables, basta probar todas las particiones de las variables (un bloque distinguido = banco 0, hasta \(b-1\) bloques más, cada bloque \(\le s\) variables).

Cómo evaluar una asignación sin expandir. Parsea el programa a un árbol (hojas Vi, nodos de concatenación, nodos de repetición Rn ... E). Para una asignación fija, calcula recursivamente el resumen \((\textit{primero},\ \textit{ultimo},\ \textit{bloques})\) de la secuencia de accesos no nulos. Usa enteros de 64 bits: \(\textit{bloques}\) puede llegar a \(\sim 10^{12}\). El total \(T\) se calcula con la misma idea (cuentas).

Pseudocódigo 
VACIO = (primero=∅, ultimo=∅, bloques=0)

combinar(A, B):                 // concatenación A·B
    si A es VACIO: regresa B
    si B es VACIO: regresa A
    fusion = (A.ultimo == B.primero) ? 1 : 0
    regresa (A.primero, B.ultimo, A.bloques + B.bloques - fusion)

evaluar(nodo, color):           // color[v]=0 ⇒ banco 0 (transparente)
    si nodo == Vi:
        si color[i] == 0: regresa VACIO
        si no: regresa (color[i], color[i], 1)   // 1 bloque
    si nodo == Concat(h1..hk):
        acc = VACIO
        para cada h: acc = combinar(acc, evaluar(h, color))
        regresa acc
    si nodo == Rep(n, cuerpo):
        P = evaluar(cuerpo, color)
        si P es VACIO: regresa VACIO
        fus = (P.primero == P.ultimo) ? 1 : 0
        // n copias: n-1 empalmes internos, cada uno fusiona si fus
        regresa (P.primero, P.ultimo, n*P.bloques - (n-1)*fus)

Resolver():
    arbol = parsear(programa)
    T = totalReferencias(arbol)          // 64 bits
    best = +infinito
    // banco 0 = subconjunto Z de variables, |Z| <= s (transparentes)
    para cada subconjunto Z con |Z| <= s:
        resto = variables \ Z
        // particiona 'resto' en <= b-1 grupos, cada grupo <= s
        para cada partición válida de 'resto':
            color: vars de Z -> 0 ; cada grupo -> un id distinto > 0
            r = evaluar(arbol, color).bloques
            best = min(best, r)
    regresa T + best

Verificación con el ejemplo 4 (\(b=4,s=1\), poniendo V1 en el banco 0): la traza es V1 V2 V4 V1 V1 V3 V2 V4 V1 V1 V3, \(T=11\). Tachando V1 queda V2 V4 V3 V2 V4 V3: 6 bloques. Total \(11+6=17\), justo lo esperado. (Cualquier otra elección del banco 0 da 7 bloques.)

Complejidad. Sea \(V \le 13\) el número de variables y \(m \le 1000\) el tamaño del árbol. Cada evaluación es \(O(m)\). El número de asignaciones a probar está acotado por las particiones de \(V\) elementos (números de Bell), a lo más \(\sim B(13)\approx 2.8\times 10^7\) en el peor caso, así que el total es \(O(B(V)\cdot m)\). El generoso límite de 10 s está pensado justo para esta búsqueda exponencial. Memoria \(O(m)\).
