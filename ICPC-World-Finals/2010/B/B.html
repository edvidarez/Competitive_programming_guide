ICPC World Finals · 2010

Problema B — Barcodes

PDF original del concurso: ../problems.pdf

El problema

Code-11 es un sistema de código de barras para codificar caracteres, usado sobre todo para etiquetar equipo de telecomunicaciones. Los caracteres que se pueden codificar son los dígitos del 0 al 9, el guion (–) y un carácter especial de inicio/fin (start/stop) que aparece al principio y al final de cada código de barras Code-11.

Code-11 es un sistema discreto: cada carácter se codifica de forma independiente. La codificación de un carácter se representa con cinco regiones (barras) adyacentes que se alternan entre un color oscuro y uno claro, empezando por una barra oscura. El ancho de cada barra es angosto o ancho, según el esquema de abajo, donde 0 representa una barra angosta y 1 una barra ancha.

Así, el carácter 1 se codifica como: una barra oscura ancha, una barra clara angosta, una barra oscura angosta, una barra clara angosta y, por último, una barra oscura ancha. Los códigos de los caracteres individuales deben ir separados por una barra clara angosta cuya única función es separar los caracteres.

Una codificación de dos anchos como la de Code-11 tiene la ventaja de ser simple: como solo hay que distinguir una barra angosta de una ancha, Code-11 admite una gran tolerancia de impresión en condiciones de baja calidad.

Caracteres de verificación C y K

Para detectar errores, los códigos de barras Code-11 que usamos llevan dos caracteres de verificación, C y K, agregados al final del mensaje (antes del carácter de fin). Si los \(n\) caracteres a codificar (de izquierda a derecha) son \(c_1\) hasta \(c_n\), entonces el peso del carácter de verificación \(C\) es:

\[ \left(\sum_{i=1}^{n}\Big((n-i)\bmod 10 + 1\Big)\cdot w(c_i)\right)\bmod 11 \]

donde \(w(c_i)\) es el peso asociado al carácter \(c_i\). Los pesos de los dígitos 0 al 9 son 0 al 9; el peso del guion es 10. (Nota: \(\bmod\) tiene mayor precedencia que \(+\), así que \((n-i)\bmod 10 + 1\) significa \(\big((n-i)\bmod 10\big)+1\).)

El peso del carácter de verificación \(K\) es:

\[ \left(\sum_{i=1}^{n+1}\Big((n-i+1)\bmod 9 + 1\Big)\cdot w(c_i)\right)\bmod 11 \]

donde \(c_{n+1}\) es el carácter de verificación \(C\). Por ejemplo, si el mensaje a codificar es 123–45, entonces el carácter de verificación \(C\) es 5 y el \(K\) es 2. Los códigos de los ocho caracteres 123–4552, precedidos y seguidos por el código del carácter de inicio/fin, forman la codificación Code-11 completa del mensaje.

La lectura

Los lectores simples de código de barras miden la intensidad de la luz reflejada por el código hacia un arreglo lineal con cientos de pequeños sensores CCD, cada uno reportando claro u oscuro. Se identifican las regiones claras y oscuras, y el ancho de cada región lo usa el software de decodificación para validar el código y obtener la información codificada.

Como la orientación del código no es fija, el software debe poder decodificarlo sin importar si se escanea de izquierda a derecha o de derecha a izquierda.

Tu problema es decodificar la información obtenida al escanear un código de barras Code-11, dados los anchos de las regiones claras y oscuras detectadas por el lector. Supón que una barra ancha está pensada para ser el doble de ancha que una angosta. Debido a inconsistencias entre impresoras, el ancho de una barra puede ser hasta 5 % mayor o menor de lo previsto. No hay mensajes de longitud cero (códigos que contengan solo inicio/fin, caracteres de verificación y las separaciones entre caracteres).

Entrada

La entrada contiene varios casos de prueba, cada uno representando un intento de escaneo. Los datos de cada caso comienzan con un entero \(m \le 150\) que indica el número de regiones detectadas por el lector. A continuación vienen \(m\) enteros \(d_1\ldots d_m\) (\(1 \le d_i \le 200\)) que dan el número de sensores en cada región (dentro de una región, todos los sensores reportan la misma intensidad de luz). Los datos de cada caso comienzan y terminan con una barra oscura (no hay espacio en blanco inicial ni final).

El último caso de prueba va seguido por un único entero 0.

Salida

Para cada caso de entrada, imprime una línea con el número de caso y el resultado de la decodificación:

  • Si el código se decodifica correctamente, muestra la secuencia de caracteres del mensaje (sin sus caracteres de verificación).
  • Si la decodificación es exitosa pero el carácter de verificación \(C\) es incorrecto, muestra bad C.
  • Si la decodificación es exitosa y \(C\) es correcto pero \(K\) es incorrecto, muestra bad K.
  • Muestra bad code si el código no se puede decodificar por anchos de barra fuera del rango permitido, por códigos de inicio/fin faltantes o inválidos, o por alguna otra condición inválida.

Sigue el formato de la salida de ejemplo.

Resumen

Te dan \(m\) anchos enteros de regiones claras/oscuras de un código de barras Code-11 (la primera y la última son oscuras). Debes clasificar cada región como angosta o ancha respetando la tolerancia, reconstruir los caracteres (5 barras cada uno, separados por una barra clara angosta), verificar que haya inicio/fin en ambos extremos, validar los dígitos de control \(C\) y \(K\), e imprimir el mensaje (o bad C / bad K / bad code).

Lo que importa de los límites \(m \le 150\): todo es \(O(m)\), no hay nada que optimizar. El reto es la corrección: clasificación robusta de anchos, estructura exacta y las dos fórmulas de verificación bien aplicadas.

Pistas

Pista 1 — ¿Qué tamaño debe tener un código válido?

¿Ya contaste cuántas regiones produce un código con \(T\) caracteres (incluyendo inicio, fin, \(C\) y \(K\))?

Cada carácter son 5 barras, y entre dos caracteres consecutivos hay 1 barra separadora. Con \(T\) caracteres eso da \(5T + (T-1) = 6T-1\) regiones. Entonces \(m\) debe cumplir \(m\equiv 5\pmod 6\) y \(T=(m+1)/6\). Como el mensaje tiene \(n=T-4\) caracteres (quitando inicio, fin, \(C\) y \(K\)) y no hay mensajes vacíos, necesitas \(T\ge 5\) (es decir \(m\ge 29\)). Si esto no se cumple ya es bad code.

Pista 2 — ¿Angosta o ancha? La clasificación

¿Cómo decides si una región es angosta o ancha si no conoces la unidad \(u\) (ancho de una barra angosta)?

Hay un hueco limpio: las angostas caen en \([0.95u,\,1.05u]\) y las anchas en \([1.9u,\,2.1u]\), y \(1.05u < 1.9u\). Además siempre existen barras angostas (las separadoras) y siempre existen barras anchas (toda codificación de la tabla tiene al menos un 1). El ancho mínimo \(d_{\min}\) es por fuerza una angosta. Toda angosta es \(\le 1.105\,d_{\min}\) y toda ancha es \(\ge 1.81\,d_{\min}\); un umbral como \(1.5\,d_{\min}\) separa los dos grupos sin ambigüedad.

Pista 3 — Validar la tolerancia de verdad

Clasificar con un umbral no basta: ¿cómo verificas que existe una sola unidad \(u\) que hace válidos todos los anchos a la vez?

Cada región impone un intervalo para \(u\): una angosta de ancho \(d\) exige \(u\in[d/1.05,\,d/0.95]\); una ancha de ancho \(d\) exige \(u\in[d/2.1,\,d/1.9]\). El código es consistente solo si la intersección de todos esos intervalos es no vacía (máximo de las cotas inferiores \(\le\) mínimo de las superiores). Si está vacía: bad code. Tip: trabaja con racionales o multiplica en cruz para no sufrir con los \(\pm 5\%\) en los bordes.

Pista 4 — La orientación desconocida

El escaneo puede venir al derecho o al revés. ¿Cómo sabes cuál orientación es la correcta?

El patrón de inicio/fin es 00110; al invertirlo queda 01100, que es un 6, no un inicio/fin. Por eso a lo más una de las dos lecturas (la secuencia tal cual, o invertida) tiene inicio/fin en ambos extremos. Intenta decodificar en las dos direcciones y quédate con la que sí presenta inicio/fin a ambos lados. La verificación de tolerancia (Pista 3) es la misma en ambas direcciones (mismo multiconjunto de anchos), así que solo cambia el mapeo de los 5 bits a caracteres.

Pista final — Solución completa (upsolving)

Procesa cada caso de forma independiente. La idea es: validar el conteo, clasificar anchos, validar la unidad, decodificar en la dirección correcta y, por último, recomputar \(C\) y \(K\).

Paso a paso

  1. Conteo. Si \(m\not\equiv 5\pmod 6\) o \(T=(m+1)/6 < 5\) → bad code. Si no, \(n=T-4\).
  2. Clasificación. \(d_{\min}=\min d_i\). Marca cada región como angosta si \(2d_i \le 3\,d_{\min}\) (o sea \(d_i\le 1.5\,d_{\min}\)), si no, ancha. Obtienes una secuencia de bits (0 = angosta, 1 = ancha).
  3. Tolerancia. Calcula \(\,[lo,hi]\) intersectando los intervalos de \(u\) de la Pista 3. Si \(lo > hi\) → bad code.
  4. Estructura y decodificación (probar 2 direcciones). Para la secuencia de bits \(b\) (y luego para su reverso): el carácter \(j\) (\(0\le j<T\)) ocupa los bits \([6j,\,6j+4]\); el bit \(6j+5\) (para \(j<T-1\)) es el separador y debe ser angosto (0). Toma cada grupo de 5 bits, búscalo en la tabla:
    • Si algún grupo no está en la tabla, o algún separador es ancho → esa dirección falla.
    • El primer y el último carácter deben ser inicio/fin; los \(T-2\) de en medio deben ser dígitos o guion (no inicio/fin) → si no, falla.
    Si ninguna dirección funciona → bad code. Si una funciona, usa sus \(T-2\) caracteres intermedios: los primeros \(n\) son el mensaje, el siguiente es el \(C\) leído y el último el \(K\) leído.
  5. Verificación. Recomputa \(C\) y \(K\) con las fórmulas usando los pesos (\(0\!-\!9\) para los dígitos, \(10\) para el guion).
    • Si \(C_{\text{leído}}\ne C_{\text{calc}}\) → bad C.
    • Si no, si \(K_{\text{leído}}\ne K_{\text{calc}}\) → bad K.
    • Si ambos coinciden → imprime el mensaje (los \(n\) caracteres).
    Para \(K\), recuerda que \(c_{n+1}\) es el \(C\) calculado (o el leído, que coincide porque solo llegamos a verificar \(K\) cuando \(C\) ya fue correcto).

Por qué es correcto: el hueco de tolerancia garantiza que la clasificación por umbral es la única posible para datos válidos, y la prueba de intersección descarta anchos fuera de rango. La asimetría del patrón de inicio/fin (00110 ≠ 01100) garantiza que a lo más una orientación es válida, eliminando ambigüedad. El resto es aplicar literalmente la estructura y las dos fórmulas.

Pseudocódigo 
tabla = { "00001":'0', "10001":'1', "01001":'2', "11000":'3',
          "00101":'4', "10100":'5', "01100":'6', "00011":'7',
          "10010":'8', "10000":'9', "00100":'-', "00110":'$' }   # $ = inicio/fin
peso = {'0':0,...,'9':9, '-':10}
char(x): 0..9 -> '0'..'9';  10 -> '-'        # valor de control a caracter (10 = guion)

resolver(d[1..m]):
    if m mod 6 != 5: return "bad code"
    T = (m+1)/6
    if T < 5: return "bad code"
    n = T - 4

    dmin = min(d)
    bit[i] = 0 si 2*d[i] <= 3*dmin si no 1           # angosta / ancha

    # validar unidad u (tolerancia 5%)
    lo = 0; hi = +inf
    para cada region i:
        if bit[i]==0: lo = max(lo, d[i]/1.05); hi = min(hi, d[i]/0.95)
        else:         lo = max(lo, d[i]/2.10); hi = min(hi, d[i]/1.90)
    if lo > hi: return "bad code"                      # usar racionales/cruz

    para dir en [bits, reverso(bits)]:
        chars = []; ok = true
        para j en 0..T-1:
            g = dir[6j .. 6j+4]
            if g no en tabla: ok=false; break
            chars.push(tabla[g])
            if j < T-1 and dir[6j+5] != 0: ok=false; break   # separador ancho
        if not ok: continuar
        if chars[0]!='$' or chars[T-1]!='$': continuar
        medio = chars[1 .. T-2]                         # T-2 caracteres
        if algun car de medio == '$': continuar         # inicio/fin en el centro
        # exito estructural
        msg = medio[0 .. n-1]; Cleido = medio[n]; Kleido = medio[n+1]
        Ccalc = ( sum_{i=1..n} ((n-i) mod 10 + 1)*peso[msg[i]] ) mod 11
        if char(Ccalc) != Cleido: return "bad C"
        c = msg + [char(Ccalc)]                          # c_1..c_{n+1}
        Kcalc = ( sum_{i=1..n+1} ((n-i+1) mod 9 + 1)*peso[c[i]] ) mod 11
        if char(Kcalc) != Kleido: return "bad K"
        return msg
    return "bad code"

Nota: char(x) convierte el valor de control al carácter correspondiente: \(0\to{}\)'0', …, \(9\to{}\)'9' y, de forma clave, \(10\to{}\)'–' (guion). \(C\) y \(K\) están en \(0\ldots10\): el residuo \(10\) sí es representable, justo como el guion (su codificación 00100 está en la tabla y su peso es \(10\)). Por eso un carácter de verificación \(C\) o \(K\) puede ser el guion en entradas válidas —por ejemplo el mensaje «–» (un solo guion, \(n=1\)) da \(C=(1\cdot 10)\bmod 11=10\), es decir \(C={}\)'–'. Con \(char(10)={}\)'–' la comparación \(char(C_{\text{calc}})\ne C_{\text{leído}}\) (e igual para \(K\)) queda correcta también en ese caso límite.

Complejidad: \(O(m)\) tiempo y \(O(m)\) memoria por caso; con \(m\le 150\) es instantáneo.
