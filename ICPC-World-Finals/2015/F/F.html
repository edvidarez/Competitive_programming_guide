ICPC World Finals · 2015

Problema F — Keyboarding

PDF original del concurso: ../problems.pdf

El problema

¿Cuántas pulsaciones de tecla hacen falta para escribir un mensaje de texto? Podrías pensar que es igual al número de caracteres del texto, pero eso solo es cierto cuando cada pulsación genera un carácter. En los dispositivos de bolsillo, las opciones para escribir texto suelen ser limitadas. Algunos aparatos ofrecen apenas unos cuantos botones, muchísimos menos que el número de letras del alfabeto. En esos dispositivos puede que se necesiten varias pulsaciones para escribir un solo carácter.

Un mecanismo para lidiar con estas limitaciones es un teclado virtual mostrado en una pantalla, con un cursor que se puede mover de tecla en tecla para seleccionar caracteres. Cuatro botones de flecha controlan el movimiento del cursor y, cuando el cursor queda colocado sobre la tecla adecuada, al presionar el quinto botón se selecciona el carácter correspondiente y se agrega al final del texto. Para terminar el texto, el usuario debe navegar hasta la tecla Enter y seleccionarla. Así, con solo cinco botones físicos, se le permite al usuario un conjunto arbitrario de caracteres y escribir texto de cualquier longitud.

En este problema te dan el diseño de un teclado virtual y tu tarea es determinar el número mínimo de pulsaciones necesarias para escribir un texto dado, donde presionar cualquiera de los cinco botones físicos cuenta como una pulsación. Las teclas están acomodadas en una cuadrícula rectangular, de manera que cada tecla virtual ocupa una o más casillas unitarias conectadas de la cuadrícula. El cursor empieza en la esquina superior izquierda del teclado y se mueve en las cuatro direcciones cardinales, de tal forma que siempre salta hasta la siguiente casilla unitaria en esa dirección que pertenezca a una tecla distinta. Si no existe tal casilla, el cursor no se mueve.

La Figura F.1, que ilustra el Sample Input 1, muestra una manera posible de escribir CONTEST usando 30 pulsaciones en ese teclado virtual de ejemplo. Los puntos rojos representan las teclas virtuales donde se presionó el botón de selección.

Entrada

La primera línea contiene dos enteros \(r\) y \(c\) (\(1 \le r, c \le 50\)), que dan el número de filas y columnas de la cuadrícula del teclado virtual. El teclado virtual se especifica en las siguientes \(r\) líneas, cada una con \(c\) caracteres. Los valores posibles de estos caracteres son letras mayúsculas, dígitos, un guion (-) y un asterisco (*, que representa la tecla Enter). Solo hay una tecla para cada carácter dado. Cada tecla está formada por una o más casillas de la cuadrícula, que siempre forman una región conectada. La última línea de la entrada contiene el texto que hay que escribir. Este texto es una cadena no vacía de a lo más \(10\,000\) de los caracteres disponibles, distintos del asterisco.

Salida

Muestra el número mínimo de pulsaciones necesarias para escribir todo el texto, incluyendo la tecla Enter al final. Se garantiza que el texto se puede escribir.

Resumen

Te dan un teclado en una rejilla de \(r \times c\) donde cada tecla es una región conectada de casillas con la misma letra. Un cursor empieza arriba a la izquierda; cada flecha lo manda a la siguiente casilla, en esa dirección, que pertenezca a una tecla distinta. Pulsar selección teclea la letra de la tecla actual. Calcula el mínimo de pulsaciones (flechas + selecciones) para escribir el texto dado, terminando con un * (Enter).

Lo que importa de los límites Hasta \(2500\) casillas y un texto de hasta \(10^4\). El espacio de estados \(\text{casilla}\times\text{índice}\) es \(\le 2.5\times10^7\): cabe una BFS por capas. Las pulsaciones tienen todas costo 1, así que es BFS pura (no Dijkstra).

Pistas

Pista 1 — ¿Cómo modelarlo?

¿Ya pensaste en esto como un grafo donde un estado es "dónde está el cursor" y "cuánto del texto ya escribí"? ¿Qué pulsaciones cambian cada parte de ese estado?

El cursor vive en una casilla \((i,j)\). Llevas escritos los primeros \(k\) caracteres del objetivo. Una flecha cambia la casilla pero no \(k\); una selección (válida) deja la casilla igual y aumenta \(k\) en 1. Toda pulsación cuesta 1, así que buscas el camino más corto en un grafo no ponderado: eso es BFS.

Pista 2 — El objetivo real a escribir

¿Notaste que el texto de entrada no trae el Enter, pero la salida sí lo cuenta? ¿Cómo incorporas eso sin casos especiales?

Define el objetivo como \(T = \text{texto} + \texttt{*}\). Tienes que teclear exactamente esos \(m = |\text{texto}|+1\) caracteres en orden. El estado final es haber tecleado los \(m\).

Pista 3 — El movimiento de las flechas

¿Cómo calculas a dónde llega una flecha cuando una tecla ocupa varias casillas? ¿Conviene precomputarlo una sola vez?

Desde \((i,j)\), moverte en una dirección recorre casillas en esa dirección hasta encontrar la primera con letra distinta a la de \((i,j)\); ahí aterrizas. Si te sales de la rejilla sin cambiar de letra, no te mueves. Precomputa los cuatro destinos de cada casilla en \(O(r c \cdot \max(r,c))\); luego cada flecha es \(O(1)\).

Pista 4 — Una observación que ahorra memoria

¿De verdad necesitas guardar la distancia para los \(2.5\times10^7\) estados a la vez? ¿Qué tienen en común todos los estados con el mismo \(k\)?

El índice \(k\) solo crece. Procesa capa por capa: con las distancias de todas las casillas que tienen la letra \(T[k]\) (donde puedes pulsar selección), haz una BFS de fuentes múltiples por las flechas para llegar a todas las casillas, y de ahí salta a las casillas de la letra \(T[k+1]\) sumando 1 (la selección). Solo necesitas el arreglo de distancias de una capa, \(O(rc)\) memoria.

Pista final — Solución completa (upsolving)

Modelo. Estado \((i,j,k)\): cursor en la casilla \((i,j)\), ya tecleados los primeros \(k\) caracteres de \(T=\text{texto}+\texttt{*}\). Inicio \((0,0,0)\). Aristas de costo 1:

  • Flecha en cualquiera de 4 direcciones: \((i,j,k)\to(\text{dest},k)\), donde dest es la casilla precomputada (la siguiente con letra distinta, o la misma si no se mueve).
  • Selección, solo si \(\text{letra}(i,j)=T[k]\): \((i,j,k)\to(i,j,k+1)\).

Como todas las aristas cuestan 1, el camino mínimo desde \((0,0,0)\) a cualquier estado con \(k=m\) se halla con BFS. Esa distancia es la respuesta.

Por qué es correcto. Cualquier secuencia válida de pulsaciones es exactamente un camino en este grafo: las flechas mueven el cursor según la regla del enunciado y las selecciones añaden el siguiente carácter requerido (no sirve pulsar selección sobre una letra que no toca, porque el texto debe quedar idéntico a \(T\)). BFS da el camino con menos aristas, es decir, menos pulsaciones. El Enter se cuenta porque es el último carácter de \(T\).

Implementación por capas (memoria \(O(rc)\)). Equivale a la BFS de estados, pero solo guarda una capa de distancias:

Pseudocódigo 
leer r, c, rejilla, texto
T = texto + "*"           // objetivo, longitud m
precomputar nxt[i][j][d]  // destino de cada flecha (4 direcciones)
    para cada casilla (i,j) y dirección d:
        avanzar en d mientras misma letra y dentro de la rejilla
        nxt = primera casilla con letra distinta; si no hay, (i,j)

INF = +infinito
dist[i][j] = INF para toda casilla      // distancia a la capa actual
dist[0][0] = 0                          // cursor inicia arriba-izquierda

para k = 0 .. m-1:
    objetivo = T[k]
    // BFS de fuentes múltiples por las flechas:
    // siembra la cola con TODAS las casillas que ya tienen dist finita
    // (son los puntos desde los que arrancamos para alcanzar T[k])
    cola = todas las (i,j) con dist[i][j] < INF, en orden creciente de dist
    BFS_flechas(dist, nxt, cola)   // relaja dist por movimientos de costo 1

    // ahora dist[i][j] = mínimas pulsaciones para tener el cursor en (i,j)
    // habiendo tecleado k caracteres
    // pulsar selección en las casillas con letra == objetivo  (sumar 1)
    ndist[i][j] = INF para toda casilla
    para cada casilla (i,j) con letra(i,j) == objetivo:
        ndist[i][j] = dist[i][j] + 1
    dist = ndist                   // ahora representa k+1 caracteres tecleados

respuesta = min sobre todas las casillas de dist[i][j]   // estados con k = m
imprimir respuesta

Detalle de BFS_flechas: como una misma tecla puede ocupar varias casillas conectadas (p.ej. QQ, WW en el ejemplo 2, o la tecla B ocupando toda una columna en el ejemplo 4), tras una pulsación de selección hay potencialmente varias casillas fuente —todas las casillas de la tecla T[k]— y cada una con su propia distancia. La BFS por flechas de cada capa debe sembrarse con todas esas casillas en orden no decreciente de distancia (BFS de fuentes múltiples ordenada, estilo 0-1), no desde una sola casilla. La forma más simple y siempre correcta es la BFS plana sobre estados \((i,j,k)\) con cola FIFO, ya que todas las aristas valen 1.

Complejidad. Precómputo de flechas \(O(rc\cdot\max(r,c))\le 50\cdot2500\). La BFS recorre cada estado una vez: \(O(rc\cdot m) = O(2500\cdot10^4)=2.5\times10^7\) con 5 aristas por estado. Memoria \(O(rc)\) por capa (o \(O(rc\cdot m)\) si se hace la BFS de estados plana). Entra de sobra en 4 segundos.
