ICPC World Finals · 2012

Problema D — Fibonacci Words

PDF original del concurso: ../problems.pdf

El problema

La secuencia de palabras de Fibonacci es una sucesión de cadenas de bits que se define así:

\[ F(n) = \begin{cases} 0 & \text{si } n = 0 \\[2pt] 1 & \text{si } n = 1 \\[2pt] F(n-1) + F(n-2) & \text{si } n \ge 2 \end{cases} \]

Aquí el signo \(+\) denota la concatenación de cadenas (pegar una después de la otra). Los primeros elementos son:

n     F(n)
0     0
1     1
2     10
3     101
4     10110
5     10110101
6     1011010110110
7     101101011011010110101
8     1011010110110101101011011010110110
9     1011010110110101101011011010110110101101011011010110101

Dado un patrón de bits \(p\) y un número \(n\), ¿cuántas veces aparece \(p\) dentro de \(F(n)\)?

Entrada

La primera línea de cada caso de prueba contiene el entero \(n\) (\(0 \le n \le 100\)). La segunda línea contiene el patrón de bits \(p\). El patrón \(p\) no está vacío y tiene una longitud de a lo más \(100\,000\) caracteres.

Salida

Para cada caso de prueba, muestra el número de caso seguido por la cantidad de apariciones del patrón de bits \(p\) dentro de \(F(n)\). Las apariciones pueden traslaparse. El número de apariciones será menor que \(2^{63}\).

Resumen

Te dan \(n\) (\(\le 100\)) y un patrón binario \(p\) (\(|p|\le 10^5\)). Hay que contar cuántas veces aparece \(p\), con traslapes, en la palabra de Fibonacci \(F(n)\), que se define por \(F(n)=F(n-1)+F(n-2)\). La respuesta cabe en 64 bits, pero \(F(n)\) es exponencialmente larga y NO se puede construir.

Lo que importa de los límites \(|F(n)|\) crece como el Fibonacci \(\text{Fib}(n)\): para \(n=96\) tiene del orden de \(10^{20}\) bits, así que es imposible generarla. En cambio \(n\le 100\) y \(|p|\le 10^5\) invitan a una recursión sobre \(n\) que solo mira los bordes de cada concatenación.

Pistas

Pista 1 — ¿Construir o contar?

¿De verdad necesitas escribir toda la cadena \(F(n)\) para contar las apariciones? ¿Qué tan grande es \(F(100)\)?

La longitud de \(F(n)\) es un número de Fibonacci: explota exponencialmente. Construirla es imposible. Pero el conteo sí cabe en 64 bits, lo que sugiere una fórmula recursiva sobre \(n\) en lugar de fuerza bruta sobre la cadena.

Pista 2 — La observación clave: el conteo se parte en tres

Como \(F(n)=F(n-1)+F(n-2)\), ¿dónde puede caer una aparición de \(p\)? ¿Cuántos casos distintos hay?

Sea \(C(n)\) el número de apariciones de \(p\) en \(F(n)\). Toda aparición está completamente dentro de \(F(n-1)\), completamente dentro de \(F(n-2)\), o cruza la junta entre las dos. Por lo tanto: \[ C(n) = C(n-1) + C(n-2) + \text{cruces}(n). \] Las dos primeras partes ya están resueltas por recursión. Solo falta contar las apariciones que pisan la frontera.

Pista 3 — Las apariciones de la junta solo dependen de los bordes

Una aparición que cruza la junta usa a lo más \(|p|-1\) caracteres de cada lado. ¿Necesitas todo \(F(n-1)\) y \(F(n-2)\), o solo un trocito?

Una ocurrencia de longitud \(m=|p|\) que cruza la frontera toma como mucho \(m-1\) caracteres del final de \(F(n-1)\) y como mucho \(m-1\) del principio de \(F(n-2)\). Entonces basta guardar, para cada \(F(k)\): su prefijo y su sufijo recortados a longitud \(m-1\) (más la longitud real, acotada). Con esos bordes formas una ventana corta alrededor de la junta y cuentas ahí las apariciones que tocan la frontera.

Pista 4 — ¿Cómo mantener prefijos y sufijos sin la cadena completa?

Si \(F(k)=F(k-1)+F(k-2)\), ¿puedes obtener el prefijo (y el sufijo) recortado de \(F(k)\) a partir de los de \(F(k-1)\) y \(F(k-2)\)?

Sí. Prefijo de \(A+B\): toma el prefijo de \(A\); si \(A\) es más corto que el recorte, complétalo con el de \(B\). Sufijo de \(A+B\): toma el sufijo de \(B\); si \(B\) es más corto que el recorte, anteponle el sufijo de \(A\). Llevando longitudes "saturadas" (capadas a, digamos, \(2\cdot 10^5\)) evitas el desbordamiento y todo cabe.

Pista final — Solución completa (upsolving)

Sea \(m=|p|\) y \(L=m-1\) (cuántos caracteres de cada lado pueden tocar una junta). Para cada \(k=0,1,\dots,n\) guardamos un registro con cuatro datos:

  • \(\text{len}[k]\): longitud de \(F(k)\), saturada en una cota \(\ge 2L+1\) (por ejemplo \(2\cdot 10^5+5\)) para que no desborde.
  • \(\text{cnt}[k]\): número de apariciones de \(p\) en \(F(k)\) (exacto, en entero de 64 bits sin signo; nunca pasa de la respuesta final \(<2^{63}\)).
  • \(\text{pre}[k]\): los primeros \(\min(\text{len}[k],L)\) caracteres de \(F(k)\).
  • \(\text{suf}[k]\): los últimos \(\min(\text{len}[k],L)\) caracteres de \(F(k)\).

Casos base. \(F(0)=\)0, \(F(1)=\)1. Calcula \(\text{cnt}\), \(\text{pre}\), \(\text{suf}\), \(\text{len}\) de cada uno directamente. Si \(n\le 1\) responde \(\text{cnt}[n]\) y termina.

Transición para \(k\ge 2\), con \(F(k)=F(k-1)+F(k-2)\):

  • Longitud: \(\text{len}[k]=\min(\text{CAP},\ \text{len}[k-1]+\text{len}[k-2])\).
  • Prefijo: empieza con \(\text{pre}[k-1]\); si \(\text{len}[k-1]
  • Sufijo: empieza con \(\text{suf}[k-2]\); si \(\text{len}[k-2]
  • Conteo: \(\text{cnt}[k]=\text{cnt}[k-1]+\text{cnt}[k-2]+\text{cruces}\).

Cruces (lo único delicado). Forma la ventana \(M = (\text{últimos } a \text{ de } \text{suf}[k-1]) + (\text{primeros } b \text{ de } \text{pre}[k-2])\), donde \(a=\min(\text{len}[k-1],L)\) y \(b=\min(\text{len}[k-2],L)\). Corre KMP de \(p\) sobre \(M\) y cuenta solo los matches cuyo inicio \(s\) cumple \(s\le a-1\) y \(s+m-1\ge a\) (es decir, que pisan la frontera situada entre los índices \(a-1\) y \(a\)). Eso da \(\text{cruces}\).

¿Por qué es correcto? Toda aparición de \(p\) en \(F(k)\) está dentro de \(F(k-1)\) (contada en \(\text{cnt}[k-1]\)), dentro de \(F(k-2)\) (contada en \(\text{cnt}[k-2]\)), o cruza la junta. Las que cruzan usan a lo más \(m-1\) caracteres de cada lado, así que la ventana \(M\) (de longitud \(\le 2L\)) las contiene todas, exactamente; ninguna aparición interior se cuenta dos veces porque exigimos que el match pise la frontera. Los recortes \(\text{pre}/\text{suf}\) se mantienen correctos por construcción del prefijo/sufijo de una concatenación.

Pseudocódigo 
resolver(n, p):
    m = longitud(p);  L = m - 1;  CAP = 2*100000 + 5
    fail = prefijo_KMP(p)              # tabla de fallos de p

    # --- casos base ---
    base[0] = registro("0");  base[1] = registro("1")
    si n <= 1: regresa base[n].cnt

    R[0] = base[0];  R[1] = base[1]
    para k de 2 a n:
        A = R[k-1];  B = R[k-2]
        len  = min(CAP, A.len + B.len)

        # prefijo recortado a L
        pre = A.pre
        si A.len < L:  pre = (pre + B.pre)[0 : L]

        # sufijo recortado a L
        suf = B.suf
        si B.len < L:  suf = (A.suf + suf)[fin-L : fin]

        # apariciones que cruzan la junta A|B
        a = min(A.len, L);  b = min(B.len, L)
        izq = ultimos_a_de(A.suf, a)
        der = primeros_b_de(B.pre, b)
        M = izq + der                 # |M| <= 2L
        cruces = 0
        para cada match s de p en M (con KMP, fail):
            si s <= a-1 y s + m - 1 >= a:   cruces += 1

        cnt = A.cnt + B.cnt + cruces
        R[k] = registro(len, cnt, pre, suf)

    regresa R[n].cnt

# registro(s): construye len/cnt/pre/suf de una cadena CORTA s
#   cnt = #apariciones de p en s ;  pre = s[0:L] ;  suf = s[-L:]

Verificación con los ejemplos. \(F(6)=1011010110110\): 10 aparece en \(0,3,5,8,11\Rightarrow 5\) (Caso 1). \(F(7)\): 10 \(\Rightarrow 8\) (Caso 2). \(F(6)\) con 01 da las posiciones \(1,4,6,9\Rightarrow 4\) (Caso 3). Todos coinciden con la salida esperada; el Caso 5 (\(n=96\)) se obtiene de la misma recursión sin construir la palabra.

Complejidad. Para cada \(k\) se hace un KMP sobre una ventana de tamaño \(O(m)\), más manejo de prefijos/sufijos de tamaño \(O(m)\). Con \(n\le 100\) niveles: \(O(n\cdot m)\) tiempo y \(O(n\cdot m)\) memoria (esta última se puede bajar a \(O(m)\) guardando solo los dos últimos registros). Con \(m\le 10^5\) y \(n\le 100\) es trivialmente rápido.
